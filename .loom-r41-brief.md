## Scope

PLAN 0.26.0 (MINOR) — **hooks 보조 센서 (claude 워커 상태 힌트) — 스파이크 최소 배선 5단계 구현**. claude 워커의
상태 전이(승인 대기 · 턴 종료 · 작업 시작)를 브릿지는 지금 pane 스크레이프 글자 패턴으로만 짐작한다 — 승인 대기를
자주 놓치고(가짜 `agent_blocked`), still-running 유예는 최대 5분 휴리스틱에 의존한다. 이 PLAN은 스파이크가 기술
가능성을 확정한 **최소 배선 5단계**(① 스폰 시 hook 주입 → ② 워커 hook이 로컬 write → ③ 브릿지가 소켓 watch →
④ `flight.hookHint` 적재 + 우선 분기 → ⑤ hook 부재 시 폴백)를 구현해, claude Code hook(`Stop` ·
`Notification` matcher `permission_prompt`/`idle_prompt` · `UserPromptSubmit`)이 상태 전이를 브릿지-로컬
소켓으로 무전하면 그것을 **완료 힌트로만** 쓴다.

핵심 원칙(정본 = `docs/COMPETITIVE_NOTES.md` §2.5.2/§2.5.3): 스크레이프는 공통 눈으로 유지(모든 벤더 공통
경로) · hook은 **있으면 힌트 · 없으면 현행 그대로(fail-open)** · 본문 정본은 여전히 **§5.1 artifact** ·
**자동 close·자동 approve 없음**(done_proposal과 동형). v1은 **claude 워커만** 배선(Codex/Grok 어댑터는 이득
증명 후).

**신규 신뢰 경계 변경**(브릿지가 외부 프로세스=워커 hook의 로컬 소켓 입력을 수신)이라 MINOR·**R41 필수**. 스파이크가
**경량 R{n}**으로 예상(신규 wire/MCP 없음 · `inject-control`(승인된 loomDir 0600 소켓)이 선례).

## 리뷰 초점 (PLAN Review impact 절 + 아키텍트 판단 요청 — verdict의 핵심)

1. **D4 신규 신뢰 경계의 방어 충분성** (핵심): 브릿지가 **서버**(방향 역전 — inject-control은 클라이언트, 여기선
   브릿지가 리스너로 수신)로서 외부 프로세스(워커 hook)의 로컬 소켓 입력을 수신하는 신규 능력의 방어가 충분한지 —
   `loomDir()/hook-<cardId>.sock` 0600 소켓 + `isPathUnderLoomDir` 가드 + 비정상/파싱 실패 조용히 무시(카운터만) +
   실제 결과는 스크레이프로 회수(신뢰 노출 최소)의 조합이 `inject-control` 선례 대비 등가 이상인지, 방향 역전
   (브릿지=서버)이 선례에 없던 신규 노출면(스테일 소켓·재-cardId 충돌 U3)을 도입하는지.
2. **D3 힌트 전용 원칙의 실효성**: hook 이벤트가 `flight.hookHint`에 적재돼 **딱 두 지점**(`onCardHerdrEvent` 우선
   참조 · still-running 유예 입력)의 우선 분기에만 쓰이고, **자동 close·자동 approve 배제**가 실제로 보장되는지 —
   `Stop` 힌트가 still-running 유예를 **앞당기되 done을 단독 확정하지 않고**(done 최종 권위는 herdr status +
   스크레이프 확증) `permission_prompt`가 승인 대기 마커로만(가짜 `agent_blocked` 1:1 교정) 소비되는 설계가
   §2.5.2③·§2.5.3 규칙 2(done_proposal 동형)를 코드 분기 수준에서 강제하는지.
3. **D5 폴백 무회귀(fail-open) 어서션의 검증 가능성**: hookHint 부재·`no_listener`·소켓 오류 → 현행 herdr status +
   스크레이프 + 5분 유예 경로가 **0.25.0과 바이트 동일**해야 한다는 무회귀 어서션이 실제로 검증 가능한지 — 우선
   분기를 "얹기만" 한다는 주장이 검증 계획(무회귀 어서션 · fail-open)으로 실증 가능한 형태로 명세됐는지,
   `inject-control.ts:51,99` `no_listener` 폴백 동형 논증이 완결되는지.
4. **U4 설계 리스크** (`Stop` 힌트 vs still-running 상호작용): `Stop`이 워커 백그라운드 커맨드 실행 중에도 발화하면
   (0.23.7 still-running과 동일 통증) `Stop` 힌트가 오히려 조기 완료를 유발할 수 있다 — D3의 "done 최종 권위는 기존
   경로" 원칙이 이를 막는지(유예를 앞당기되 done 단독 확정 불가), `Stop` 힌트를 스크레이프 still-running 지표와 어떻게
   AND 결합할지가 설계로 닫혔는지 아니면 라이브 실측(U4)에 남는 리스크인지.
