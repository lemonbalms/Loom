### 1.1 grok 레인 구동 지식 — **R44 처방의 정정** (재현 가치 높음)

**R44 원장은 4회차 성공의 원인을 *"읽기 전용 커맨드 allowlist를 세션 스코프로 부여해 성공"* 이라고
적었다. 그 진단은 틀렸다.** R45에서 같은 allowlist를 주고도 실패했고, 실제 원인과 해법은 다음과 같다.

**실패 기제 (실측):**

1. 전역 `~/.claude/settings.json`의 **`defaultMode: "auto"`** 를 grok CLI가 **조용히 상속**한다.
2. `auto` 모드는 커맨드마다 **LLM 권한 분류기**를 띄운다.
3. 분류기가 **미인식 커맨드에 `"ask"`** 를 반환한다 — allowlist에 없는 형태(파이프·서브셸·낯선 플래그
   조합)는 전부 여기로 떨어진다.
4. **헤드리스에는 프롬프트를 띄울 표면이 없다** → 취소가 아니라 **런 전체가 중단**된다.

**→ allowlist로는 해결되지 않는다.** allowlist는 "인식된 커맨드"의 목록을 넓힐 뿐,
**분류기 자체를 끄지 못한다.** 미인식 커맨드가 하나라도 나오면 같은 자리에서 다시 죽는다.

**작동 구성 (정본):**

```
--permission-mode dontAsk --sandbox read-only
```

- `dontAsk` = **분류기를 우회**한다(ask를 반환할 자리를 없앤다).
- `read-only` = **커널 강제 샌드박스**. 이것이 안전 보장의 본체다.

**샌드박스는 실증됐다 — 그리고 allowlist보다 강하다.** 쓰기 시도가
**`IO Error: Operation not permitted`** 로 실패했고 `git status`는 불변이었다.
**allowlist는 커맨드 문자열을 검사하는 애플리케이션 층 방어라 셸 리다이렉션(`>`·`tee`·`sh -c`)으로
우회 가능하지만, `read-only` 샌드박스는 커널이 write 시스템콜을 거부하므로 우회 경로가 없다.**
R44가 자랑한 "git 서브커맨드 allowlist로 커밋·푸시를 권한 계층에서 차단"은 **약한 보장**이었다 —
같은 목적을 샌드박스가 더 강하게, 그리고 더 적은 설정으로 달성한다.

**read-only 확인:** HEAD 불변 · `git status` 불변 · 사용자 설정 파일 미수정.

### 4.1 M1 기각 경위 — **"검증자의 지적도 검증 대상"의 두 번째 실례**

grok은 `herdr-lifecycle.test.ts:366-367`을 **main HEAD 트리로 열었다.** 그러나 §5.1의 해당 표는

### 4.2 M2 처방 정밀화 경위 — **왜 provisional 강등이 아니라 범위 명시인가**

grok 처방은 *"구현 전 ⑨ 종결 또는 U8을 provisional로 표기"* 였다. **축자 반영하지 않았다(교훈 31 —
처방이 아니라 결함을 본다).** 결함의 실체는 "U8이 틀렸을 수 있다"가 아니라 **"U8이 이번 버전에
하중을 주는지가 문서에 없다"** 이기 때문이다. 아키텍트 실측:

- **PATCH별 락 목록에 U8이 없다** — PATCH 1 `U1·U2·U3·U4·U7` / PATCH 2 `U2·U10` /
  PATCH 3 `U1·U3·U4·U5·U6·U7` / PATCH 4 `U7`(+`U1`·`U5`) / PATCH 5 `U10·U11`.
- **G-1~G-11의 판정 근거에도 U8이 없다.**
- **코드 표면 영향 0** — U8이 다루는 presence 계열 배선은 **후속 C**이고, v0.28.0에서
  `presence_unknown`은 **도달 불가**로 명시돼 있다(설계 §2.5 · §4.2 `presence_unknown` 행 · §7).

**U8을 provisional로 강등하면 정확성이 오히려 떨어진다.** provisional은 "이 락을 믿고 코드를 쓰되
나중에 바뀔 수 있다"는 신호인데, **v0.28.0은 U8을 믿고 쓰는 코드를 한 줄도 만들지 않는다.**
락을 흔드는 대신 **범위를 명시하는 것**이 사실에 부합한다. 반영 문안의 골자:

1. U8은 **어떤 PATCH·게이트에도 걸리지 않는다**(코드 표면 영향 0).
2. 따라서 **⑨ 미검증은 v0.28.0 구현 착수를 막지 않는다.**
3. 단 **⑨는 후속 C 착수의 선결 조건**이다 — presence를 실제로 배선하는 시점에 U8 문안이 하중을 받는다.
4. **codex 레인 복구(2026-07-25 14:27) 후 최우선 재검증 지정 유지.**
5. **정직성 고지**: R43b §4가 "(C) 본체 R 게이트 재확인"으로 미룬 사안이고 **v0.28.0이 그 본체이되
   U8이 범위 밖이라 이번 게이트에서 축자 대조가 이뤄지지 않았음**을 명기. **"충돌 없음"을 주장하지
   않는다** — 주장하는 것은 "이번 버전의 코드 표면에 닿지 않으므로 판단을 미뤄도 안전하다"까지다.
