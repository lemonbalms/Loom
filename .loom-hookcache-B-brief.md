# [HOOKCACHE-B] 원인 B 해소 브리프 — claude-mem 헤더 분 단위 ts 제거 (B-4 소스 패치)

> 저장소 내부 untracked 브리프. 커밋 대상 아님.

당신은 **codex 레인 구현자**다. 프롬프트 캐시가 매 세션 깨지는 **원인 B — claude-mem이 SessionStart로
주입하는 컨텍스트 헤더에 분 단위 타임스탬프가 박혀 캐시 재사용 상한이 1분으로 고정되는 문제**를
소스 패치로 해소한다.

## ⓪ 최우선 경고 — blast radius (읽고 시작하라)

**이 카드가 손대는 대상은 리포 안이 아니라 `~/.claude` 홈 = 전 프로젝트 공용 플러그인 설치본이다.**
오작동은 이 리포가 아니라 **오너의 모든 프로젝트**에 번진다.

- **④에 지정된 함수(`qJ()` / `eQ()`) 외에는 어떤 파일도, 어떤 줄도 건드리지 마라.**
- `~/.claude/settings.json`·`~/.claude/CLAUDE.md`·`~/.claude-mem/settings.json`·다른 플러그인 ·
  `enabledPlugins` 플래그 — **전부 범위 밖**. 플러그인 비활성화(B-5)·hooks.json handler 제거(B-3)는
  **기각된 선택지**다. 실행하지 마라.
- 수정 전 **원본 백업 필수**(예: 같은 디렉터리에 `worker-service.cjs.pre-hookcache`). 백업 경로를
  회신에 명기하라.
- 애매하면 **멈추고 보고**하라. 홈 디렉터리에서 "일단 해보고 되돌리자"는 허용되지 않는다.

## ① 스펙 SSOT

정본은 `docs/spikes/HOOK-CACHE-FIX-DESIGN.md` **§5 원인 B 선택지 표(B-4 행)** · **§1.6 소스 확정** ·
**§8-6 미해결 질문(실사용 로드 경로)** 이다. 배경은 `docs/spikes/WARM-BASE-FORK-SPIKE.md`.

**오너 확정(락)**: 원인 B 처리 = **B-4 소스 패치**. B-3(hooks.json handler 제거)·B-6(수용)은 **기각**.

왜 B-6이 안 되는가: 원인 A만 고치고 B를 두면 "같은 분 안에 재시작할 때만" 캐시가 히트한다 —
원인 A 수정의 이득이 대부분 상쇄된다. 미스 1회의 실비용은 **33,915토큰 cache-write**(세션당 1회)로
실측돼 있다.

원본(비-미니파이):

function formatHeaderDateTime(): string {

### ⚠️ 설계 문서(§5)의 지목 파일은 **틀렸다** — 아키텍트 실측으로 교정됨

설계 정본과 초기 브리프는 `scripts/context-generator.cjs`의 `rt()`/`lt()`를 지목했다. **그 파일은 데드
번들이다** — 13.11.0 전체에서 `grep -rl "context-generator"` 결과 **0건**, 어디에서도 호출되지 않는다.
**거기를 고치면 런타임 동작이 전혀 바뀌지 않는다.** 절대 그 파일을 패치하지 마라.

진짜 실체는 **`scripts/worker-service.cjs`**다. SessionStart hooks.json이 실행하는 것도 이 파일이고,
헤더 생성 로직이 이 번들 안에 **인라인 복제**돼 있다. 대상은 두 함수다:

```js
// …/cache/thedotmack/claude-mem/13.11.0/scripts/worker-service.cjs  (미니파이, 2427행)
// 953행 — 플레인텍스트 헤더
function qJ(){let t=new Date,e=t.toLocaleDateString("en-CA"),r=t.toLocaleTimeString("en-US",{hour:"numeric",minute:"2-digit",hour12:!0}).toLowerCase().replace(" ",""),n=t.toLocaleTimeString("en-US",{timeZoneName:"short"}).split(" ").pop();return`${e} ${r} ${n}`}
// 소비처: function HJ(t){return[`# [${t}] recent context, ${qJ()}`,""]}

// 955행 — ANSI 컬러 헤더 (본문 동일)
function eQ(){ …같은 형태… }
// 소비처: function tQ(t){return["",`${ne.bright}${ne.cyan}[${t}] recent context, ${eQ()}${ne.reset}`,…]}
```

**앵커는 함수명이 아니라 본문 패턴으로 잡아라.** 미니파이어가 버전마다 이름을 다르게 붙인다(13.10.4에선
같은 함수가 `ts()`/`ps()`였다). 줄 좌표도 마찬가지로 불안정하다. 식별 시그니처는
`toLocaleDateString("en-CA")` + `timeZoneName:"short"` 조합이다.

`server-service.cjs`에는 이 로직이 없다(`recent context` 0건). 추가 패치 불요.

**설정으로는 못 고친다** — `~/.claude-mem/settings.json`에 헤더 포맷/타임스탬프 관련 키는 존재하지
않는다(`CLAUDE_MEM_CONTEXT_*` 전수 확인 완료). 재조사 불요.

## ④ 패치 내용 (이것만 하라)

`qJ()` / `eQ()` 가 **날짜까지만** 반환하도록 수정한다.

- 반환값에서 **시:분 성분과 타임존 성분을 제거**하고 `toLocaleDateString("en-CA")` 산출(`YYYY-MM-DD`)만
  남긴다. 즉 헤더가 `# [proj] recent context, 2026-07-20` 형태가 되면 성공이다.
- **함수 시그니처·호출부·이름은 변경 금지.** 호출부를 고치지 말고 함수 본문만 고쳐라 —
  미니파이 번들에서 호출부를 건드리면 추적 불가능한 파손이 난다.
- **최소 diff 원칙**: 미니파이 코드를 재포맷하지 마라. 한 줄 안에서 필요한 토큰만 바꿔라.
  파일 전체 prettier/재빌드 금지(diff가 폭발해 재적용 절차가 무의미해진다).
- `qJ()`와 `eQ()`가 **둘 다** 같은 형태로 고쳐졌는지 확인하라 — 각각 플레인/ANSI 헤더를 담당하므로
  한쪽만 고치면 출력 경로에 따라 분이 남는다.

## ⑤ 검증 — 실측으로 확인하라 (코드 읽기로 끝내지 마라)

1. 패치 후 **새 세션**을 시작해 claude-mem이 주입한 헤더를 직접 확인 —
   `# [<project>] recent context, …` 줄에 **분 단위 ts가 사라졌는지** 육안 실측.
   (같은 분 안에서 두 번 확인하면 판별이 안 된다. **분 경계를 넘겨** 최소 2회, 헤더가 동일한지 본다.)
2. claude-mem 기능 무회귀 확인: 세션 시작 메모리 주입이 **여전히 뜨는지**(내용 자체가 사라지면 실패다).
3. 위 두 결과를 **헤더 문자열 실물과 함께** 회신에 붙여라. "고쳤다"는 서술만으로는 불합격이다.

## ⑤-bis 감지장치 — **오너 확정 필수 산출** (이것이 이 카드의 절반이다)

`known_marketplaces.json`에서 thedotmack은 **`autoUpdate: true`**다. 즉 새 버전 디렉터리가 생기는 순간
패치는 **사람이 알아챌 계기 없이** 사라지고, 증상은 "어느 날부터 캐시가 다시 깨짐"이라 원인 추적이
지극히 어렵다. **lessons에 절차를 적어두는 것만으로는 이 실패 모드를 못 막는다** — 아무도 안 읽기 때문이다.
오너 결정은 **패치 + 감지장치**이며, autoUpdate는 끄지 않는다(보안·버그수정 수신을 늦추지 않기 위해).

리포에 검사 스크립트를 신설하라:

- `scripts/check-mem-header.ts` (기존 `scripts/` 형식·`handoff:lint` 선례를 따를 것)
- 동작: **현재 활성 claude-mem 설치본을 런타임에 재해석**(경로 하드코딩 금지 — `installed_plugins.json`의
  `installPath`를 읽어라. 버전이 오르면 경로가 바뀐다)한 뒤, `scripts/worker-service.cjs`에서 헤더 생성
  함수 본문의 **분 단위 시그니처**(`timeZoneName:"short"` + `minute:"2-digit"` 조합)가 살아 있으면 **실패**.
- 실패 메시지는 원인·조치·lessons 앵커를 한 화면에 담아라. 미래의 독자는 배경을 전혀 모른다.
- `package.json`에 `check:mem-header` 별칭 추가.
- **설치본이 없거나 경로 해석에 실패하면 fail-open(통과)**하되 그 사실을 stderr로 알려라 — 다른 머신·다른
  오너 환경에서 이 스크립트 때문에 게이트가 막히면 안 된다. 막을 대상은 "패치가 조용히 사라진 것"뿐이다.
- 유닛 테스트: 분 시그니처가 있는 합성 문자열 → 실패, 날짜만 있는 합성 문자열 → 통과. **실제 홈 파일에
  의존하지 마라**(테스트가 오너 머신 상태에 묶이면 안 된다).

**전체 `bun test`·리포 회귀 판정은 이 카드의 대상이 아니다.** 리포 변경은 ⑤-bis 스크립트와 ⑥ lessons뿐이며,
신규 유닛만 돌려 확인하라.

## ⑥ 필수 산출 — 재적용 절차 기록 (이것을 빠뜨리면 카드 미완료다)

B-4의 유일한 결함은 **비영속**이다 — claude-mem 플러그인이 업데이트되면 패치가 **조용히 원복**되고
캐시가 다시 깨진다. 그런데 그때는 아무도 원인을 기억하지 못한다. 따라서 기록이 패치 본체만큼 중요하다.

1. **`tasks/lessons/platform.md`에 전문 추가** — 다음을 포함하라:
   - 증상(플러그인 업데이트 후 세션당 33,915토큰 cache-write 재발) → 원인(헤더 분 단위 ts) →
     조치(`worker-service.cjs`의 `qJ()`/`eQ()` 날짜까지만 — **`context-generator.cjs`는 데드 번들이라
     고쳐도 무효**라는 사실을 반드시 함께 적어라. 이 함정으로 한 번 헛발질할 뻔했다).
   - **패치한 경로 전부**(②에서 확정한 목록) + 백업 파일 경로.
   - **재적용 절차를 그대로 따라 할 수 있게** 적어라 — 어느 파일의 어느 함수를, 어떻게 찾고(미니파이라
     줄 좌표는 버전마다 변하므로 **함수명 grep으로 재특정**), 무엇으로 바꾸는지.
   - **탐지 방법**: `bun run check:mem-header`(⑤-bis)가 1차 방어선이라는 점 + 육안으로는 세션 헤더에
     분이 다시 보이면 원복이라는 점. **`autoUpdate: true`이므로 원복은 예고 없이 일어난다**고 명기하라.
   - 업스트림(B-7, thedotmack/claude-mem 헤더 granularity 옵션화 요청)이 항구 해법이라는 점 1줄.
2. **`tasks/lessons.md`는 열지도 마라 — 인덱스 줄은 "초안 텍스트"로 회신에 제출한다.**
   - **이유**: 동시 발사된 HOOKCACHE-C 카드가 **같은 파일의 인덱스 32줄을 전면 재작성 중**이다. 네가
     append 하면 그 작업과 정면 충돌한다. 병합은 **아키텍트가 C 완료 후 직접** 한다.
   - 따라서 너는 **`tasks/lessons/platform.md` 전문 추가까지만** 파일로 쓰고, 인덱스 줄은 아래 형식의
     **한 줄 텍스트를 회신 본문에 적어라**(파일 수정 금지).
   - 형식: `- [platform] 2026-07-20 (n) 제목: 본문` · **≤120자** · 트리거("claude-mem 업데이트 후 캐시
     재파괴")가 줄에 남아 있을 것. 번호 `(n)`은 비워두고 `(n)`으로 둬라 — 아키텍트가 채운다.

## ⑨ 완료 신호

모든 작업 후 **최종 줄**에 정확히 다음 형식의 마커를 출력하라:

```
[HOOKCACHE-B-DONE] paths=<패치한 파일 수> minute-ts=<gone|present> mem-inject=<ok|broken> detector=<added|skipped> lessons=<added|skipped>
```

예: `[HOOKCACHE-B-DONE] paths=1 minute-ts=gone mem-inject=ok detector=added lessons=added`.
(로드 경로는 ②에서 13.11.0 한 곳으로 확정됐으므로 `paths=1`이 정상이다.)
실패·미확정이 있으면 실제 값으로 정직하게 보고하고, 무엇이 왜 그런지 마커 위에 짧게 적어라.
