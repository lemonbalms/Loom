# [HOOKCACHE-A] 원인 A 코드 축 구현 브리프 — `--part all` 병합 + 라우드 절단 + `session-context:lint`

> 저장소 내부 untracked 브리프(grok 샌드박스가 `/tmp`를 못 읽어 리포 루트에 둠). 커밋 대상 아님.

당신은 Loom dogfood 룸의 **grok-impl 구현자**다. SessionStart hook이 매 세션 프롬프트 캐시를
깨뜨리는 **원인 A(한 이벤트에 hook 커맨드 2개 → 완료 순서 concat = 순열 뒤집힘)** 를 코드로
소거하는 카드다.

## ① 스펙 SSOT (착수 전 정독 필수)

**정본은 `docs/spikes/HOOK-CACHE-FIX-DESIGN.md`다.** 특히 **§2 M-1**(구현 개요 1~4·예산 산술),
**§4-bis**(본세션 판정 — 「단일 커맨드 병합 + 구조 분리 + 결정론 검증」), **§6 V-1/V-2**(검증),
**§7-bis (f)**(구현 범위·센티널 주의), **§8-2**(SOFT_CAP 근거)를 그대로 따르라. 원인 규명 배경은
`docs/spikes/WARM-BASE-FORK-SPIKE.md`. 문구·좌표가 어긋나면 설계 정본이 우선한다.

대상 파일은 **`scripts/session-context.ts` · `package.json`(스크립트 별칭) · 신규 유닛 파일**뿐이다.

## ② 배경 실측 (재조사 불요 — 그대로 전제로 삼아라)

- hook 결과 캡은 **커맨드당 10,000자**(총량 아님). 초과 시 조용한 절단이 아니라 **디스크 영속화 +
  2,000자 프리뷰 + 세션마다 바뀌는 UUID 2개가 박힌 경로 봉투**로 치환된다 → 영구 캐시 미스 + 왕복 부활.
  **캡 초과는 실패로 취급한다.**
- 현재 `HARD_CAP = 9500`(`session-context.ts:17`)의 `truncateContext()`는 **조용히** 자른다.
  lessons 블록 실측 9,399자 = 여유 101자 → 인덱스 1줄(중앙값 237자) 추가 시 즉시 조용한 손실.
- `.claude/settings.json:14-27` SessionStart matcher `startup|clear` 하나에 handler 2개
  (`--part state` / `--part lessons`). **이 파일은 이번 카드 범위 밖이다(⑥ 참조).**

## ③ 구현 항목 (D1~D5)

### D1 — `buildAllContext()` + `--part all` 추가

```
buildAllContext() = [buildStateContext(), buildLessonsContext()].join("\n\n---\n")
```

- 구분자는 **현행 하네스가 hook 결과 사이에 넣는 구분자와 동형**(`\n\n---\n`)으로 둔다 —
  문서 형태 변화를 최소화해 기존 앵커·폴백을 살린다.
- `parsePart()`(`:85-91`)에 `"all"`을 허용값으로 추가. `emit()`(`:72-83`)이 `all`이면
  `buildAllContext()`를 쓰도록 분기. **`state`/`lessons` 분기는 그대로 둔다**(⑥).
- 순서는 **state → lessons 고정**. 랜덤·조건부 순서 금지(결정론이 이 카드의 본체다).

### D2 — `<details>` 접기 구간을 주입에서 제외 (락 (a)의 필연적 귀결)

오너 확정: lessons 압축 방식 = **구조 분리 — 같은 파일 `<details>`**(이동안 기각). 경위·커밋해시·
재범기록은 `tasks/lessons.md` **안에** `<details>…</details>`로 남고, 인덱스 줄의 **트리거 문구만**
주입된다. 그런데 `buildLessonsContext()`(`:67-70`)는 파일 **전문**을 읽으므로, 접기만으로는
**바이트가 1자도 줄지 않는다.**

따라서 `buildLessonsContext()`는 파일을 읽은 뒤 **`<details>` 여는 태그부터 대응하는 `</details>`
닫는 태그까지(태그 포함)를 제거**한 결과를 주입한다.

- 제거는 **줄 기준**으로 하라 — `<details`로 시작하는 줄부터 `</details>`로 끝나는 줄까지 삭제,
  남은 빈 줄은 1개로 정규화. 정규식 lazy 매칭으로 한 줄 안에서 잘라내는 구현은 피하라(중첩·미닫힘
  내성이 없다).
- **미닫힘 `<details>`**(닫는 태그 없음)를 만나면 그 지점 이후를 통째로 버리지 말고 **해당 블록만
  버리되 경고 마커를 붙여라**(D3의 라우드 원칙 동일 적용). 조용히 삼키는 경로를 새로 만들지 마라.
- 원본 파일은 **수정하지 않는다** — 순수 함수 필터다. `tasks/lessons.md` 문안 작업은 별도 카드
  (HOOKCACHE-C)가 담당한다.

### D3 — `truncateContext()` 조용한 절단 → **라우드 마커**

- 예산 초과 시 **블록 맨 앞**에 경고 줄을 붙이고 **그 다음에** 절단한다. 형식:

  ```
  ⚠ [LOOM-SESSION-CONTEXT] 예산 초과 — N자 초과, tasks/lessons.md 정리 필요
  ```

  맨 앞이어야 하는 이유: 절단은 꼬리에서 일어나므로 꼬리 공지는 절단으로 함께 사라질 수 있고,
  에이전트가 블록 첫 줄에서 즉시 인지해야 한다. 기존 꼬리 `…[truncated N chars …]` 공지는
  **유지해도 되나, 앞 마커가 없어지는 일은 없어야 한다**(마커 길이를 keep 계산에 반영하라).
- 경고 문구는 **사실 서술형**으로 쓴다 — 명령형 주입문은 프롬프트-인젝션 방어에 걸릴 수 있다
  (lessons platform (18)③).

### D4 — `SOFT_CAP` + `session-context:lint` 신설

- `SOFT_CAP = 8500` 상수 추가(캡 10,000 대비 15% 여유. 근거는 공학적 판단이지 실측이 아님 — 설계 §8-2).
  `HARD_CAP = 9500`은 **런타임 절단 임계로 존치**한다.
- `scripts/session-context.ts`에 `--lint` 모드 추가. 동작: `--part all` 산출물(=D2 필터 적용 후)의
  `additionalContext.length`가 **`SOFT_CAP` 초과면 비영 종료**(초과분·현재 길이를 stderr에 보고),
  이하면 0 종료.
- `package.json` `scripts`에 `"session-context:lint": "bun run scripts/session-context.ts --lint"` 추가.
  **선례는 바로 위 줄 `"handoff:lint": "bun run scripts/session-status.ts --lint"`(23행)** — 형식을 맞춰라.
- **락 — fail-open 예외**: `import.meta.main` 블록(`:93-105`)의 `catch { process.exit(0) }` fail-open은
  **hook 주입 경로 보호용**이다. **`--lint` 경로는 fail-open에서 제외**하라 — 린트가 예외를 삼키고
  0으로 끝나면 커밋 게이트가 무의미해진다. hook 경로(`--part …`)의 fail-open 거동은 **무변경**.

### D5 — 센티널 문자 그대로 보존 (락)

`AGENTS.md`의 무훅 폴백 분기가 **`[LOOM-SESSION-CONTEXT` 센티널 문자열**에 의존한다(센티널이 보이면
①status·②HANDOFF·③lessons 인덱스 읽기를 생략, 없으면 v2 표 전체가 정본). 병합 블록에
`[LOOM-SESSION-CONTEXT v1 · state]`와 `[LOOM-SESSION-CONTEXT v1 · lessons]` **두 센티널을 문자 그대로,
현행 표기 그대로 유지**하라. 이름·공백·`·`(U+00B7) 한 글자라도 바꾸면 폴백이 **조용히** 깨진다.

## ④ 범위 밖 — 절대 손대지 마라

- **`.claude/settings.json` handler 2→1 컷오버는 이번 카드에 포함하지 않는다.** 병합 블록이 예산에
  들어온 것(HOOKCACHE-C 완료 + `session-context:lint` green)을 확인한 뒤 **별도 컷오버**한다.
  지금 컷오버하면 13,497자 → 캡 초과 → 영속화 봉투로 상황이 더 나빠진다.
- **`--part state` / `--part lessons`는 진단용으로 존치**한다. 제거·개명 금지 — 기존 테스트·스파이크
  문서 앵커가 여기에 걸려 있다.
- `tasks/lessons.md` 및 `tasks/lessons/*.md` **문안 수정 금지**(HOOKCACHE-C 담당).
- `scripts/session-status.ts`·`handoff:lint` 무변경. `packages/` 일체 무변경.

## ⑤ 검증 (V-1 · V-2 + 린트)

신규 유닛 파일(예: `scripts/session-context.test.ts` — `bun test`가 리포 전역 `*.test.ts`를 잡는다.
`bunfig.toml` preload가 걸려 있으니 별도 설정 불요)에 아래를 전수 고정하라.

- **V-1 sha 결정론**: `--part all` 산출물을 **20회** 생성 → sha256 **20/20 동일**.
  합격 기준 = 20/20. 커맨드가 1개면 순열이 1개라는 것은 구조적 사실이나, 회귀 방지용으로 유닛에 못박는다.
  (자식 프로세스 20회 spawn이 느리면 `buildAllContext()`를 직접 20회 호출하는 형태도 허용 — 단
  "동일 입력 → 동일 출력"이 실제로 검사돼야 한다.)
- **V-2 센티널 보존**: 산출물에 `[LOOM-SESSION-CONTEXT v1 · state]`와
  `[LOOM-SESSION-CONTEXT v1 · lessons]`가 **문자 그대로** 각 1회 이상 존재. 하드코딩 리터럴로 어서션할 것
  (상수를 참조하면 상수 개명 사고를 못 잡는다).
- **캡 준수**: `--part all` 길이 < 10,000(하드). `SOFT_CAP` 대비 현재 길이를 **테스트가 아니라 회신에
  수치로 보고**하라 — HOOKCACHE-C 완료 전에는 SOFT_CAP 초과가 정상이므로 이것을 실패 어서션으로
  걸지 마라(⑦ 참조).
- **린트 거동**: 인위적으로 부풀린 입력에 대해 `--lint`가 **비영 종료**. 예산 이하 입력에 대해 0 종료.
- **`<details>` 필터(D2)**: `<details>` 블록이 산출물에서 사라지고 그 **바깥의 인덱스 줄은 보존**됨 ·
  중첩/미닫힘 입력에서 크래시 없음(+ 미닫힘은 경고 마커).
- **라우드 절단(D3)**: `HARD_CAP` 초과 입력에서 경고 마커가 **블록 첫 줄**에 존재.
- **fail-open 무회귀**: `--part` 누락/미지값 → stdout 없이 exit 0(현행 거동 유지).

**실행 커맨드 (결과를 회신에 수치로 포함):**

1. `env -u LOOM_RELAY_TOKEN -u LOOM_RELAY_URL bun test scripts/` (또는 신규 파일 지정) — 신규 유닛 green.
2. `bun run --filter '*' typecheck` — 전 패키지 green.
3. `bun run session-context:lint` — **현재는 실패가 예상 정상**(⑦). 종료코드와 보고된 길이를 그대로 회신.

**전체 `bun test`는 아키텍트 몫이다 — 네가 돌리지 마라.** 신규 유닛 + typecheck까지만 하고, 중도
중단되거나 통과 못 한 항목이 있으면 **정직하게 보고**하라(성공으로 포장 금지 — DONE 마커도 보류하고
사유를 적어라).

## ⑥ 적용 순서상의 위치 (혼동 방지)

설계 §4 적용 순서: ① 인덱스 다이어트(문안, = HOOKCACHE-C) → ② `--part all` + 린트 추가(**= 이 카드**)
→ ③ settings handler 2→1(별도) → ④ V-3 라이브 캐시 실측(별도).
**이 카드는 ②만 한다.** ①이 아직 안 끝났을 수 있으므로 `session-context:lint`가 지금 빨간 것은
**결함이 아니라 설계된 신호**다. 린트를 통과시키려고 `SOFT_CAP`을 올리거나 `tasks/lessons.md`를
건드리지 마라 — 그것이 이 카드의 가장 흔한 실패 방식이다.

## ⑦ 커밋/푸시 금지 (표준)

**검증까지만 수행하고 커밋·푸시는 하지 마라. ship 여부는 아키텍트/오너 결정이다.**
`git add`/`git commit`/`git push`/브랜치 조작 일절 금지. 작업 트리에 변경만 남겨두면 된다.

## ⑧ 완료 신호

모든 작업 후 **최종 줄**에 정확히 다음 형식의 마커를 출력하라:

```
[HOOKCACHE-A-DONE] units=<통과>/<전체> typecheck=<통과>/<전체> all-len=<자수> lint=<pass|fail(expected)>
```

예: `[HOOKCACHE-A-DONE] units=9/9 typecheck=6/6 all-len=9204 lint=fail(expected)`.
실패가 있으면 실제 수치로 정직하게 보고하고, 무엇이 왜 실패했는지 마커 위에 짧게 적어라.
