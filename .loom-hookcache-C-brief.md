# [HOOKCACHE-C] lessons 인덱스 문안 재작성 — 구조 분리(`<details>`)로 예산 다이어트

> 저장소 내부 untracked 브리프. 커밋 대상 아님.

당신은 **문안 작업 레인(claude pane 카드)** 이다. `tasks/lessons.md`의 **인덱스 32줄**을
**같은 파일 `<details>` 구조 분리**로 재작성해, SessionStart 주입 블록을 예산 안으로 되돌린다.
제품 코드는 건드리지 않는다 — **이 카드가 손대는 파일은 `tasks/lessons.md` 하나뿐**이다.

## ① 스펙 SSOT

정본은 `docs/spikes/HOOK-CACHE-FIX-DESIGN.md` — **§2 M-1(3항 인덱스 예산·예산 산술)** ·
**§4-bis(구조 분리 = 무손실 압축)** · **§6 V-4(리추얼 회귀 검수)** · **§7-bis (d)(작업 부담 실측·상위 8줄)**.
`tasks/lessons.md` 상단 **"이 파일 사용 규칙 (3줄)"** 도 반드시 읽어라 — 인덱스의 존재 이유가 거기 있다.

**오너 확정(락)**: lessons 압축 방식 = **구조 분리 — 같은 파일 `<details>`**.
M-1 원안의 **"카테고리 파일로 이동"은 기각**됐다. `tasks/lessons/<category>.md`로 내용을 옮기지 마라.

## ② 왜 하는가 (한 문단)

SessionStart hook 출력은 **커맨드당 10,000자 캡**이고, 초과하면 조용한 절단이 아니라 **세션마다 바뀌는
UUID 경로가 박힌 영속화 봉투**로 치환된다 → **영구 캐시 미스 + 왕복 부활**. 현재 lessons 블록은
9,399자로 자체 `HARD_CAP 9500` 대비 **여유 101자** — 인덱스 1줄(중앙값 237자)만 늘어도 **조용한 정보
손실이 즉시 시작**된다. 이 카드는 그 벽을 없앤다.

## ④ 목표 예산

전 줄에 120자 상한을 그냥 적용하면 합이 3,836자로 목표를 258자 초과한다 — 상위 8줄을 120자보다
더 짧게 잡아 총량을 맞춰라.

## ⑤ 작업 규칙 (락)

### 규칙 1 — 트리거 문구는 **절대 삭제 금지**

각 인덱스 줄의 존재 이유는 **"어떤 작업 유형에서 이 카테고리 파일을 열어야 하는가"** 를 알려주는 것이다
(파일 상단 사용 규칙 2). 트리거가 사라지면 리추얼이 **실질 붕괴**한다.

- 남길 것: **트리거(작업 유형/상황) + 규칙의 알맹이(무엇을 하라/하지 마라)**.
- 접을 것: **경위·서사·커밋해시·재범 기록·실증 날짜·좌표(`file.ts:123`)·번호 세부(①②③ 나열)**.
- 판단이 서지 않으면 **남기는 쪽**으로 기울여라. 예산을 8자 넘기는 것보다 트리거를 잃는 것이 훨씬 나쁘다.

### 규칙 2 — 접기는 **같은 파일 `<details>`**, 이동이 아니다

정보를 삭제하지도, 다른 파일로 옮기지도 않는다. **무손실 구조 분리**다.

- 각 인덱스 줄 **바로 아래**에 그 줄의 상세를 담는 `<details>` 블록을 둔다.
- 형식은 아래로 **통일**하라(여는 `<details>`와 닫는 `</details>`는 **각각 독립된 줄**에 둘 것 —
  주입 필터가 줄 기준으로 동작한다):

```markdown
- [platform] 2026-07-19 (15) WSL 노드 브릿지 5함정: 데몬 SSH 동반사망·PATH 미소싱·root 거부·config 재주입·분류기 독립 레이어.
<details><summary>경위·좌표</summary>

① 데몬도 SSH 종료 동반 사망 — `setsid nohup … &`. ② 비로그인 셸 `~/.bashrc` PATH 미소싱 …
(원문 그대로 이어붙임)

</details>
```

- **`<details>` 안에는 원문을 그대로 넣어라** — 다시 요약하지 마라. 요약하면 무손실이 아니게 되고,
  이 방식을 택한 이유가 사라진다.
- 중첩 `<details>` 금지. `<details>`를 열고 닫지 않는 일이 없도록 하라(미닫힘은 주입 필터에서 경고를 낸다).
- 접을 내용이 없는 짧은 줄(이미 120자 이하)은 `<details>`를 만들지 마라 — 빈 껍데기 금지.

### 규칙 3 — 손대지 말 것

- 파일 상단 **제목·"이 파일 사용 규칙 (3줄)"·카테고리 목록 줄·`## <category>` 섹션 헤딩** — 무변경.
- **줄의 순서·카테고리 소속·번호(`(n)`)·날짜** — 무변경. 재정렬·재번호 금지.
- **`tasks/lessons/*.md` 5개 파일** — 이번 카드에서 무변경(이동안 기각).
- `scripts/` · `packages/` · `.claude/settings.json` — 무변경.

## ⑥ 착수 순서

1. **상위 8줄 먼저**(⑤ 규칙대로 접기) → 총량 재측정. 여기서 이미 ~2,726자가 회수된다.
2. 남은 24줄 중 120자 초과분을 긴 순서로 처리.
3. 총량이 ≤3,600자에 들어오면 중단하라 — **필요 이상으로 깎지 마라**. 예산을 맞춘 뒤 더 깎는 것은
   트리거 판별력만 갉아먹는다.

## ⑦ 자체 검수 게이트 V-4 (필수 — 생략 시 카드 미완료)

재작성 완료 후, **32줄 전부**에 대해 다음을 스스로 검사하라:

> **"이 줄 하나만 보고, 내가 지금 하려는 작업이 이 카테고리 파일을 열어야 하는 작업인지 판별할 수 있는가?"**

- **합격 기준: 32/32.**
- 판별 불가한 줄이 있으면 **그 줄 목록을 회신에 그대로 보고**하라(줄 번호 + 원문 + 왜 판별이 안 되는지).
  임의로 되돌리거나 얼버무리지 말 것 — 아키텍트가 판정한다.
- 이 검수는 **다이어트 후** 텍스트로 하라. 원문을 기억한 채로 보면 통과처럼 보인다 — 원문을 참조하지 말고
  새 줄만 읽고 판정하라.

**정직 기술 요구**: 이 다이어트가 교훈 재범률을 올리지 않는다는 근거는 **현재 없다**(리포에는 이미
교훈 재범 2회 실증이 있고, 그것은 "인라인 세부가 있어도 재범했다"는 뜻이지 "세부를 빼도 안전하다"의
증거가 아니다). 이것이 이 작업의 최대 미검증 리스크이며 V-4는 사후 검사밖에 못 한다.
**애매한 줄을 통과시키지 말고 보고하라.**

### ⑦-bis 최종 판별력 검수는 **아키텍트가 fable-advisor로 수행한다** (네 몫이 아니다)

오너 지시로 이 카드에는 **fable 자문 레인**이 붙는다. 단 `fable`은 herdr pane agentKind가 아니고
(enum = claude/codex/grok), fable-advisor는 규약상 **자문 전용·never implements**다. 따라서 분업은:

- **너(claude pane)** = 문안 재작성 + ⑦ V-4 자체 게이트까지. 여기까지가 DONE 조건이다.
- **아키텍트** = 네 산출물을 받아 `fable-advisor`(model: fable, read-only)에게 **트리거 판별력·압축 밀도
  유지선**을 검수시킨다. 이때 락 (a)가 지켜졌는지(트리거 보존·`<details>` 안은 원문 그대로)도 함께 본다.

그러니 **fable-advisor를 네가 직접 스폰하지 마라.** 대신 검수자가 판단할 수 있도록, ⑧ 수치와 V-4 결과,
그리고 **네가 애매하다고 느낀 줄**을 숨기지 말고 그대로 올려라 — 통과처럼 보이게 다듬는 것이 가장 나쁘다.

## ⑧ 회신에 반드시 포함할 수치

- 인덱스 총량: **before → after** (자수)
- 줄당 최대/중앙값: after
- 120자 초과로 남은 줄 수 + 그 줄 목록(있다면 사유)
- 접은 `<details>` 블록 수
- V-4 결과: `N/32` + 판별 불가 줄 목록
- 파일 전체 자수: before → after (`<details>` 포함이므로 **총량은 늘어난다** — 정상이다.
  줄어드는 것은 **주입되는 양**이다)

## ⑨ 인접 카드와의 관계 (충돌 방지)

- **HOOKCACHE-A**(코드 축)가 `session-context.ts`에 `<details>` 구간을 주입에서 **제외하는 필터**를
  넣는다. 그 필터가 들어오기 전까지는 이 카드의 작업이 **주입량을 줄이지 않는다** — 정상이다.
  예산 검증(`bun run session-context:lint`)은 A 완료 후에 의미가 생긴다.
- **HOOKCACHE-B**(claude-mem 패치)가 같은 파일 인덱스에 **`platform` 카테고리 1줄을 append** 할 수 있다.
  충돌하면 멈추고 보고하라 — 상대 줄을 지우거나 덮어쓰지 마라.

## ⑩ 검증 커맨드

리포 테스트를 돌릴 일이 생기면 반드시:

```
env -u LOOM_RELAY_TOKEN -u LOOM_RELAY_URL bun test
```

단 **전체 스위트 회귀 판정은 아키텍트 몫**이다 — 네가 전체 스위트를 돌려 판정하지 마라. 중도 중단되면
**정직 보고하고 DONE 마커를 보류**하라(성공으로 포장 금지). 이 카드는 문서 파일 1개만 바꾸므로
테스트 영향은 원칙적으로 없다.

## ⑪ 커밋/푸시 금지 (표준)

**검증까지만 수행하고 커밋·푸시는 하지 마라. ship 여부는 아키텍트/오너 결정이다.**
`git add`/`git commit`/`git push`/브랜치 조작 일절 금지. 작업 트리에 변경만 남겨두면 된다.

## ⑫ 완료 신호

모든 작업 후 **최종 줄**에 정확히 다음 형식의 마커를 출력하라:

```
[HOOKCACHE-C-DONE] index=<before>→<after>자 max-line=<자수> v4=<N>/32 folded=<블록수>
```

예: `[HOOKCACHE-C-DONE] index=8633→3542자 max-line=119 v4=32/32 folded=27`.
V-4 미달이 있으면 실제 수치로 정직하게 보고하고, 판별 불가 줄을 마커 위에 목록으로 적어라.
