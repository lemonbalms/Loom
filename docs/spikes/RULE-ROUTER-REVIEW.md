## 2. §8 일곱 질문에 대한 답 (확정)

### 2-1. P2 대응은 성립하는가 — **조건부로만 성립. 반례 실재.**

반례: `RULE-ENFORCEABILITY` §5.1 **A1(M-1 dispatcher allowlist, H·L0)의 deny는
log+ignore(무성)**이다. 2026-07-20에 모델이 규칙을 모른 채 `mac-node`로 디스패치 → 무성
deny → 침묵을 대기로 오독 → 오너 오진 보고가 **실제 발생**했다. H는 위반 *효과*를 차단하지만
모델의 무지는 못 고치고, deny가 무성이면 세션이 진단 불능에 빠진다.

따라서 "H → 라우팅 1순위"는 다음 두 조건을 만족하는 유닛에만 적용한다.

1. **deny 가시성** — 가드의 deny가 구조화·교육적(위반한 규칙의 좌표를 반환)이어야 한다.
   그때 deny 자체가 JIT 전달 채널이 된다. **무성-deny H 유닛은 `pin: true` 기본값**
   (deny 가시화가 배선되면 해제 — 기계 판정 가능한 조건이므로 재리뷰 불요).
2. **X2 범위 커버** — H 주장 범위가 좁은 유닛(예: A3 = Claude Agent/Task 경로 한정)은
   무지 상태의 모델이 **범위 밖 등가 경로**(Bash 직접 호출 등)로 샐 수 있다. 좁은-범위 H는
   범위 밖 경로가 열거·차단되기 전까지 라우팅 대상이 아니다.

레지스트리 요건: 유닛별 `guard: { deny_visible: bool, scope: … }` 필드 필수(§4 delta ①).

### 2-2. pinned 초기 판정 권한 — **3원천 분해.**

| 원천 | 판정 | 성격 |
|---|---|---|
| J등급 → pinned | **자동** (레지스트리 규칙) | 판단 아님 — P2 정의 그대로 |
| 무성-deny H → pinned | **자동** (2-1의 파생) | 판단 아님 — 가드 가시성 필드에서 기계 파생 |
| 그 외 재량 pin (예: §5.1 예시 A등급 `traps.card-done`) | **오너 선포 + 리뷰어 감사** | D3 default(리뷰어 승인)와 정합 |

### 2-3. JIT의 타 하네스 범위 — **범위 선언으로 해소. 타 하네스 P-A는 명시적 미해결.**

A3 선례와 동형으로 라우터의 주장 범위를 **Claude Code 하네스 한정**으로 선언한다.
Codex/Grok/직접 CLI는 capability 부재 = **`NOT_APPLICABLE` → 전량(리추얼)** — 기존 NORMS
정책(Codex 미실측 fail-closed · Grok ritual-only)과 같은 형태다. 타 하네스의 P-A는 이 제안이
해소하지 않으며, §11 미확정 목록에 그대로 남긴다(가짜 커버리지 주장 금지).

### 2-4. G1의 J-miss=0은 못-실패 게이트인가 — **부분적으로 그렇다. 재배치 필요. (이 리뷰의 핵심 발견)**

J 유닛은 pinned라서 라우팅 자체가 없다 → **J-miss=0은 구조상 자동 성립**한다. 성능 지표가
아니라 **pin 배선의 contract test**다. 더 나쁜 순환이 있다: miss 가중치가 *주장된* 등급을
따르므로, **J여야 할 규칙이 H로 오분류되면 그 miss는 저가중 H-miss로 계상되어 그물을
빠져나간다** — 등급 오류가 등급 가중 지표로는 보이지 않는다.

처방(§4 delta ②):

1. J-miss=0은 **contract test로 재배치** (pin 경로 배선 검증).
2. 실성능 게이트 = **사전등록된 위험가중 recall 임계(라우팅 대상 G/A 유닛 대상)**.
3. **강한 positive(실제 위반이 난 턴)의 miss는 등급 무관 → 해당 유닛 등급 재감사 트리거.**
   등급 오분류가 빠져나가는 유일한 구멍을 여기서 막는다.

### 2-5. R{n}인가 spike REVIEW인가 — **spike REVIEW (이 문서).**

선례 `SESSION-START-UNIFIED-*`(하네스 규범 → spike REVIEW)와 일치하고 제품 코드 변경이
없다. D2 default 유지. 오너가 D2에서 달리 정하면 그때 `plan_review.md`로 승격한다.

### 2-6. α 해석 — **리뷰어가 확정 불가. 오너 확인을 Phase 2b 선결로.**

"그래프+α = 그래프 순회 + 보조 스코어 신호" 독해는 합리적이라고 판정하지만, 오너 발화의
해석 문제라 리뷰어가 확정할 수 없다. **Phase 2b 착수 전 오너 확인 필수**(B 정의가 M4 비용
추정을 바꾼다). Phase 1은 B 정의와 무관하므로 비차단.

### 2-7. M7 임계 사전등록 주체 — **딜레마는 "결과"의 정의 확정으로 해소.**

"결과 관측 전" = **replay 측정 전**이다. 레지스트리 통계는 결과가 아니라 **입력**이므로,
Phase 1 종료 후(레지스트리 규모 가시) · Phase 2 replay 실행 전에 고정하면 자기평가도
임의값도 아니다. 절차:

> Phase 1 종료 시 **설계자가 레지스트리 통계 기반 근거와 함께 임계 제안** → **리뷰어 승인**
> → **커밋으로 봉인**(git 순서가 사전등록을 검증 가능하게 만든다) → 그 후에만 Phase 2 replay.

D7 default(리뷰어 승인)와 정합.
