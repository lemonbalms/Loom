## 1. Objective

브릿지는 claude 워커의 상태 전이(승인 대기 · 턴 종료 · 작업 시작)를 지금 pane 스크레이프 글자 패턴으로만
짐작한다 — 승인 대기를 자주 놓치고(가짜 `agent_blocked`), still-running 유예는 최대 5분 휴리스틱에 의존한다.
이 PLAN은 스파이크가 기술 가능성을 확정한 **최소 배선 5단계**를 그대로 구현한다: 스폰 시 hook 주입(D2) →
워커 hook이 로컬 소켓 write → 브릿지가 소켓 리스너로 watch(D4) → 수신 이벤트를 `flight.hookHint`에 적재 +
우선 분기(D3) → hook 부재 시 폴백(D5). 여기에 **최소 계측(D6)**과 **VERSION 0.26.0 동기(D7)**를 더한다.
claude Code hook(`Stop` · `Notification` matcher `permission_prompt`/`idle_prompt` · `UserPromptSubmit`)이
상태 전이를 브릿지-로컬 소켓으로 무전하면 그것을 **완료 힌트로만** 쓴다.

**완료 정의:** ① 스폰 시 claude argv에 hook 주입(D2) · ② 브릿지-로컬 0600 소켓 리스너가 attempt(seq)-스코프로
hook 이벤트 수신(D4) · ③ `flight.hookHint` 적재 + `onCardHerdrEvent`·still-running poll 우선 분기(D3) ·
④ hook 부재/no_listener/오류 시 현행 경로 폴백(D5) · ⑤ loomDir 하위 JSONL 계측 append(D6) · ⑥ VERSION
0.26.0 동기(D7) — 이 6부가 배선되고, PLAN §0.26.0 테스트 계획 8행이 전부 신규 유닛으로 고정되며, 6패키지
typecheck green. **커밋/푸시 금지 — 검증까지만.**

**D1~D7 요약(정본 요약 — 계약 전문은 §4):** **D1** 스코프 = 스파이크 최소 배선 5단계 · **claude 워커 1종만**
(codex/grok 어댑터·스크레이프 폐기·모바일 푸시·moshi API 전부 비목표) · **D2** 주입 = `agentArgv.claude`에
`--settings '<인라인 JSON>'` 세션별 override(워커별 소켓 경로 격리) 우선 · env 채널 보조 · hook handler는
`async:true` 논블로킹 순수 사이드이펙트 · **D3** 힌트 전용 — `flight.hookHint` 적재 후 딱 두 지점(a: `onCardHerdrEvent`
`bridge-runtime.ts:1945` b: still-running poll)의 우선 분기 입력에만 · **자동 close·approve 절대 금지** · **D4**
브릿지-로컬 0600 소켓 리스너(브릿지=서버, 방향 역전) + `isPathUnderLoomDir` 가드 + attempt(seq)-스코프 + 생명주기
정리 · 비정상 페이로드 조용히 무시(카운터만) · **D5** 폴백 무회귀(fail-open) — hook 부재/no_listener/오류 시 현행
herdr status + 스크레이프 + 5분 유예 그대로 · **D6** loomDir 하위 JSONL 계측(메타데이터-only) · **D7** VERSION 0.26.0
동기.

## 3. Interfaces

### `flight.hookHint` 단일 슬롯 자료형 (M-2 ①)
`flight.hookHint`는 **최신 hook 이벤트 1개만** 보유하는 단일 슬롯(last-event-wins)이며 **누적하지 않는다**.
자료형은 최소 메타(이벤트 종류 + 수신 타임스탬프 + 필요 시 matcher)만 — 예: `{ kind: "Stop" | "permission_prompt" |
"idle_prompt" | "UserPromptSubmit"; at: number }`(정확한 필드는 구현 재량이되 **payload 본문 필드는 담지 않는다** —
§4 L-3·D3 payload 본문성). herdr `working` 재전이·후속 hook 이벤트가 이 슬롯을 **대체·소거**한다(§4 M-2②).

### 소켓 경로 규약 (M-1 · L-1)
`loomDir()/hook-<cardId(sanitize+slice)>-<seq>.sock` — **attempt(dispatch seq)-스코프**. cardId는 `sanitizeRunId`
(`inject-control.ts:20-22`) 동형으로 sanitize + slice하되 **slice는 cardId 부분에만 적용해 `-<seq>` 접미의 유일성을
보존**한다(unix 소켓 경로 macOS ~104B 한계 대비 — L-1). cardId charset은 이미 `TaskIdSchema`(`card-contract.ts:16`
`/^task_[a-f0-9]+$/i`)가 wire 단에서 잠가 셸·경로 주입을 차단하므로(길이만 무상한) 소켓 경로·D2 `--settings` JSON
embed의 안전 근거다. seq는 herdr agent명 `loom-<cardId>-<seq>`의 seq 접미(`bridge-runtime.ts:1058-1064`)와 동형.

### 계측 JSONL 레코드 형태 (L-3 메타데이터-only)
`loomDir()` 하위 append-only JSONL. 레코드는 **이벤트 종류·타임스탬프·카운터만** 기록하고 hook payload 본문 필드
(`last_assistant_message`·`prompt`·`tool_input` 등)는 **미기록**한다(본문성 §5.1 원칙 · 간접 프롬프트 주입 표면 차단 ·
크기 관리). 카운트 대상 4종: (a) hookHint 발화 횟수·이벤트 종류 · (b) 폴백(hook 부재/no_listener) 횟수 ·
(c) `permission_prompt` 감지 횟수 · (d) 가짜 `agent_blocked` 교정 횟수. JSONL 회전/크기 상한은 Unknowns U6 — v1 범위 밖.

### hook 설정(`--settings` 인라인 JSON) 규약 (D2 · D4 스크립트 규약)
`hooks` 3단 중첩(이벤트 → matcher → handler), handler `type:"command"` + `async:true`(논블로킹 순수 사이드이펙트).
배선 이벤트 3종: `Stop` · `Notification`(matcher `permission_prompt`|`idle_prompt`) · `UserPromptSubmit`. **hook 스크립트는
항상 exit 0**(관측은 워커를 막지 않음 — exit 0=정상·1=논블로킹·2=블로킹 일부만) + **stdout 비움**(`UserPromptSubmit`·
`SessionStart` stdout은 Claude 컨텍스트에 주입되는 함정 — 파일/소켓 append만).

### 소켓 리스너 반환/폴백 셰이프
`inject-control.ts`의 `{ ok: true } | { ok: false; reason: "no_listener" | ... }` 유니언을 따르라. hookHint 부재·
소켓 오류·no_listener는 **폴백 신호**로 소비(§4 D5) — 예외로 던지지 말 것.

## 4. Constraints (락 — 위반 금지, 전문 인용)

### D3 락 — hookHint 소비 계약 (R41 M-2 lock — 4항, 전문 전사)

> **hookHint 소비 계약 (R41 M-2 lock — 4항, 스테일 `permission_prompt` 마커가 정상 done을 blocked로
> 오교정하는 회귀 경로 차단):** ① **자료형 = 단일 슬롯 last-event-wins** — `flight.hookHint`는 최신 이벤트 1개만
> 보유하고 **누적하지 않는다**. ② **소거 규칙** — herdr `working` 재전이(`bridge-runtime.ts:1978-1988`) 및 후속
> hook 이벤트가 기존 마커를 대체·소거한다. **툴 승인 *허가*는 배선 3종 이벤트(`Stop`·`Notification`·
> `UserPromptSubmit`) 중 아무것도 발화시키지 않으므로**, 승인 후 정상 완주 시 스테일 `permission_prompt` 마커가
> 잔존하는 것이 기본 경로임을 명기한다(이 소거가 없으면 스테일 마커가 정상 done을 오염). ③ **교정 결정표** —
> `permission_prompt` 마커가 **최신·미소거일 때만** 완료-클래스 판정(`onCardHerdrEvent` done/idle
> `bridge-runtime.ts:2000`·indicator-clear 완료 `:1917`·상한 소진 `:1932`)을 승인-대기(`agent_blocked`)로 교정하고,
> **그 외에는 판정에 무개입**(스테일·소거된 마커는 어떤 교정도 하지 않음 — 정상 done 무교정 보장). ④ **`Stop`×
> still-running AND 결합** — `Stop` 힌트는 poll 스케줄 가속·5분 상한 우회 **입력만**이며, 완료 확정은 indicator-clear
> 스크레이프 확증이 **필수**다(indicator 잔존 중 `Stop` 단독 완료 금지 — U4를 설계로 폐쇄, 잔여는 이득 크기 실측으로 격하).

### D4 락 — 신뢰 경계 소켓 리스너 (R41 M-1 lock — 전문 전사)

> **소켓 식별 단위 = attempt(dispatch seq) 스코프 (R41 M-1 lock):** 소켓 경로는
> `loomDir()/hook-<cardId>-<seq>.sock`(`injectSocketPath` `inject-control.ts:24-26` 패턴 · cardId만이 아니라 herdr
> agent명 `loom-<cardId>-<seq>`의 seq 접미와 **동형**으로 attempt를 스코프한다). cardId 단위(`hook-<cardId>.sock`)는
> 같은 cardId 재디스패치가 **이전 pane 생존 중** 발생함(`bridge-runtime.ts:1058-1064` "Fix 2 (live-measured)" 실측 —
> herdr agent명 seq 접미의 존재 이유)을 근거로 두 attempt가 같은 소켓을 공유하게 해 (a) 구 워커 hook 이벤트의 신
> flight 오귀속(스테일 `Stop`/`permission_prompt`가 새 판정 오염) (b) 브릿지=서버 재-bind EADDRINUSE/unlink 경합을
> **결정적으로** 유발하므로 배제한다(U3(unknown)가 아니라 기존 코드가 실증한 결정 사항). **생명주기 정리 (R41 M-1
> lock):** ① **bind 전 기존 경로 unlink**(브릿지 재시작 시 잔류 스테일 소켓 흡수) · ② **flight 소멸 시
> (`clearStillRunningState`/`flights.delete` 시점과 동기) 리스너 close + 소켓 파일 unlink** · ③ **늦게 도착한 hook
> 이벤트는 flight 동일성 가드로 드롭**(`bridge-runtime.ts:1879,:1907`의 `flights.get(paneId) !== flight` 동형 — 구
> 워커 스테일 이벤트가 새 flight를 오염시키지 못하게 함). U3는 이 설계로 폐쇄되고 **"잔여 정리-경합·unlink 시점 세부
> 실측"으로 축소 재정의**된다. **cardId sanitize+slice (R41 L-1):** cardId는 `sanitizeRunId`(`inject-control.ts:20-22`)
> 동형으로 sanitize+slice하되 **slice는 cardId 부분에만 적용해 seq 접미의 유일성을 보존**한다(unix 소켓 경로 macOS
> ~104B 한계 대비). cardId charset은 이미 `TaskIdSchema`(`card-contract.ts:16` `/^task_[a-f0-9]+$/i`)가 wire 단에서
> 잠가 셸·경로 주입을 차단하므로(길이만 무상한) 소켓 경로·D2 `--settings` JSON embed의 **안전 근거**다.

브릿지가 여는 hook 소켓 리스너는 `inject-control.ts`(로컬 0600 소켓 선례)와 **동형**으로 신뢰 경계를 세운다 — 생성 시
**0600 퍼미션** + 경로는 `isPathUnderLoomDir`(`inject-control.ts:32-35`, `relative` 기반 loomDir 하위 강제) 가드를
통과해야만 watch한다. **방향은 역전 — 브릿지가 서버**(inject-control은 클라이언트, 여기선 브릿지가 리스너로 수신).
**비정상·파싱 실패 페이로드는 조용히 무시**(카운터만 증가 — 로그 스팸·주입 표면 억제, D6 계측이 빈도 관측).

### D5 락 — 폴백 무회귀 어서션 스코프 (R41 L-2 재정의, 전문 전사)

> **무회귀 어서션 스코프 (R41 L-2):** hookHint가 없는 모든 경우 **판정·wire 관측 동작이 0.25.0과 동일**해야 한다 —
> 단 D2가 **모든** claude 스폰 argv에 `--settings` hook 주입을 덧붙이고(hook 미발화 시에도 스폰 표면이 다름) D6 계측
> append도 신규 부작용이므로, "0.25.0과 바이트 동일"은 문자 그대로 성립 불가다. 어서션은 **"스폰 argv hook 주입·계측
> append을 제외하면 판정·wire 관측 동작이 0.25.0과 동일"**로 규정해야 유닛으로 검증 가능한 형태가 된다(무회귀 어서션이
> 검증 계획의 핵심).

hookHint 부재·`no_listener`·소켓 오류 → **현행 herdr status + 스크레이프 + 5분 유예 경로 그대로**(`inject-control.ts:51,99`
`no_listener` 폴백과 동형). 훅 없음 = 현행과 완전 동일(fail-open) — 기존 판정 경로에 hookHint 우선 분기를 **얹기만** 한다.

### 그 외 락 (위반 금지)
- **자동 close·자동 approve 절대 금지(D3 · §2.5.3 규칙 2 · done_proposal 동형):** hook은 워커/훅 스크립트가 emit하므로
  "훅이라서 신뢰"가 성립하지 않는다. hook은 보드/유예 힌트만 — done 확정·승인 결정은 타워 전권 유지. **`Stop`은 done을
  단독 확정하지 않는다**(최종 권위 = 기존 herdr status + 스크레이프 확증).
- **본문 정본 = §5.1 artifact(D3):** hook은 "완료 순간 + 짧은 메시지"에만. `Stop`의 `last_assistant_message`·`prompt`는
  대용량이라 §5.1로 — 결과 회수는 스크레이프 유지, hook payload 본문은 소비·기록하지 않는다.
- **wire·스키마 무변경:** relay wire protocol v1 · conv wire · MCP 도구 · herdr RPC 표면 전부 무변경. 이 PLAN은 판정
  입력(hookHint)만 신설하고 프로토콜·전송 규약은 손대지 않는다.
- **claude 워커 1종만(D1):** codex/grok 어댑터 배선 금지(비목표 — 이득 증명 후). 스크레이프 폐기 금지(공통 눈 존치).
- **hook 스크립트 규약(D4):** 관측 hook은 **항상 exit 0** + **stdout 비움**(`UserPromptSubmit`/`SessionStart` 컨텍스트
  주입 함정 회피 — 파일/소켓 append만).
- **계측 메타데이터-only(D6 · L-3):** JSONL에 payload 본문(`last_assistant_message`·`prompt`·`tool_input`) 미기록.
- **커밋/푸시 금지 — 검증까지만, ship은 아키텍트/오너 결정.** `docs/plan_review.md` 편집 금지. **전체 `bun test`는
  아키텍트 몫**(lessons orchestration (18)) — 워커 deliverable = **신규 유닛 + 구현까지**. 단 아래 §5 검증(신규 유닛·관련
  스위트·6패키지 typecheck)은 `env -u LOOM_RELAY_TOKEN -u LOOM_RELAY_URL` 하에 **직접 실행**하고 결과를 카드 회신에 포함.
- **라이브 스모크는 브리프 범위 밖(아키텍트 후속):** U1(claude 버전별 matcher 지원 편차)·U2(`--settings` 인라인 JSON
  실작동 방식 — 인라인/파일/env 중 채택)의 라이브 실측은 아키텍트가 스폰 후 수행한다. 워커는 **유닛으로 검증 가능한
  형태까지**만 구현하고 라이브 주입/실발화 확인은 시도하지 말 것.
