# orphan durable 룸 정리 — 오너 `!` 커맨드 팩 (2026-07-20)

orphan durable 룸이란 relay가 durable 모드에서 룸을 `create`한 뒤 스냅샷 `persist`
과정이 실패했거나(**R39 L-1** — `docs/plan_review.md:178`), 프로브·테스트로 만들었다가
버려진 멤버-0/무활동 룸이다. GC는 미구현이 확정돼 있고(**D7** — `docs/UNKNOWNS.md:75`,
`room.ts:765-767` `maybeGc` no-op), room delete/rm/prune 같은 삭제 CLI 표면도 전무하다
(grep 0건). 따라서 유일한 정리 수단은 **스냅샷 JSON 파일을 수동 삭제**하는 것이다.
스냅샷 파일명 규칙은 `sha256(roomId)[:16] + ".json"`(`packages/relay/src/persist.ts:67-69`),
저장 위치는 `~/.loom/relay-state/`(같은 파일 `:64`)이며, 최상위 JSON은 `.room.id` /
`.room.name` / `.room.inviteCode` / `.members[]`를 담는다(`:30-58`).

> ## 🛑 절대 불가침 — 아래 2개 파일은 어떤 경우에도 삭제 금지
> - **로컬 mac** `~/.loom/relay-state/693b8a0deca3d739.json` = **loom-local**
>   (`room_32f3322b595456b7`, 10개 프로필 참조 — 현 운영 룸)
> - **Windows** `C:\Users\34970\.loom\relay-state\ef0245c2a460a8b4.json` = **loom-dev**
>   (`room_ca184b781cfdabdc`, 7개 프로필 + 초대 LOOM-GT4B)
>
> 각 §의 삭제 대상 파일명을 실행 직전 눈으로 다시 대조하라. 파일명이 위 2개 중 하나와
> 한 글자라도 겹치면 **중단하고 아키텍트에 회신**.

## 정리 대상 (orphan 3건 — 해시 재계산 대조 완료, profiles/bridge 참조 0건)

| relay | 파일 | roomId | 룸 이름 | 멤버 | 생성 |
|-------|------|--------|---------|------|------|
| 로컬 mac | `23e61ce62e34eaf1.json` | room_8b7dbfc460b2b333 | **d10-roundtrip** | 0 | 07-19 |
| 로컬 mac | `98549b083a206dcd.json` | room_5ac72dcc333f1c9d | **t018** | 2 | 07-11 테스트 |
| Windows | `8d66842a136bcac7.json` | room_d04f4c4832295004 | **winprobe-room** | 2 | 07-19 프로브 |

## ⚠️ 삭제 순서가 중요한 이유 (GC no-op의 함정)

`maybeGc`가 no-op이라 **가동 중 relay는 룸을 메모리에 유지**한다. 파일만 지우면 relay
재시작 전까지 유령 룸이 메모리에 잔존한다(대상 룸은 멤버-0/무활동이라 재-flush로 파일이
되살아나는 일은 없다). **깨끗한 제거 = relay 정지 → `rm`/`del` → relay 재시작.** 이 팩의
§A·§B는 전부 이 순서를 강제한다. 순서를 건너뛰고 파일만 지우지 말 것.

현재 로컬 relay는 bun **pid 45613** 가동 중이다(`relay.pid`의 49293은 stale — 무관).

## §A 로컬 mac relay (오너 Mac 터미널에서 `!` 실행)

### A-1. 삭제 직전 대상 2건의 `room.id`·`room.name`·멤버수 확인 (읽기)

```
! for f in 23e61ce62e34eaf1 98549b083a206dcd; do echo "== $f.json =="; jq -r '"id=\(.room.id)  name=\(.room.name)  invite=\(.room.inviteCode)  members=\(.members|length)"' ~/.loom/relay-state/$f.json; done
```

기대 출력:
```
== 23e61ce62e34eaf1.json ==
id=room_8b7dbfc460b2b333  name=d10-roundtrip  invite=...  members=0
== 98549b083a206dcd.json ==
id=room_5ac72dcc333f1c9d  name=t018  invite=...  members=2
```

**중단 조건:** 어느 파일이든 `name`이 `d10-roundtrip`/`t018`이 아니거나 `id`가 위 표와
불일치하면 **삭제하지 말고 중단** — 파일-룸 매핑이 브리핑과 어긋난 것이므로 아키텍트 회신.
(`jq` 미설치 시 대체: `for f in 23e61ce62e34eaf1 98549b083a206dcd; do echo "== $f =="; python3 -c "import json,sys; d=json.load(open(sys.argv[1])); print('id='+d['room']['id'],'name='+d['room']['name'],'members='+str(len(d['members'])))" ~/.loom/relay-state/$f.json; done`)

### A-2. 로컬 relay 정지

```
! loom relay local stop && loom relay local status
```

기대: `relay local stop: ...` 후 `relay local status: down`.

**중단 조건:** `status`가 여전히 `up`이면 stop이 pid 45613을 잡지 못한 것 — `rm`으로
진행하지 말고 아래로 pid 확인 후 판단(임의 kill 금지, 아키텍트 회신):
```
! loom relay local status; pgrep -fl 'relay --' | grep -v grep
```

### A-3. orphan 스냅샷 2건 삭제

```
! rm -v ~/.loom/relay-state/23e61ce62e34eaf1.json ~/.loom/relay-state/98549b083a206dcd.json
```

기대: 두 경로에 대해 `removed` 출력. **불가침 `693b8a0deca3d739.json`은 이 명령에
포함돼 있지 않음 — 실행 전 명령줄에 그 파일명이 없는지 눈으로 확인.**

### A-4. 로컬 relay 재시작

```
! loom relay local start && loom relay local status
```

기대: `relay local start: started daemon ...`(또는 `already up`) 후 `status: up`.

### A-5. 활성 룸 loom-local 생존 확인 (읽기)

```
! test -f ~/.loom/relay-state/693b8a0deca3d739.json && jq -r '"loom-local SURVIVES: id=\(.room.id)  members=\(.members|length)"' ~/.loom/relay-state/693b8a0deca3d739.json || echo "FATAL: loom-local snapshot MISSING"
```

기대: `loom-local SURVIVES: id=room_32f3322b595456b7  members=...`.
**중단 조건:** `FATAL` 출력 시 즉시 아키텍트 회신(운영 룸 손실).

## §B Windows relay (오너 Mac 터미널에서 `!` — SSH 경유)

접속: `ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113`.
Windows 상시 relay는 **durable + Scheduled Task `LoomRelayTeam`**(AtLogOn·RestartCount5)로
구동되며, 재배포·재시작 절차는 `HANDOFF_WINDOWS.md:246-256,397-401`에 확정돼 있다:
재시작 = `Stop-ScheduledTask -TaskName LoomRelayTeam` → `Start-ScheduledTask -TaskName LoomRelayTeam`.
Task를 멈추면 relay 프로세스가 내려가 §A와 동일한 정지→삭제→재시작 순서가 성립한다.

> Windows 측 relay-state 경로는 `C:\Users\34970\.loom\relay-state\`이다. 아래 커맨드는
> SSH 기본 셸이 **PowerShell**임을 전제로 작성했다(안정적 JSON 확인·파일 조작 위해).
> 만약 접속 셸이 `cmd`면 각 커맨드 앞에 `powershell -NoProfile -Command "..."`로 감싸
> 실행하라.

### B-1. 삭제 직전 대상의 `room.name`·멤버수 확인 (읽기)

```
! ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113 "powershell -NoProfile -Command \"\$p='C:\Users\34970\.loom\relay-state\8d66842a136bcac7.json'; \$d=Get-Content \$p -Raw | ConvertFrom-Json; 'id='+\$d.room.id+'  name='+\$d.room.name+'  members='+\$d.members.Count\""
```

기대: `id=room_d04f4c4832295004  name=winprobe-room  members=2`.

**중단 조건:** `name`이 `winprobe-room`이 아니거나 `id`가 `room_d04f4c4832295004`가
아니면 **삭제하지 말고 중단** — 아키텍트 회신.
(대체: `type` 원문 확인 — `ssh ... "type C:\Users\34970\.loom\relay-state\8d66842a136bcac7.json"` 후 `"name":"winprobe-room"` 문자열을 눈으로 확인.)

### B-2. Windows relay 정지 (Scheduled Task 중단)

```
! ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113 "powershell -NoProfile -Command \"Stop-ScheduledTask -TaskName LoomRelayTeam; Start-Sleep -Seconds 2; (Get-CimInstance Win32_Process -Filter 'Name=\\\"loom.exe\\\" OR Name=\\\"bun.exe\\\"' | Where-Object { \$_.CommandLine -match 'relay' } | Measure-Object).Count\""
```

기대: 마지막 줄이 `0`(relay 프로세스 종료됨). relay는 `loom.exe`가 아니라 **bun 프로세스**로
뜨므로(`HANDOFF_WINDOWS.md:369`) `bun.exe`도 함께 센다.

**중단 조건:** 카운트가 `0`이 아니면 relay가 아직 살아있는 것 — `del`로 진행하지 말고,
치트시트의 강제 종료(`HANDOFF_WINDOWS.md:251-253`)로 잔여 프로세스를 내린 뒤 다시 카운트
확인. 그래도 안 내려가면 아키텍트 회신.

### B-3. orphan 스냅샷 삭제

```
! ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113 "powershell -NoProfile -Command \"Remove-Item -Path 'C:\Users\34970\.loom\relay-state\8d66842a136bcac7.json' -Verbose\""
```

기대: `Removed ... 8d66842a136bcac7.json`. **불가침 `ef0245c2a460a8b4.json`은 이 명령에
없음 — 실행 전 명령줄에 그 파일명이 없는지 눈으로 확인.**
(대체 cmd: `ssh ... "del C:\Users\34970\.loom\relay-state\8d66842a136bcac7.json"`.)

### B-4. Windows relay 재시작

```
! ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113 "powershell -NoProfile -Command \"Start-ScheduledTask -TaskName LoomRelayTeam; Start-Sleep -Seconds 3; try { (Invoke-WebRequest -Uri 'http://127.0.0.1:7842/health' -UseBasicParsing -TimeoutSec 3).Content } catch { 'HEALTH FAIL: ' + \$_.Exception.Message }\""
```

기대: health 응답(`ok`/`auth:true` 포함 JSON).
**중단 조건:** `HEALTH FAIL` 출력 시 `Get-Content $env:USERPROFILE\.loom\relay.err.log -Tail 40`으로
원인 확인 후 아키텍트 회신.

## §C 사후 검증 (양 relay — 읽기 전용)

### C-1. 로컬 mac: orphan 소거 + 불가침 무손상

```
! echo "== local relay-state =="; ls -1 ~/.loom/relay-state/*.json | xargs -n1 basename; echo "-- orphans gone? --"; for f in 23e61ce62e34eaf1 98549b083a206dcd; do test -f ~/.loom/relay-state/$f.json && echo "STILL PRESENT: $f" || echo "removed OK: $f"; done; echo "-- loom-local intact? --"; jq -r '.room.name' ~/.loom/relay-state/693b8a0deca3d739.json
```

기대: `removed OK` × 2, 마지막 줄 `loom-local`. `STILL PRESENT`가 뜨면 삭제 미반영.

### C-2. Windows: orphan 소거 + 불가침 무손상

```
! ssh -i ~/.ssh/id_ed25519_loom_windows 34970@100.65.103.113 "powershell -NoProfile -Command \"\$d='C:\Users\34970\.loom\relay-state'; Get-ChildItem \$d -Filter *.json | Select-Object -ExpandProperty Name; '-- orphan gone? --'; if (Test-Path (Join-Path \$d '8d66842a136bcac7.json')) {'STILL PRESENT'} else {'removed OK'}; '-- loom-dev intact? --'; (Get-Content (Join-Path \$d 'ef0245c2a460a8b4.json') -Raw | ConvertFrom-Json).room.name\""
```

기대: `removed OK`, 마지막 줄 `loom-dev`. `STILL PRESENT`가 뜨면 삭제 미반영.

### C-3. Loom 레벨 활성 2룸 무손상 최종 확인

```
! loom relay local status; loom rooms 2>/dev/null || loom room list 2>/dev/null
```

기대: 로컬 relay `up`, 활성 룸 목록에 **loom-local**·(Windows 바인딩 경유) **loom-dev**가
정상 존재. orphan 3룸(d10-roundtrip·t018·winprobe-room)은 목록에 없어야 함.
