# [ENFORCEABILITY] 규칙 강제가능성 판별 기준 — 설계 카드 (codex 레인, 설계만)

> 저장소 내부 untracked 브리프. 커밋 대상 아님.

당신은 **codex 설계자**다. 이 프로젝트의 운영 규칙을 **"코드로 강제할 수 있는 것"과 "그럴 수 없는
것"으로 가르는 판별 기준**을 설계한다. **제품 코드를 구현하지 마라 — 산출물은 설계 문서 하나다.**

## ⓪ 왜 이 카드가 있는가

오너 지적(2026-07-20): *"규칙으로 넣어도 작동이 그때그때 다르잖아. 코드로 명문화하는 방법을
찾아야지."*

이 프로젝트는 규칙을 `CLAUDE.md` · `AGENTS.md` · `docs/DOGFOOD_LOOP.md` · `tasks/lessons.md`에
글로 적어 왔다. 그런데 **글로 적힌 규칙은 지켜지지 않는다**는 것이 오늘 하루 세션에서
통계적으로 드러났다. 아래가 그 원자료다. **가설이 아니라 같은 세션에서 관측된 사실이다.**

## ① 오늘의 원자료 (이것이 설계 근거다)

### A. 코드로 강제된 규칙 — **3/3 전부 위반을 막았다**

| 규칙 | 강제 수단 | 무슨 일이 있었나 |
|---|---|---|
| M-1 디스패처 allowlist | 제품 런타임 검사(브릿지) | 아키텍트가 `mac-node` 신원으로 카드 3장을 디스패치 → 브릿지가 3장 전부 `M-1 deny`. 아키텍트는 그제서야 로스터상 정식 디스패처가 `claude-impl`임을 확인했다 |
| L-2 result.node ↔ card assignee 정합 | 제품 런타임 검사(`applyCardResult`) | 죽은 pane에서 긁힌 **가짜 `card.done`**(내용이 codex TUI 초기 화면)이 보드를 `done`으로 오염시키려 했으나 거부됨 |
| Orchestration 규칙 2 (Agent `model` 명시 필수) | **Claude Code PreToolUse hook** (`scripts/hooks/check-agent-model.ts`) | 아키텍트가 `model` 없이 `Agent`를 스폰하려 하자 hook이 차단. 에러 메시지가 규칙 출처(`CLAUDE.md` 규칙 2)와 기본값(`opus`)까지 안내했다 |

### B. 글로만 적힌 규칙 — **4/4 전부 위반됐다** (전부 같은 세션, 같은 아키텍트)

| 규칙 | 적힌 곳 | 위반 양상 |
|---|---|---|
| 위임 시작 전 `orchestration` 스킬 로드 | `CLAUDE.md` 규칙 4 | 조사·브리프 위임을 먼저 시작. 오너가 "오케스트레이션도?"라고 지적해서야 로드 |
| 디스패치 전 board claim(`doing`+assignee) | `docs/DOGFOOD_LOOP.md` §1.1 | 통째로 생략. 오너가 "도그푸딩을 왜 안 사용해?"라고 지적한 뒤에야 수행 |
| 실작업은 herdr pane 카드가 기본, in-harness Agent는 폴백 | `tasks/lessons.md` | 조사·브리프를 in-harness Agent로 처리. 오너가 "codex, grok, fable pane 모드로 작업하라고 했는데?"라고 지적 |
| 마커 감시 시 **echo 오탐 주의** | `tasks/lessons.md` (bridge-ops) | 감시 도구가 주입 프롬프트의 지시문(`… 최종 줄에 정확히: [X-DONE] fix1=<a\|b>`)을 완료 마커로 오인. **기록이 이미 있었는데 재범** |

**주목**: 마지막 항목은 "기록해 두면 다음에 안 틀린다"는 가정 자체의 반례다. 기록이 있었고,
세션 시작 시 인덱스를 읽었고, 그럼에도 같은 함정에 빠졌다.

### C. 코드화했으나 **불충분했던** 것 — 2건 (이것이 설계의 난점이다)

| 대상 | 코드화한 내용 | 그런데 |
|---|---|---|
| `scripts/check-mem-header.ts` | auto-update 원복 감지 | 실제 훅의 경로 해석 순서를 복제하지 않아 **구버전을 검사해 false OK** 가능. 앵커 미발견 시 SKIP/exit 0 이라 **감지해야 할 상황에서 침묵** |
| `scripts/watch-card.ts` | 워커 감시(마커/소멸/한도/타임아웃) | 첫 실사용에서 **프롬프트 에코를 마커로 오인**해 발사 직후 "완료" 판정 |

즉 **코드화 자체가 답이 아니다.** 잘못 코드화된 가드는 "지켜지고 있다는 착각"을 만들어
글로 적힌 규칙보다 더 위험할 수 있다. 판별 기준은 이 위험까지 다뤄야 한다.

### D. 애초에 코드화 불가로 분류된 것 — 선례

`docs/spikes/HOOK-CACHE-FIX-DESIGN.md` **§0-bis**는 이미 유사한 프레임을 쓴다:
"TS로 잠글 수 있는가"로 축을 갈라, hook 순서·예산 캡·sha 결정론은 ✅(코드로 잠금),
"lessons 요약 문안"·"교훈 간 우선순위"는 ❌(의미 보존 압축 = 사람 판단)로 분류했다.
**당신의 설계는 그 프레임의 일반화다** — 다만 §0-bis는 한 문서 안의 결정 축을 갈랐고,
이번 것은 **운영 규칙 전반**에 적용되는 기준이어야 한다.

### E. 실패의 관측 방식도 자료다

- 아키텍트가 임시 bash 감시를 **3연속** 손으로 짰고 3번 다 서로 다른 구멍이 났다
  (마커만 봄 → pane 소멸 뺌 → bash 3.2 연관배열 미지원으로 즉사하며 **"35분 경과" 거짓 성공 보고**).
- 아키텍트가 로그 몇 줄로 인과를 구성해 오너에게 **틀린 진단을 확신에 차서 보고**했다
  (`PaneDied`가 herdr 경고인데 브릿지가 버렸다고 해석). codex 교차검증이 기각했다.

## ② 설계해야 할 것

### 핵심 산출: 판별 기준

어떤 규칙을 받았을 때 **"이것은 코드로 강제 가능한가"를 판정하는 기준**을 만들어라.
단순 이분법이 아니라, 다음이 드러나야 한다:

1. **판정 축** — 무엇을 보고 가르는가. (예시일 뿐이니 그대로 쓰지 말고 자기 축을 세워라:
   위반을 기계가 관측 가능한가 / 판정이 결정적인가 / 위반 시점에 개입 지점이 존재하는가 /
   오탐 비용과 미탐 비용의 비대칭 / 규칙이 의미 판단을 포함하는가)
2. **강제 수단의 스펙트럼** — "코드냐 문서냐"는 너무 거칠다. 오늘 자료만 봐도 최소 네 층이 있다:
   **제품 런타임 락**(M-1·L-2) · **하네스 hook**(PreToolUse) · **lint/테스트/CI 게이트** ·
   **문서**. 각 층이 어떤 규칙에 적합한지, 층 선택 기준은 무엇인지.
3. **C 유형(잘못 코드화된 가드)을 막는 조건** — 가드를 만들 때 **무엇을 함께 갖춰야
   "지켜지고 있다는 착각"을 피하는가.** (오늘 두 사례 모두 "가드가 침묵하는 경로"가 문제였다.
   침묵을 성공으로 읽지 않게 하는 설계 요건은 무엇인가.)
4. **코드화 불가 규칙의 차선책** — 사람 판단이 필요해 강제할 수 없는 규칙은 어떻게 다루는가.
   그냥 문서에 두는 것 말고, 위반 확률을 낮추는 수단이 있는가(체크리스트 강제, 산출물 형식
   요구, 리뷰 게이트, 관측 후 사후 감지 등). **오늘 B 유형 4건 각각에 적용해 보라.**

### 필수: 오늘 사례 전수 분류

①의 A·B·C 사례 **전부**를 당신 기준으로 분류하고, 각각에 **권고 강제 수단**을 붙여라.
특히 B 유형 4건은 "왜 지금까지 문서에만 있었는가 / 코드로 옮길 수 있는가 / 옮긴다면 어느 층인가"를
구체적으로 답하라. 기준이 실제로 판별력을 갖는지는 이 대조로만 확인된다.

### 필수: 자기 기준의 한계

당신 기준으로도 **애매한 케이스**를 최소 2개 제시하고, 왜 애매한지 적어라.
모든 규칙을 깔끔히 가르는 기준이 나왔다면 그것은 기준이 아니라 희망이다.

## ③ 제약

- **제품 코드·hook 구현 금지.** 설계 문서만. 구현은 별도 카드로 나간다.
- 오늘 자료를 **재해석하지 말고 근거로 쓰라.** 사실관계가 의심되면 로그·코드로 직접 확인하라
  (`~/.config/herdr/herdr-server.log` · `~/.loom/bridge/mac-node.stderr.log` ·
  `scripts/hooks/check-agent-model.ts` · `packages/host/src/card-ops.ts` 등).
- **낙관 금지.** 오늘 `HOOK-CACHE-FIX-DESIGN.md` §5가 확신에 차서 틀린 파일을 지목했고 그대로
  갔으면 무효 패치였다. 모르는 것은 미확정으로 남겨라.
- 다른 카드가 편집 중인 파일(`scripts/session-context.ts` · `scripts/check-mem-header.ts` ·
  `scripts/watch-card.ts` · `tasks/lessons.md`)을 건드리지 마라.
- git commit/push 금지.
