Architecture Recommendation · 논의 종합

Loom×Herdr

Windows를 컨트롤 타워로, WSL·Mac·Linux를 실행 노드로 두는 분산 멀티에이전트 오케스트레이션. 칸반 카드를 태스크로 발행해 각 노드의 에이전트로 실행하고 상태를 회수하는 구조에 대한 검증과 권고.

Loom · Bun + TypeScript Herdr · Rust Transport · WebSocket ⇄ Unix socket Windows herdr · 회피 검증 방식 · 소스 실측
종합 판정

구조는 채택할 만하다. "Windows는 Loom만, herdr는 Linux/Mac 네이티브만" 분리가 herdr의 가장 약한 부분(Windows 소켓 베타)을 정확히 회피한다. 단 Loom↔herdr 접점 어댑터가 아직 존재하지 않으며, 이것이 프로젝트의 실질적 무게중심이다.

01

두 시스템, 다른 축

Loom과 Herdr는 이름이 둘 다 "여러 개를 하나로 묶는다"지만, 묶는 대상이 다르다. 이 차이가 전체 구조의 근거다. 아래는 각 저장소 소스를 직접 읽어 확정한 사실이다.

Loom · lemonbalms/Loom

휴먼 협업 컨트롤 타워

Bun 모노레포 · TypeScript 5.8. WebSocket relay·MCP 서버를 외부 프레임워크 없이 자체 구현. 핵심 외부 런타임 의존은 사실상 zod 하나. Room·Peer·Handoff·per-peer inbox로 사람들이 협업한다. 데스크톱 셸은 Tauri 2(Rust). 극단적 의존성 미니멀리즘.

Herdr · ogulcancelik/herdr

에이전트 실행·관제 멀티플렉서

단일 Rust 바이너리(~10MB) · Ratatui TUI. tmux를 에이전트 시대에 다시 쓴 것. 로컬 Unix 소켓(Windows는 named pipe)으로 pane·에이전트를 제어하고 상태 5종(idle/working/blocked/done/unknown)을 롤업한다. 에이전트들을 한 사람이 관제한다.

결합점은 MCP

언어도(TS↔Rust) 전송도(WebSocket↔Unix socket) 대상 축도(휴먼↔에이전트) 다르지만, 둘 다 MCP를 에이전트 조율의 접점으로 쓴다. Loom의 mcp-server가 herdr의 소켓 이벤트를 도구로 노출하면 두 시스템이 자연스럽게 만난다. 이것이 아래 전체 구조가 성립하는 이유다.

02

제안 구조 — 계층도

Windows 호스트에서 Loom relay만 돌린다. herdr는 전부 Linux/Mac 네이티브(Unix 도메인 소켓)에서만 실행 — 소켓이 가장 성숙한 환경에서만 쓴다. 각 노드에는 "Loom에 붙고 로컬 herdr를 통역"하는 얇은 브릿지가 필수로 놓인다.

FIG.01 — 노드 토폴로지control ⇄ bridge ⇄ herdr
Loom / relay (WebSocket, 하향 발행) Bridge / herdr (로컬 소켓) 상태·결과 회수 (상향) agent pane

왜 브릿지가 선택이 아니라 필수인가

herdr 소켓은 로컬 전용이다(~/.config/herdr/herdr.sock / named pipe). 네트워크 소켓이 아니므로 Loom relay가 원격 노드의 herdr를 직접 찌를 수 없다. herdr의 --remote는 "로컬 터미널을 원격 herdr 서버의 클라이언트로" 만드는 것이지 소켓 API 원격 호출이 아니다. 따라서 각 노드에 "relay에 WebSocket으로 붙고 + 로컬 herdr 소켓을 통역"하는 데몬이 구조적으로 강제된다.

03

카드 한 장의 왕복

칸반 카드를 herdr 에이전트로 주입·실행·회수하는 건 소켓 API로 완전히 성립한다. "칸반"은 herdr에 없는 개념이라 Loom(외부)이 소유하고, herdr는 pane·에이전트·상태만 안다. 아래는 카드 하나의 생애주기를 실제 소켓 메서드로 매핑한 것이다.

FIG.02 — 카드 디스패치 시퀀스event-driven · 폴링 없음

결과만 받으면 폴링 루프는 필요 없다

완료 감지는 events.subscribe / wait agent-status --status done로 push를 받고, 결과 회수는 done 직후 pane.read를 단 한 번 호출한다. 이벤트 페이로드에 출력 본문이 없는 건 버그가 아니라 백프레셔 설계다 — 출력은 고용량이라 이벤트에 싣지 않고 pull로 뺀다. read 루프가 필요한 건 오직 "카드에 진행 로그를 실시간으로 흘려보내고 싶을 때"뿐이다.

04

강점과 약한 고리

매력적인 그림일수록 경계면에서 깨진다. 소스로 확인 가능한 것과 가정으로 남는 것을 갈라 우선순위 순으로 정리한다.

왜 이 구조가 옳은가

강점 01

Windows 소켓 문제 우회

Windows에선 herdr를 아예 안 돌린다. named pipe·베타·CLI 래퍼 강제 같은 제약을 정면으로 피한다. Windows는 Bun만 있으면 된다.

강점 02

각자 잘하는 걸 소유

Loom은 relay·inbox·MCP를 자체 구현했고 herdr는 pane 관제·상태 롤업을 갖췄다. 재발명이 없다.

강점 03

WSL로 PTY 문제 격리

Windows 네이티브 PTY 이슈(ConPTY·경로·시그널)를 WSL2 리눅스 커널로 밀어넣으면 herdr가 네이티브로 돌아 이 층위 문제가 사라진다.

어디서 깨지는가

RISK · HIGH
약한 고리 01

Loom↔herdr 접점이 없다

Loom MCP는 handoff·board 도구만 노출하고 herdr를 부르는 코드는 없다. 어댑터를 처음부터 짜야 한다. 이건 가정이 아니라 확정된 갭이며 프로젝트의 실질적 무게중심.

RISK · MID
약한 고리 02

herdr 소켓은 로컬 전용

네트워크 소켓이 아니다. 원격 노드에 직접 못 붙으므로 각 노드의 로컬 브릿지(3단: relay ⇄ bridge ⇄ socket)가 불가피하다.

RISK · MID
약한 고리 03

WSL2 네트워킹

Windows relay ↔ WSL 브릿지가 가상 네트워크 경계를 넘어야 한다. networkingMode=mirrored + relay LAN 바인딩 필요. 실측 선행 대상.

05

노드 브릿지 — 통역 계층

브릿지는 두 프로토콜을 잇는 얇은 데몬이다. 위로는 Loom relay에 WebSocket 클라이언트로 붙고(Loom이 이미 relay:lan 스크립트로 0.0.0.0 바인딩 지원), 아래로는 로컬 herdr 소켓을 CLI 래퍼 또는 raw JSON으로 호출한다. 아래는 매핑의 개념 스케치다.

// node-bridge · 개념 스케치 (실제 코드 아님)
// 위: Loom room bus (WebSocket)   아래: herdr 로컬 소켓

relay.on("card.dispatch", async (card) => {
  // 1) 카드 → herdr 워커 스폰 (카드 ID를 env로 각인)
  await herdr("agent start", card.node, {
    env: { LOOM_CARD: card.id },
    argv: card.agentArgv          // 예: ["claude"]
  });

  // 2) 프롬프트 주입
  await herdr("agent send", card.paneId, card.prompt);

  // 3) 완료를 push로 대기 — 폴링 루프 없음
  await herdr.subscribe({
    type: "pane.agent_status_changed",
    pane_id: card.paneId, agent_status: "done"
  });

  // 4) 결과 단발 회수 후 카드에 붙여 relay로 반환
  const result = await herdr("pane read", card.paneId,
                                 { source: "recent", lines: 200 });
  relay.emit("card.done", { id: card.id, result });
});
배치 (a) · 권장

각 노드에 브릿지 데몬

노드마다 "relay WebSocket 클라이언트 + 로컬 herdr 통역" 프로세스. herdr 소켓이 로컬이라 원격 직접 호출 불가 → 로컬 브릿지 필수. 장기 이벤트 구독이 안정적.

배치 (b) · 단순하나 취약

Loom이 SSH로 herdr CLI 실행

ssh node "herdr agent start …". 단순하지만 events.subscribe 장기 스트림이 SSH 세션 위라 취약. herdr 문서도 ssh+tmux의 이미지 붙여넣기 깨짐을 지적 — 장기 스트림엔 (a)가 안전.

06

가장 먼저 할 것 — 수직 슬라이스

멀티 노드·멀티 OS는 나중이다. 가장 얇은 수직 슬라이스 하나가 뚫리면 Mac/Linux 노드는 복제일 뿐이다. "범위를 긋지 않으면 과잉 구현한다"는 원칙 그대로, 검증 후에 늘린다.

0

WSL2 네트워킹 PoC 선행

networkingMode=mirrored + Loom relay를 LOOM_RELAY_HOST=0.0.0.0로 바인딩. WSL 안에서 Windows relay에 WebSocket이 붙는지부터 확인. 여기가 안 뚫리면 나머지는 무의미.

1

노드 1개 슬라이스

WSL herdr 노드 1개 + 로컬 브릿지 1개. Loom relay에서 카드 1장 발행 → 워커 스폰 → done 회수. 이 한 줄이 전체의 증명.

2

Loom MCP 도구 추가

브릿지가 소비할 card.dispatch / card.done 계약을 Loom 쪽 MCP 도구로 정식화. 카드↔pane 매핑을 LOOM_CARD env + report_metadata 토큰으로.

3

노드 복제 · 멀티 OS 확장

Mac·Linux 노드는 동일 브릿지의 복제. 슬라이스가 검증된 뒤에만 늘린다. 라우팅(@node)은 Loom board가 소유.

설계 전제로 못 박을 세 가지

① 노드 브릿지는 선택이 아니라 필수 — herdr 소켓이 로컬 전용이라. ② Loom↔herdr 어댑터가 최대 미검증 영역이자 프로젝트 무게중심. ③ WSL2 네트워킹은 본 설계 전에 PoC로 실측 선행.

07

검증 범위와 한계

  • 실측 확인됨: Loom·herdr 양쪽 저장소를 clone해 소스로 스택·소켓 메서드·이벤트 스키마·CLI 서브커맨드를 직접 읽어 확정.
  • 가정(Assumption): WSL2 mirrored networking + relay LAN 바인딩으로 Windows↔WSL 경계가 해결된다는 것 — 사용자 환경을 직접 보지 못해 미검증. PoC 필요.
  • 버전 시점: herdr는 pre-1.0(v0.7.x)이고 Windows는 베타. 프로토콜 버전이 있어 ping/herdr status로 확인 후 의존해야 함. 향후 릴리스에서 변동 가능.
  • 라이선스: herdr는 AGPL-3.0 + 상용 듀얼. 소켓 API를 호출만 하는 브릿지는 문제없으나, herdr 코드를 포크·임베드해 서비스로 노출하면 소스 공개 의무 — 결합 방식에 따라 검토 필요.
  • 범위 밖: 카드 실시간 진행 로그 스트리밍(별도 read 루프)은 이 구조의 필수가 아니며, 결과만 회수하는 이벤트 드리븐 경로로 충분.
