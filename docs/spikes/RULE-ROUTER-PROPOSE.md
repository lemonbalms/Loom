# Propose (rev-4) — 룰 분리기(Rule Router)

작성 2026-07-23 · 레인: 본세션(topology `single`) 설계 · **제품 코드·hook·주입 변경 없음**
상태: **pending-review** — 이 문서는 문제와 목적의 명문화가 본체이고, 설계는 리뷰 가능한 수준까지만 고정한다.

> 착안: Cursor Router(2026-07-22) — 모델을 실행하기 **전에** 요청을 분류해 작업에 맞는 모델로
> 보내는 분류기. 우리가 이식하는 것은 *모델 선택*이 아니라 **규범 선택**이다.
> 같은 자리(실행 전 분류)에 같은 제약(캐시·예산·보상 신호)을 놓되, 손실 구조가 다르므로
> 최적화 목표를 바꾼다(§3-P1).

관련 정본: [`RULE-ENFORCEABILITY.md`](./RULE-ENFORCEABILITY.md) (등급 H/G/A/J · 층 L0–L5) ·
[`SESSION-START-UNIFIED-PROPOSE.md`](./SESSION-START-UNIFIED-PROPOSE.md) rev-3 (주입 계약·NORMS 팩) ·
[`SESSION-INJECT-VIEW-DESIGN.md`](./SESSION-INJECT-VIEW-DESIGN.md) (뷰≠모델).

## 0. Binding invariant (단 하나)

**규범을 컨텍스트에서 내리는 결정은, 그 규범의 위반을 다른 층이 잡는다는 증거가 있을 때만
성립한다.** 증거가 없거나 분류가 실패하면 축소가 아니라 전량 주입이다. 절감은 목표가 아니라
이 불변식이 지켜진 뒤 남는 잔여물이다.

## 1. Problem — 측정된 사실

### 1.1 규범 자산 대비 자동 전달률 (2026-07-23 실측)

| 축 | 값 |
|---|---|
| 규범 소스 총량 (13개 파일) | **168,772 chars / 249,690 bytes** |
| SessionStart 자동 주입 — state+lessons | 9,500 chars / 11,632 bytes (HARD_CAP 도달) |
| SessionStart 자동 주입 — NORMS 3팩 | 3,657 chars / 4,101 bytes |
| **자동 전달 합계** | **13,157 chars ≈ 규범 자산의 7.8%** |
| 나머지 **92.2%** | “작업 유형이 매칭되면 착수 전 로드” — **사람의 기억에 의존** |

내역: `CLAUDE.md` 7.2KB · `AGENTS.md` 13.5KB · `HANDOFF.md` 5.4KB · `tasks/traps.md` 2.2KB ·
`tasks/lessons.md` 35.3KB · `tasks/lessons/*.md` 5개 129.4KB · `docs/WORKFLOW.md` 20.5KB ·
`docs/DOGFOOD_LOOP.md` 29.2KB · `docs/SESSION-START.md` 7.0KB (bytes 기준).

### 1.2 이 92.2%가 실제로 실패한다는 증거

`RULE-ENFORCEABILITY.md` §5는 2026-07-20 한 세션의 전수 분류다.

| 부류 | 결과 |
|---|---|
| A — 실행 경로를 코드가 중개한 규칙 3건 | **3/3 위반 차단** |
| B — 문서에만 있던 규칙 4건 | **4/4 위반** (일부는 기록된 교훈의 재범) |
| C — 코드화했으나 증거 없는 경로를 PASS로 축약한 2건 | 가짜 안전 |

B 4건(B1 스킬 로드 · B2 dispatch 전 claim · B3 pane 우선 · B4 marker 오탐)의 공통점은
**규칙의 내용이 틀린 것이 아니라, 그 규칙이 필요한 순간에 모델의 컨텍스트에 없었다는 것**이다.
`tasks/lessons.md` 인덱스에도 같은 계열이 남아 있다 — *“기록해도 재범”*, *“재범 3회”*.

### 1.3 문제의 분해 — 4개

| # | 문제 | 현재 상태 | 왜 지금 구조로는 못 고치나 |
|---|---|---|---|
| **P-A** | **전달 실패** — 필요한 규범이 필요한 턴에 컨텍스트에 없다 | 92.2%가 사람 재량 로드 | 로드 의무를 문장으로 더 강하게 써도 그 문장 자체가 로드되어야 한다(순환) |
| **P-B** | **무차별 주입** — 모든 턴이 같은 고정 팩을 받는다 | NORMS 3팩 + state 9,500자 고정 | 예산이 이미 HARD_CAP에 닿아, 규범이 늘면 축을 **삭제**하는 압력이 생긴다(=P-A 악화) |
| **P-C** | **판정 근거 부재** — 무엇을 빼도 안전한지 판단할 기준이 없다 | 등급 체계는 §RULE-ENFORCEABILITY에 있으나 **규칙 단위에 부착되어 있지 않다** | 문서 산문에 등급이 있으면 기계가 못 읽는다 |
| **P-D** | **조용한 축소** — 예산 초과 시 축이 사라지고 경고 한 줄만 남는다 | `fitParts…` 의 `inject omitted:` | 축소가 곧 P-A. 미판정과 정상 축소가 같은 경로를 탄다 |

### 1.4 비-문제 (오해 방지)

- 이것은 **비용 절감 과제가 아니다.** 절감은 §0의 부산물이다. 토큰이 남아돌아도 P-A는 남는다.
- 이것은 **규범을 줄이자는 제안이 아니다.** `SESSION-INJECT-VIEW-DESIGN`의 “뷰≠모델, 영구 slim-delete 금지”와 같은 편에 선다. 오히려 **규범을 늘려도 안전해지는 구조**를 만드는 것이 목적이다.
- 이것은 **규칙 문장을 다시 쓰는 과제가 아니다.** 소스는 그대로 두고 좌표만 붙인다(§3-P5).

## 2. Goal — 무엇을 달성하면 성공인가

### 2.1 목적문 (한 문장)

**턴을 실행하기 전에 그 턴에 적용되는 규범을 기계가 분류해, 필요한 규범은 반드시 도달시키고
불필요한 규범은 예산에서 비켜 두되, 분류가 실패하면 전량으로 되돌아가는 관문을 만든다.**

### 2.2 성공 조건 (G1–G5)

| # | 목표 | 측정 방법 | 합격선 |
|---|---|---|---|
| **G1** | P-A 해소 — 필요한 규범의 도달률 | 과거 트랜스크립트 replay에서 risk-weighted recall | **J등급 miss = 0** (필수), 가중 recall은 Phase 2에서 임계 고정 |
| **G2** | P-C 해소 — 모든 규칙 유닛이 등급·층·surface를 가진다 | `rules:check` 결정론 게이트 | 미분류 유닛 **0건** |
| **G3** | P-D 해소 — 축소와 미판정이 구별된다 | 라우터 4상태(PASS/ROUTED/UNKNOWN/NOT_APPLICABLE) contract test | UNKNOWN → 전량 주입 경로가 테스트로 고정 |
| **G4** | P-B 해소 — 규범이 늘어도 축이 삭제되지 않는다 | 규범 소스 +20% 주입 시 축 삭제 0 | 예산 초과가 **삭제**가 아니라 **라우팅**으로 흡수 |
| **G5** | 회귀 없음 | 세션당 규칙 위반 건수 (shadow 기간 대비) | 증가 없음. 절감폭은 **부수 지표**로만 보고 |

### 2.3 Non-goal (이번 범위 밖 — 명시적으로 하지 않는다)

- 모델 라우팅(어떤 모델/레인에 보낼지). 그건 `DOGFOOD_LOOP §0.5`의 소관이며 이 문서는 건드리지 않는다.
- 규범 문장의 재작성·통폐합·삭제.
- 제품(Loom) 런타임 변경. 이 제안은 **리포 하네스**에 한정된다.
- “학습된 신경망 분류기”. §7 참조 — 우리 데이터 규모에서는 **가중치·임계값 보정**까지만 주장한다.

## 3. Principles — 설계 원칙 5

**P1 · 손실은 비대칭이다.** 모델 라우터는 잘못 골라도 결과가 조금 나쁘다. 룰 라우터는 잘못 빼면
락 위반·데이터 오염이 난다. 따라서 최적화 목표는 정확도가 아니라 **위험가중 recall**이며,
등급별 miss 비용을 다르게 매긴다.

**P2 · 무엇을 뺄 수 있는지는 등급이 답한다.** `RULE-ENFORCEABILITY` H/G/A/J를 그대로 쓴다.

| 등급 | 위반을 잡는 주체 | 라우팅 가능성 |
|---|---|---|
| **H** | 코드 가드가 효과 전 차단 | **컨텍스트에서 내려도 위반이 차단된다** → 라우팅 1순위 |
| **G** | 부분 중개 | 해당 surface 접근 시 조건부 주입 |
| **A** | 사후 탐지 | 조건부 주입 + 탐지 유지 |
| **J** | 사람 판단뿐 — **프롬프트가 유일한 전달 경로** | **pinned** (라우팅 금지) |

즉 “프롬프트에서 뺄 안전성 = 그 규칙을 잡는 코드 가드의 강도”다. 이 대응이 이 제안의 핵심이며,
등급이 없는 규칙은 정의상 뺄 수 없다(G2).

**P3 · 라우팅 시점은 캐시가 결정한다.** Cursor가 캐시 미스 비용을 평가에 포함시킨 것과 같은
이유로, 세션 도중 프롬프트 prefix를 바꾸면 절감이 상쇄된다. 따라서:

| 시점 | 무엇을 결정 | 캐시 영향 |
|---|---|---|
| SessionStart 1회 | prefix 규범 집합 | 없음(세션 시작) |
| 턴 중(도구 호출 직전) | **추가만**(append-only) | prefix 불변 → 유지 |
| 게이트/트랙 전환 | 재결정 허용 | 1회 미스 감수 |

턴마다 prefix를 재작성하는 설계는 이 문서에서 **금지**한다(§10).

**P4 · 미판정은 전량이다.** 4상태 프로토콜(`RULE-ENFORCEABILITY` §3.1)을 그대로 따른다.
분류 실패·앵커 소실·hook 미기동은 “축소해도 되는 상태”가 아니라 UNKNOWN이며 전량 주입 + 표기다.
C1/C2가 남긴 교훈 — **증거 없는 경로를 PASS로 축약하지 않는다** — 가 그대로 적용된다.

**P5 · SSOT를 쪼개지 않는다.** 규칙 본문을 새 파일로 복사하지 않는다. `norms-receipt.ts`가 이미
증명한 **결정론 추출기 + sha8 digest** 방식을 확장하고, 레지스트리는 *좌표와 메타데이터만* 갖는다.
소스가 바뀌면 digest가 깨져 게이트가 잡는다.

## 5. 설계 개요 (리뷰 가능한 최소 수준)

### 5.1 규칙 유닛 레지스트리

```yaml
# rules/registry.yaml — 본문 없음. 좌표 + 메타 + digest.
- id: orch.lane-placement
  source: { file: CLAUDE.md, anchor: "Orchestration standing rules/5", sha8: "…" }
  grade: G            # RULE-ENFORCEABILITY 등급
  layer: [L1, L2]
  surface: [dispatch, delegation]
  triggers: [디스패치, 위임, grok, codex, pane, subagent]
  pin: false
  cost_chars: 412
- id: traps.card-done-not-complete
  source: { file: tasks/traps.md, anchor: "활성 함정/card.done", sha8: "…" }
  grade: A
  pin: true           # 완료 오판정은 치명 → 라우팅 금지
```

### 5.2 선택 메커니즘 — **후보 3안 (미결 · §6.5에서 채택)**

rev-1은 3층 하이브리드를 확정된 것처럼 기술했다. 그것은 근거 없는 선택이었으므로 **후보로
강등**한다. 채택은 §6.5 bake-off의 결과이며, 그 전에는 어느 안도 정본이 아니다.

먼저 rev-1이 뭉갠 **두 축을 분리**한다. 오너가 제시한 3안은 이 두 축이 섞여 있었다.

| 축 | 선택지 | 비고 |
|---|---|---|
| **축1 · 선택 메커니즘** | 결정론 규칙 / LLM 추론 / 혼합 | 후보 A·B·C가 갈리는 진짜 축 |
| **축2 · 룰 표현** | 플랫 리스트 / **관계 그래프**(선행·충돌·상위-하위) | **세 후보 모두 쓸 수 있다** — 접근법이 아니라 자료구조 |
| **축3 · 정책 저작 주체** | 설계자 도출 / **오너 선포 카테고리 표** | §5.2.1 — 셋 공통 전제. 후보가 아니다 |

즉 “그래프”는 B의 전유물이 아니다. 그래프를 A에 주면 A도 선행/충돌을 따라 확장할 수 있다.
따라서 bake-off는 **축1을 비교하고, 축2·축3은 세 후보에 동일하게 적용**한다(교란변수 제거).

#### 5.2.1 카테고리 정책 층 — 오너 저작 (지시 2026-07-23)

오너 제안: **“이럴 땐 이렇게, 저럴 땐 저렇게” 카테고리를 정해 기본 룰을 선포하고, 애매한
나머지에만 A/B/C를 쓴다.** 이것은 네 번째 후보가 아니라 **셋 모두의 전제**이며, 채택 시
라우팅 문제가 둘로 쪼개진다.

| 하위 문제 | 주체 | 성격 |
|---|---|---|
| **(a) 카테고리 → 기본 룰셋 매핑** | **오너 선포** | 정본 정의. 추론 대상이 아니다 |
| **(b) 이 턴이 어느 카테고리인가** | 기계 | **여기만 A/B/C가 경쟁한다** |

이 분해가 주는 것이 세 가지다.

1. **정답의 출처가 생긴다.** §6.2의 한계는 라벨이 counterfactual이 아니라는 것이었다. 오너가
   카테고리별 정본 룰셋을 선포하면 그것이 **명세로서의 ground truth**가 되어, M1이 “과거에 실제로
   읽은 파일”이라는 약한 프록시에 덜 의존한다.
2. **출력 공간이 작아진다.** 라우터가 수백 개 룰 유닛에서 고르는 것이 아니라 **카테고리 N개 중
   하나(또는 여럿)** 를 고른다. 결정론이 이기기 쉬워지고, LLM을 써도 B-1(출력 공간 축소)이
   자동으로 충족된다.
3. **콜드스타트가 풀린다.** 라벨이 없는 상태에서 학습부터 할 필요가 없다. 정책으로 시작하고
   데이터가 쌓이면 보정한다. Cursor에서도 관리자 allow/block·모드 지정이 데이터 라우팅 위에
   얹혀 있다 — 같은 구조다.

**taxonomy 씨앗은 이미 리포에 있다** (새로 창작할 필요가 적다):
`tasks/lessons/` 5개 카테고리(orchestration · bridge-ops · verification · platform · workers) ·
트리거 lexicon 템플릿(S · A · R · plan · no-commit · review · impl) ·
DOGFOOD 역할(arch · impl · rev) · WORKFLOW 게이트 단계. 오너가 처음부터 표를 짜는 것이 아니라
**이미 있는 축을 확인·수정·승인**하는 형태로 시작할 수 있다.

##### 애매 구간의 처리 — 세 원칙

오너가 짚은 “애매한 경우”가 이 설계의 실제 난점이다. 규칙은 셋이다.

1. **애매하면 판정하지 말고 합집합.** 후보 카테고리가 둘 이상이면 룰셋의 **합집합**을 쓴다.
   카테고리는 배타 분류가 아니라 **다중 라벨**이다(위임 + ship이 한 턴에 동시 성립).
2. **어느 카테고리에도 안 맞으면 UNKNOWN → 전량**(P4). 미분류를 임의 카테고리로 흡수하지 않는다.
3. **그래서 LLM의 자리는 “선택”이 아니라 “순위”다.** 애매 구간의 기본 동작이 합집합이면, 남는
   문제는 *합집합이 예산을 넘을 때 무엇을 먼저 넣는가* 뿐이다. 순위는 틀려도 위험이 낮고
   (add-only · 예산 내 정렬), D8 단조성과 그대로 맞는다. **B/C의 존재 이유가 이 지점으로 좁혀진다.**

##### 비용과 감사

- **오너 유지보수 부담**: 룰이 추가될 때마다 매핑을 갱신해야 한다면 부담이 누적된다. 완화 =
  레지스트리 `surface` 필드와 카테고리를 조인해 매핑을 **자동 파생**하고, 오너는 **예외와 pinned만**
  관리한다.
- **사람이 쓴 표도 틀린다.** 정책 표 자체를 replay로 감사한다 — “그 카테고리 턴에서 실제로 필요했던
  룰이 매핑에 있었는가”. 오너 저작이라고 평가에서 면제되지 않는다.
- **드리프트**: 작업 유형이 바뀌면 표가 낡는다. 미분류율(M7) 상승을 **오너 리뷰 신호**로 쓴다.

| 후보 | 정의 | 매 턴 LLM 호출 | 도달 가능 등급 상한(X3) |
|---|---|---|---|
| **A · 결정론** | 발화 트리거 + surface 키워드 + 게이트 상태 → 규칙표. 기존 `classifyOwnerUtterance`(199줄) 확장 | 없음 | **H 가능** (같은 입력 → 같은 verdict) |
| **B · LLM + 그래프+α** | 매 턴 소형 모델이 룰 그래프를 순회해 필요한 유닛을 구성(+α = 임베딩 retrieval·최근 편집 신호 등 보조 스코어) | **매 턴** | **G 상한** — 재현성 보장 불가 |
| **C · 하이브리드** | A가 먼저 판정, 애매 구간만 B로 위임, 실패 시 전량 | 애매 턴만 | 경로별 상이 (A경로 H · B경로 G) |

> **α의 해석**: 오너 발화의 “+알파”를 *그래프 순회 단독이 아니라 보조 신호(retrieval·최근 편집·
> 게이트 상태)를 결합한 스코어링*으로 읽었다. 다른 의미였다면 B의 정의를 정정해야 한다(§8-6).

출력 계약은 세 후보 공통이다: `{ pinned[], routed[], mode, budget, unknown, receipt }`.
receipt 필드는 `RULE-ENFORCEABILITY` §3.2-6 그대로:
`router_version · policy_version · input_hash · selected[] · dropped[] · reason · mode · checked_at`.
**B·C도 receipt 의무를 면제받지 않는다** — LLM이 고른 이유도 provenance로 남아야 한다.

### 5.3 주입 경로 두 개

1. **prefix(SessionStart)** — `session-context.ts`가 라우터 결정으로 파트를 고른다. 축 삭제 대신 라우팅.
2. **JIT(PreToolUse, append-only)** — `Agent|Task` 호출 직전에 orchestration 유닛을, ship 계열 명령
   직전에 commit/verify 유닛을 주입. **B1(스킬 로드)·B3(pane 우선)이 요구하던 “기억해서 로드”를
   라우터가 대신 수행**하는 지점이며, 이것이 P-A에 대한 직접 처방이다.

### 5.4 모드 노브 (오너 소관)

`strict`(전량 — 락 웨이브·R{n}·보안 표면) / `balanced`(기본) / `lean`(읽기전용 보고 턴).
Cursor의 Intelligence/Balance/Cost에 대응하되 축은 비용이 아니라 **위험 허용도**다.

## 6. 학습과 평가 — 60만 건 없이 무엇을 주장할 수 있나

### 6.1 가용 데이터 (실측)

`~/.claude/projects/-Users-…-fable-advisor` = **257 세션 · 236 MB · 58,399 메시지 레코드**.

### 6.2 라벨을 사후 관측으로 만든다

| 라벨 | 추출 신호 |
|---|---|
| positive (그 턴에 필요했음) | 이후 실제로 Read한 `tasks/lessons/<cat>.md`·`docs/*`, 인용된 traps 문구, 호출된 스킬 |
| **강한 positive** | 그 규칙의 위반이 실제로 난 턴 — 오너 정정 발화, `handoff:check`/`norms:check` FAIL, revert, lessons 신규 항목 |
| negative | 세션 전체에서 그 surface 미접촉 |

강한 positive가 Cursor의 AFC 부정 신호에 대응한다. 다만 **우리 라벨은 counterfactual이 아니다**
— 과거 턴은 “규범이 다 주입된 상태”의 행동이므로, replay recall은 상한 추정치다(§9).

### 6.3 지표

- **risk-weighted recall** — miss 비용을 등급으로 가중(H 낮음 · J 최대). **J-miss는 0이 합격선**.
- **violation rate** — 세션당 규칙 위반 건수. 진짜 목표 지표.
- budget saving — 캐시 미스 포함 계산. **부수 지표**로만 보고한다.

### 6.4 평가 방식

**shadow 모드가 정본**: 라우터는 결정만 로깅하고 실제 주입은 종전대로 전량 → recall을 무위험 측정.
표본 규모상 온라인 A/B의 통계적 유의성은 주장하지 않는다(Cursor와 다른 점, 명시).

### 6.5 후보 bake-off — **채택 절차** (오너 지시 2026-07-23)

#### 6.5.1 왜 그냥 비교하면 안 되는가

C는 A와 B의 상위집합이다. 같은 예산·같은 지표로 붙이면 **C가 정의상 이긴다** — 이것은 실패할 수
없는 평가이고, `AGENTS.md`의 Next-action test(“실패하면 새로 뭘 배우나?”)에 걸린다. 공정한
비교가 되려면 세 가지를 먼저 고정해야 한다.

1. **동일 조건** — 같은 replay 코퍼스, 같은 라벨, 같은 레지스트리, 같은 축2(그래프) 적용.
2. **복잡도 페널티 명시** — C의 이득은 *운영 비용 증가분보다 커야* 채택된다. 페널티 항목:
   실패 모드 수, 비결정 경로 존재, 지연, 캐시 미스(P3), 디버깅 가능성.
3. **사전등록(pre-registration)** — 합격선·가중치를 **결과를 보기 전에** 문서에 적는다.
   결과를 본 뒤 임계값을 조정하면 그 bake-off는 무효다.

#### 6.5.2 비교 축 (모두 같은 하네스에서 측정)

| # | 축 | 측정 | 왜 결정적인가 |
|---|---|---|---|
| **M1** | risk-weighted recall · **J-miss** | 257세션 replay | 주 성능 지표. J-miss>0이면 **즉시 탈락**(등급 무관) |
| **M2** | **판정 재현성(X3)** | 동일 입력 n=5 반복의 verdict 일치율 | A=1.0 보장. B/C의 실측 분산이 **등급 상한을 정한다** |
| **M3** | **미판정 의미론(X5)** | UNKNOWN을 PASS로 뭉개는 경로 수 | C1/C2 재범 방지. 4상태 분리 실패 = 탈락 |
| **M4** | 지연·비용 | 턴당 추가 ms / 토큰 | B는 **메타 호출이 본 호출보다 싸야** 존재 이유가 있다 |
| **M5** | 캐시 영향 | prefix 재작성 횟수 (P3) | 매 턴 재구성은 절감을 상쇄 |
| **M6** | 운영 복잡도 | 실패 모드 수 · 오프라인 동작 여부 · 디버깅 경로 | C의 페널티가 여기 실린다 |
| **M7** | 커버리지 | **카테고리 미분류 + 합집합이 예산 초과한 턴의 비율** | **B/C의 존재 가치를 직접 측정**한다(§5.2.1). 이 값이 작으면 A로 끝난다 |

#### 6.5.3 사전등록 합격선

- **J-miss = 0** — 세 후보 공통 필수. 위반 시 그 후보는 탈락한다.
- **M7이 임계 미만이면 A 채택 확정** — B/C를 만들지 않는다. 임계값은 Phase 1 종료 시점에
  레지스트리 규모를 보고 **결과 관측 전에** 고정한다(현재 미정 · §8-7).
- **C는 A 대비 M1 개선이 M6 페널티를 넘을 때만 채택** — 동률이면 **단순한 쪽(A)이 이긴다**.
- **B 단독 채택은 M2·M3을 통과할 때만 가능** — 재현성이 없으면 등급 상한 G이므로, pinned 축소에
  쓸 수 없다(P2와 충돌).

#### 6.5.4 채택 단위 — 단일 승자가 아닐 수 있다

surface마다 승자가 다를 수 있다. 예: 위임·ship처럼 트리거가 뚜렷한 표면은 A, 자유형 조사 턴은
C. **“하나를 골라 전부에 적용”을 강요하지 않는다.** 단, surface별 승자를 허용하면 M6(운영
복잡도)이 올라가므로 그 비용도 같은 표에 계상한다.

#### 6.5.5 중단 규칙

Phase 2 replay에서 **A가 M1을 이미 충족하면 B·C 구현을 중단**한다. 이기고 있는 단순안을 두고
복잡안을 계속 만드는 것은 매몰비용이다.

### 6.6 M2 재현성 — 무엇이 있어야 담보되는가

M2를 “LLM은 비결정적”이라는 한 줄로 두면 측정도 설계도 못 한다. 재현성은 **세 층**이며 담보
수단이 각각 다르다.

| 층 | 정의 | 깨지는 원인 |
|---|---|---|
| **R1** 동일-입력 동일-출력 | 판정 함수의 순수성 | LLM 샘플링, Set/객체 순회 순서, 시계·랜덤 |
| **R2** 입력 안정성 | 같은 상황이 같은 입력으로 정규화되는가 | 최근 편집 목록 순서, git 상태, mtime, 로케일 |
| **R3** 시간 불변 | 오늘의 판정이 다음 달에도 같은가 | **상류 모델 사일런트 업데이트**, 룰 소스 변경, 정책 버전 |

#### 6.6.1 후보 A(결정론)도 공짜가 아니다

A가 R1을 가진다는 것은 **그렇게 구현했을 때**의 이야기다. 요건:

1. 판정 함수는 **순수 함수** — I/O·시계·`Math.random`·환경변수 직접 참조 금지.
2. 입력은 **명시적 스냅샷 객체** 하나로 봉인(발화·Line 튜플·surface·편집 경로 목록·모드).
3. 목록형 입력은 **정렬 후 주입** — Set/Map 순회 순서에 의존하지 않는다(R2).
4. `router_version` · `policy_version` · 레지스트리 `sha8`을 판정에 **함께 해싱**(R3).
5. **골든 테스트** — 입력 fixture → 기대 selected[] 고정. 룰 소스가 바뀌면 digest가 깨져 게이트가 잡는다(P5와 같은 기전).

#### 6.6.2 후보 B/C(LLM)에서 실현 가능한 것

`temperature=0` · seed는 **충분조건이 아니다**. 배치·부동소수점 비결정, 서버측 라우팅, 모델
사일런트 갱신, 캐시 상태가 남는다. 따라서 LLM 경로에서 얻을 수 있는 것은 결정론이 아니라
**측정된 안정성**이며, 그것을 담보로 바꾸는 장치는 다음 다섯이다.

| # | 장치 | 무엇을 담보 |
|---|---|---|
| **B-1** | **출력 공간 축소** — 자유 텍스트 금지, 레지스트리 **ID 집합만** schema 강제 | 표현 변이가 판정 변이로 번지는 것을 차단 |
| **B-2** | **결정 캐시** — `input_hash → decision`을 커밋된 아티팩트로 고정, 히트 시 LLM 미호출 | R1·R3를 **사후적으로** 회복. 캐시 자체가 결정론 산출물이고 리뷰·diff 가능. P3 캐시 미스도 함께 감소 |
| **B-3** | **n-샘플 안정화** — n회 호출 후 합의만 채택, 불일치는 **UNKNOWN → 전량**(P4) | 분산을 은폐하지 않고 안전 측으로 흘림 |
| **B-4** | **모델·프롬프트 pin** — 모델 ID 고정, 주변 대화 미포함, 정규화 입력만 | R3의 통제 가능한 부분 |
| **B-5** | **드리프트 카나리아** — 고정 fixture를 주기 재실행해 일치율 회귀를 탐지 | R3의 통제 불가 부분을 **탐지로 강등**(A등급) |

B-2가 가장 값이 크다. **LLM 출력을 결정론 함수의 입력으로 강등**하면, 라우터의 런타임 판정
자체는 다시 결정적이 된다 — LLM은 *오프라인 제안자*, 캐시가 *온라인 판정자*다.

#### 6.6.3 더 값싼 답 — 재현성이 필요 없는 자리로 옮긴다

재현성이 H급으로 필요한 결정은 **규칙을 빼는 결정**뿐이다. 더하는 결정은 틀려도 안전하다
(예산만 쓴다). 그러므로:

> **LLM 경로에는 add 권한만 주고 remove 권한을 주지 않는다**(단조성).

이 배치에서는 B/C의 비결정성이 **위험이 아니라 예산 변동**으로만 나타나며, pinned 축소는 언제나
A의 결정론 경로가 판정한다. P1(비대칭 손실)·P3(append-only JIT)와 이미 정합하고, §6.6.2의 다섯
장치보다 싸다. **D8의 안전 default로 둔다.**

#### 6.6.4 측정 프로토콜 (M2를 주장이 아니라 수치로)

- **일치 정의**: selected[] 집합의 **위험가중 대칭차 = 0**일 때만 일치(문자열 동일성 아님).
- **n = 5**, 같은 날 반복 + **7일 후 1회**(R3 표본).
- 보고: 일치율 · 불일치 시 UNKNOWN 전환율 · 대칭차에 포함된 최고 등급.
- **합격선**: **J·pinned 유닛에 대한 불일치 0**. 그 밖의 일치율 임계는 사전등록(§6.5.3).

#### 6.6.5 담보할 수 없는 것 (정직한 한계)

상류 모델의 무고지 갱신은 우리 통제 밖이다. B-4 pin으로 늦출 수 있으나 없앨 수 없고, 남는 것은
B-5 탐지뿐이다. 따라서 **LLM 경로의 R3는 원리적으로 H가 될 수 없다** — 이것이 §5.2의 등급 상한
G가 원리적 주장인 이유이며, 실측 일치율이 높게 나와도 바뀌지 않는다. 바뀌는 것은 B-2 캐시로
런타임 판정을 결정론 쪽으로 옮겼을 때뿐이다.

| **2** | `rule-router-eval.ts` + **후보 A 먼저** shadow replay | 없음(로깅만) | 257세션 replay · **J-miss 0** (G1) · M7 측정 |
| **2b** | **bake-off** — A vs B vs C를 §6.5 축으로 비교. §6.5.5 중단 규칙 적용 | 없음 | 사전등록 합격선 충족 · **채택 verdict 기록** |
| **3** | 채택안으로 JIT append-only 주입 (저위험 surface부터: 위임·ship) | 추가만 | 위반 fixture 카나리아 — “가드가 켜져 있고 잡는다” |
| **4** | prefix 라우팅 + 모드 노브 | prefix 축소 | G4·G5 회귀 없음 |

Phase 2b는 **A 구현 이후**에 온다. A는 어떤 후보가 이기든 필요한 하한(트리거·pinned·4상태)이며,
B·C는 A의 커버리지 부족분(M7)이 임계를 넘을 때만 착수한다.

Phase 1–2는 주입을 바꾸지 않으므로 **되돌릴 위험이 없다**. 실제 위험은 Phase 3에서 시작한다.

## 8. 리뷰어가 답해야 할 질문 (모호함 없이)

1. **P2의 대응이 성립하는가** — “H등급은 컨텍스트에서 내려도 안전”이 참인가? 반례(코드 가드가
   잡지만 모델이 그 규칙을 모르면 애초에 다른 경로로 새는 경우)가 있는가?
2. **pinned 집합의 초기 판정 권한**은 누구인가 — 설계자(본세션)인가, 리뷰어인가, 오너인가?
3. **Phase 3의 JIT 주입이 A3(PreToolUse `model` 강제)의 주장 범위를 넘지 않는가** — 다른 하네스
   (Codex/Grok/직접 CLI)에서는 라우터가 동작하지 않는다. 그 세션들의 P-A는 어떻게 처리하는가?
4. **§6.2 라벨의 counterfactual 부재**를 감안할 때 G1 합격선(J-miss 0)이 의미 있는 게이트인가,
   아니면 정의상 항상 통과하는 못-실패 지표인가?
5. 이 표면은 **R{n}(plan_review) 대상인가, spike REVIEW 대상인가?** 선례는
   `SESSION-START-UNIFIED-*`(하네스 규범 → spike REVIEW)이고, 제품 코드 변경은 없다.
6. **§5.2의 α 해석이 오너 의도와 같은가?** “그래프 + 알파”를 *그래프 순회 + 보조 스코어 신호*로
   읽었다. 다르다면 후보 B의 정의와 M4 비용 추정이 함께 바뀐다.
7. **M7 임계값(“A로 충분한 지점”)을 결과 관측 전에 누가 고정하는가?** 사전등록이 깨지면 bake-off
   자체가 사후 정당화가 된다. 설계자가 정하면 자기평가이고, 리뷰어가 정하면 레지스트리 규모를
   보기 전이라 임의값이 된다 — 이 딜레마의 처리를 요구한다.

## 9. Must not

- 규칙 본문을 레지스트리에 복사(P5 위반 · SSOT 분열).
- J등급/pinned 유닛을 라우팅 대상으로 승격.
- UNKNOWN을 축소 경로로 처리(P4 위반 — C1/C2 재범).
- 턴마다 prefix 재작성(P3 위반 — 캐시 미스로 절감 상쇄).
- 절감률을 성과 지표의 머리에 두기(§0 — 절감은 잔여물).
- 라우터 자신의 규칙을 라우팅 대상으로 두기(자기참조 — pinned 고정).
- Cursor의 30–60% 수치를 우리 기대치로 인용(다른 도메인·다른 손실 구조).
- **bake-off(§6.5) 없이 하이브리드를 채택**하거나, 상위집합이라는 이유로 C를 무비교 승자 처리.
- **결과를 본 뒤 합격선·가중치 조정**(사전등록 위반 — 그 bake-off는 무효).
- 재현성(M2)을 통과하지 못한 후보로 **pinned를 축소**(P2 위반 — 등급 상한이 G다).
- **LLM 경로에 remove 권한 부여**(§6.6.3 단조성 위반) 또는 `temperature=0`을 재현성 담보로 제시.
- **애매 구간을 임의 카테고리로 흡수**하거나 합집합 대신 단일 판정으로 좁히기(§5.2.1 원칙 1–2).
- 오너 저작이라는 이유로 **카테고리 표를 감사 대상에서 제외**.

## 11. 사실 범위와 미확정

- §1.1·§6.1의 수치는 2026-07-23 실측이다. 주입 봉투는 `session-context.ts --part all`과
  `norms-receipt.ts --format raw`의 실제 출력을 측정했다. 규범 총량은 13개 파일 concat 실측이다.
- §1.2의 3/3·4/4는 `RULE-ENFORCEABILITY.md`가 명시한 대로 **한 세션의 표본**이며, 일반 위반률로
  외삽하지 않는다.
- **절감폭은 미확정**이다. Phase 2 replay 전에는 어떤 수치도 주장하지 않는다.
- 라벨 counterfactual 부재(§6.2)는 이 설계의 **알려진 구조적 한계**이며 해소책이 없다. 그래서
  J등급을 pinned로 고정해 상쇄한다.
- `rules/registry.yaml`의 앵커 문법(파일 내 위치 지정 방식)은 Phase 1에서 확정한다. 현재
  `norms-receipt.ts`는 정확한 H2 제목·표 행 라벨로 앵커를 잡으며, 그 방식이 일반화 가능한지는
  **미확정**이다.
- 다른 하네스(Codex/Grok)의 PreToolUse 등가물 존재 여부는 확인하지 않았다(§8-3).
- **후보 B/C의 M2(재현성) 분산은 미측정**이다. 실측 일치율은 Phase 2b 전에는 알 수 없다.
  다만 §6.6.5대로, 일치율이 높게 나와도 R3(상류 모델 갱신)는 통제 밖이므로 등급 상한 판단은
  바뀌지 않는다. 바뀌는 조건은 B-2 결정 캐시로 런타임 판정을 결정론화했을 때뿐이다.
- **B-2 결정 캐시의 무효화 정책**(룰 소스 변경·정책 버전 상승 시 캐시 폐기 범위)은 미설계다.
  캐시가 stale 판정을 재생하면 C1(false OK)의 재발이 된다.
- **카테고리 표의 커버리지는 미측정**이다(§5.2.1). 씨앗 축이 리포에 있다는 것은 초안 작성 비용이
  낮다는 뜻이지, 실제 턴의 몇 %를 덮는다는 증거가 아니다. M7 초기값은 Phase 1 이후에만 알 수 있다.
- **합집합 정책의 예산 압력**도 미측정이다. 다중 라벨 합집합이 자주 예산을 넘으면 §5.2.1 원칙 3의
  “순위” 문제가 예외가 아니라 상시가 되고, 그때 B/C의 가치 평가가 달라진다.
- **M7 임계값은 현재 미정**이다(§8-7). 사전등록 없이 bake-off를 시작하면 결과가 사후 정당화가
  되므로, 임계 고정 전에는 Phase 2b를 시작하지 않는다.
- rev-1은 3층 하이브리드를 확정 기술했다. rev-2가 후보로 강등했으므로 **rev-1을 근거로 한 어떤
  구현 판단도 유효하지 않다.**
