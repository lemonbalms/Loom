# Loom 개발 목적 타당성 검토 (2026-07-11)

> 저장 목적: 다음 세션이 이 전략 검토를 이어받아 **행동**하게 하기 위함.
> 이건 기능 PLAN 게이트가 아니라 **제품 방향 결정 기록**임.

## 검토 대상 질문

"현재 단계에서 Loom의 개발 목적이 타당한가?"

## 검토된 사실 (당시 상태)

- Loom = 서로 다른 코딩 에이전트(Claude Code·Codex·Grok) 사용자가 한 Room에 모여
  작업을 주고받는 **local-first 멀티플레이어 레이어**. 에이전트를 만들지 않고 CLI를
  wrap(adapter) + presence + **오프라인-세이프 핸드오프 인박스** + MCP.
- 핵심 가치 한 줄: *"상대가 오프라인이어도 메시지는 유실되지 않는다 — 이게 제품의 전부."*
- 상태: v0.17.1, PLAN approved, blocking 0, 167 tests pass. 소스 11K LOC + 테스트 3K.
  **90커밋 / 2.5일 / 1인 / AI-heavy.**
- 의도적 유보(빌더 확인): 멀티테넌시·클라우드(= MVP라서), 배포·퍼블리시(= 2일차,
  기능 검증 우선), live board CRDT, PTY embed.
- 빌더 프레임: **현재는 테스트/학습/도그푸드**. 반응 좋으면 나중에 출시 예정.
  본 프로젝트는 6인+ 팀.

## 판정 (2인 검토 병합)

- **1차(세션 에이전트, Opus)**: 현 단계(학습/도그푸드 벡터)로 타당. 유보는 단계적합.
- **2차(fable-advisor, Fable 5)**: **PARTIALLY AGREE** — 타당하되 다음 세 군데 교정.

### 병합 최종 판정

> **현 단계 타당성 ✅ — 단, "프로세스 학습 벡터"로서 타당한 것이지 "제품 검증"으로선 아직 0.
> 그리고 배포 유보만은 프로젝트 자신의 진단과 모순.**

### Fable 5가 교정한 3가지 (더 정확함)

1. **"수요 미검증은 미래 프레임에서만 문제"는 틀림.**
   `docs/PRIORITIES.md §1`이 이미 병목을 *"남이 쓰고 믿게 만드는 것"*이라 선언했고,
   `docs/PITCH.md`는 **Slack 배포용 포맷**. 즉 스스로 "수요 검증 단계"라 진단해놓고
   계속 기능(work bus 0.16, launcher 0.17)을 찍음. → **배포 유보만은 깨끗한 YAGNI가
   아니라 자기 진단과 모순.**

2. **결정적 리스크 (제일 중요):** **1인 도그푸드로는 이 제품을 구조적으로 검증할 수 없다.**
   가치 제안이 *다른 머신의 여러 사람* 간 오프라인 핸드오프인데, 현 도그푸드는 한 사람이
   한 머신에서 alice/bob 프로필 + AI 피어를 돌리는 것 → **현 구성에선 검증 신호가
   발생 자체가 불가.** "지켜볼 약한 신호"가 아님.

3. **블라인드스팟:** 실제로 학습되는 건 Loom이 아니라 **멀티에이전트 프로세스**
   (R1–R18 게이트, 리뷰어/구현자 역할). → 프로세스 학습 성공을 제품 검증으로
   착각하지 말 것.

### 규율 투자(167 테스트·18 리뷰 게이트) 정당성

**정당함 — 단 메타 목적이 프로세스 학습이라서.** AI가 90커밋/2.5일로 생성하는 환경에서
게이트·테스트가 사람 리뷰의 대체재이고 실제 버그(M-27 pid 재사용, 타이밍 불안전 비교)를
잡았음. 순수 제품 투자로 보면 과잉이지만, 학습이 목적이면 정당.

## 다음 세션 행동 (단일 최고가치)

기능·폴리시를 더 얹지 말고:

**→ 며칠 내 Loom을 "다른 머신의 실제 사람 ≥2명"에게 태운다.**
(본인의 향후 6인 팀 = 완벽한 테스트베드. 핵심 가치가 신호를 낼 수 있는 유일한 구성.)

이를 위한 최소 선행 작업(딱 이것만, 기능 아님):
1. **최소 배포 경로** — `bun link` 대신 남이 5분 안에 설치할 단일 경로
   (예: 설치 스크립트 1개, 또는 원격 relay 1개 상시 기동 + 초대코드).
2. 외부인이 겪는 **첫 5분 마찰** 제거 (PRIORITIES P0가 이미 지적한 그것).
3. 태운 뒤 **수요 신호 1줄이라도 기록** — "이거 없으면 불편했겠다"류 반응.

## 출시(A 프레임) 전환 트립와이어

아래가 **관찰되면** 학습 벡터 → 출시 후보로 전환하고, 그때 유보한 관문
(멀티테넌시·배포·ICP·해자)을 순서대로 푼다:

- 빌더가 아닌 사람 ≥1명이 자발적으로 재사용하거나 "없으면 불편"을 표현
- 서로 다른 머신 2대 이상에서 실제 오프라인 핸드오프가 발생·수령됨
- MCP/벤더/에이전트-메모리(예: Hindsight)가 아직 이 핵심 한 줄을 흡수하지 않음

## 감시할 시간민감 리스크

**흡수 리스크**: MCP·벤더 네이티브 멀티에이전트·에이전트 메모리(Hindsight bank 등)가
"오프라인 핸드오프" 가치를 흡수하는 것은 MVP 성숙을 기다려주지 않음. 유보해도 되나
가끔 "이 핵심 한 줄이 아직 남이 안 하는 것인가"만 점검.
