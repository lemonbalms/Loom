## 배경 (아키텍트 확정 진단 — 재조사 불요, 그대로 신뢰)

현재 구현(`packages/host/src/hook-sensor.ts`)은 `appendHookTelemetry`(`:429-445`)로 4종 텔레메트리

runtime 배선을 검증하지 않는다(이것도 검증에서 결함으로 지적됨).

**핵심 사실 (락 경계 — 반드시 준수):**

## 1. Objective

hookSensor 활성 claude 카드가 hookHint 없이(또는 완료를 결정하지 않은 비-permission 힌트로) **기존 스크레이프
경로로 done 확정**될 때 `appendHookTelemetry({type:"fallback", reason:…})`를 **정확히 1회** append한다.
스폰 시점 리스너 미확립 케이스에는 `no_listener` reason을 **runtime이 실제로 방출**하도록 배선한다(현재 어디서도
방출 안 됨). 판정·wire 동작은 불변 — append(+플래그) 순수 additive 부작용만 추가한다.

**완료 정의:** ① `hook-sensor.ts`에 완료-폴백 분류 **순수 함수**를 추출·export · ② `bridge-runtime.ts`의
done 확정 **단일 초크포인트**에서 이 함수를 호출해 non-null이면 `fallback` append + 정확히-1회 플래그 set ·
③ 스폰 시점 fallback 2곳(malformed·bind 실패)에도 플래그 set 추가(완료 시점 재-append 차단) · ④ 신규 유닛

## 3. Interfaces / 설계 결정 (아키텍트 확정 — 시그니처 세부만 워커 재량)

### 3.1 순수 분류 함수 추출·export (`hook-sensor.ts`) — 필수

done 확정 시점의 힌트/리스너 상태를 폴백 reason으로 분류하는 **순수 함수**를 추가·export하라. 시그니처 예
(정확한 필드는 flight 실물에 맞춰 확정하되 **순수 + export는 필수**):

```ts
export type CompletionFallbackReason = "no_listener" | "no_hint" | "stale_hint";

/**
 * done 확정 시점 폴백 분류 (D6(b)).
 * - alreadyRecorded true  → null (정확히-1회: 스폰/기왕 폴백이 이미 기록)
 * - listenerEstablished false → "no_listener"  (hookSensor on·claude인데 리스너 미확립)
 * - hint === undefined       → "no_hint"       (리스너 정상·완료 힌트 미도착)
 * - else                     → "stale_hint"    (힌트 있으나 완료를 결정 못 함)
 */
export function classifyCompletionFallback(
  hint: HookHint | undefined,
  listenerEstablished: boolean,
  alreadyRecorded: boolean,
): CompletionFallbackReason | null;
```

**reason 어휘 근거 (중요 — 이 초크포인트는 done 경로 전용):** 완료 초크포인트는
`shouldCorrectCompletionToBlocked(flight.hookHint)`(`hook-sensor.ts:163-167`)가 **false를 반환한 done 흐름**에서만
돈다. 따라서 이 지점의 `hint`는 `undefined` 이거나 **비-permission 힌트**(`Stop`|`idle_prompt`|`UserPromptSubmit`)
뿐이다(`permission_prompt`는 상위에서 이미 `agent_blocked` 교정으로 분기됨). hook은 **완료를 단독 확정하지 않으므로**
(M-2 ④ AND 락 — `Stop`도 스크레이프 확증 필수) 이 지점의 done은 **항상 스크레이프가 짊어진 폴백**이며, `hint`
존재 여부로 `no_hint`/`stale_hint`를 가른다. → 이것이 D6(b)가 원한 빈도 지표다.

### 3.2 얇은 append 헬퍼 (`hook-sensor.ts`) — runtime 배선 검증을 유닛으로 가능케 함

runtime과 **동일한** 코드를 유닛이 통과시킬 수 있도록, 분류+append+플래그 세팅을 묶은 얇은 헬퍼를 export하라
(직접 append 유닛의 결함 재범 회피). 예:

```ts
/** done 확정 초크포인트에서 호출. hookSensor 활성 flight만. append 시 true. */
export function maybeAppendCompletionFallback(flight: {
  hookSensorActive?: boolean;
  hookListenerEstablished?: boolean;
  hookHint?: HookHint;
  hookFallbackRecorded?: boolean;
}): boolean {
  if (flight.hookSensorActive !== true) return false;              // L-2: off/비-claude 미발화
  const reason = classifyCompletionFallback(
    flight.hookHint,
    flight.hookListenerEstablished === true,
    flight.hookFallbackRecorded === true,
  );
  if (reason === null) return false;
  appendHookTelemetry({ type: "fallback", reason });
  flight.hookFallbackRecorded = true;                             // 정확히-1회
  return true;
}
```

flight 파라미터는 구조적 타입(structural)으로 받아 `bridge-runtime`의 `Flight`와 테스트 fake flight 양쪽이
그대로 통과하게 하라(순환 import 회피 — `hook-sensor.ts`가 `Flight`를 import하지 않도록).

### 3.3 Flight 타입 additive 필드 (`bridge-runtime.ts:359-385`)

기존 `hookHint?`/`hookListener?` 인접에 3개 additive:

```ts
/** PLAN 0.26.0 D6(b): hookSensor 활성(cfg.hookSensor && agentKind claude) flight 표식. */
```

**`hookListener` truthiness로 리스너 확립을 판정하지 말 것** — `disposeCardFlight`(`:1919-1922`)→

### 3.4 스폰 게이트 배선 (`bridge-runtime.ts:1088-1130` · `:1157-1168` flight 생성)

게이트 분기(`:1088`)는 flight 생성(`:1157`) **이전**에 돈다. 로컬로 상태를 캡처해 flight 리터럴에 실어라:

### 3.5 완료 단일 초크포인트 (`bridge-runtime.ts` `finishCard` `:2219-2233`) — 권장 설계

**`finishCard`가 모든 done 확정의 단일 싱크다.** 5개 done-분기(`beginCardCompletion` indicator-clear `:1969-1974` ·
settle-fail `:1948-1951` · poll indicator-clear `:2046-2054` · poll exhaust `:2069-2076` · `onCardHerdrEvent`
done/idle는 `:2168-2169`에서 `beginCardCompletion` 위임)가 전부 `finishCard(flight, "done", …)`로 수렴한다.
`onCardHerdrEvent`/poll의 `agent_blocked_correction` 분기는 `finishCard(flight,"failed",…)`이라 자동 제외된다.

**초크포인트 위치 = `finishCard` 내부, pane.read try/catch(`:2236-2251`) 직후**, `status` 확정 후:
```ts
// (2236-2251 read try/catch 뒤, status 최종 확정 지점)
if (status === "done") maybeAppendCompletionFallback(flight);
```
**왜 read 직후인가:** `finishCard`는 `:2249`에서 pane.read 실패 시 `status`를 `done→failed`로 뒤집는다.
top에서 기록하면 실제 failed로 끝난 flight에 폴백을 오기록한다. read 해소 후 `status==="done"` 게이트가
진정한-done만 계측한다. **`maybeAppendCompletionFallback` 내부가 `hookSensorActive` 게이트를 이미 갖고 있어**
grok/codex·hookSensor off flight는 발화하지 않는다(L-2). 초크포인트가 5분기를 전부 커버함을 **코드로 확인**하라
(모든 done이 finishCard를 지나는지 grep으로 검증 — `finishCard(flight, "done"` 호출부 전수).

> 대안(분기별 헬퍼 호출)은 5곳 산발이라 read-flip(`:2249`) 예외 처리와 정확히-1회 보장이 취약하므로 **비권장**.
> finishCard 단일 초크포인트를 채택하라. 부득이 분기별로 갈 경우 read-flip·중복을 워커가 코드로 방어할 것.

### 3.6 `no_listener` runtime 실방출 배선

현재 `no_listener` 문자열은 runtime 어디서도 방출 안 됨(`StartHookListenerResult.reason`은
`"path_rejected"|"bind_failed"`뿐 — `hook-sensor.ts:44-46`). 위 초크포인트에서 `hookSensorActive` flight가
`hookListenerEstablished===false`(리스너 미확립)로 done에 도달하면 `classifyCompletionFallback`이 `no_listener`를
반환해 **runtime이 실제로 이 reason을 방출**한다. 단 bind 실패 flight는 스폰 시점에 이미 `hookFallbackRecorded=true`
(→ `alreadyRecorded` → null)라 이중 계상되지 않고, 스폰 폴백의 기존 `bind_failed` reason이 그대로 남는다. `no_listener`
경로는 "hookSensor on·claude인데 startHookListener를 아예 시도 못 한/미확립인데 스폰-폴백 미기록" 케이스를 계측한다.
