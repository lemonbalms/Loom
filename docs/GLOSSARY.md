# Loom 프로젝트 용어집

> **상태:** 설명용 참조 문서(non-normative)
>
> **작성일:** 2026-07-22
>
> **대상:** `HANDOFF.md`, `docs/PLAN.md`, 설계·리뷰·운영 문서를 읽는 사람

이 문서는 Loom 프로젝트에서 반복해서 사용하는 축약어와 내부 표현을 일반적인 말로 설명한다. 새로 참여한 사람이 문서의 뜻을 빠르게 파악하도록 돕는 것이 목적이다.

이 문서는 기술 규칙의 정본이 아니다. 설명이 실제 계약과 다르면 `docs/PLAN.md`, 해당 설계 문서, `docs/PROTOCOL.md`, 코드와 테스트 순으로 현재 적용 대상을 확인한다.

## 1. 먼저 구분해야 하는 이름

| 표현 | 일반적인 뜻 | 혼동하지 말 것 |
|---|---|---|
| **Loom** | 이 저장소에서 만드는 제품. 여러 에이전트가 작업을 주고받고 상태를 관리하는 CLI·프로토콜·런타임이다. | Fable 5는 Loom 제품명이 아니다. |
| **Fable / Fable 5 / fable-advisor** | 현재는 주로 설계와 PLAN을 검토하는 리뷰 에이전트 또는 플러그인을 뜻한다. 일부 옛 코드와 호환 문서에는 Loom의 과거 이름인 Fable이 남아 있을 수 있다. | 새 제품 이름은 Loom이다. |
| **handoff** | Loom 안에서 한 피어가 다른 피어에게 작업, 맥락 또는 결과를 전달하는 제품 메시지다. | 세션 인계 문서인 `HANDOFF.md`와 다르다. |
| **`HANDOFF.md`** | 다음 개발 세션이 현재 상태와 바로 해야 할 일을 복구하도록 만든 세션 체크포인트다. | 제품 handoff 메시지 한 건의 기록이 아니다. |
| **작업일지 / WORKLOG** | 실행한 명령, 관찰, 시도, 결과를 시간순으로 남기는 기록을 뜻한다. 현재 별도 정본 파일로 확정된 이름은 아니며 제안 설계에서 사용한다. | 다음 행동을 요약하는 `HANDOFF.md`와 목적이 다르다. |
| **`docs/HANDOFF_ARCHIVE.md`** | 완료되었거나 오래된 세션 인계 내용을 보관하는 이력 문서다. | 현재 재개 지점은 `HANDOFF.md` 상단에서 읽는다. |
| **Loom room** | 여러 피어가 메시지와 handoff를 주고받는 협업 공간이다. | 터미널 창이나 Git 브랜치 그 자체가 아니다. |

### `M`은 문맥을 반드시 함께 읽는다

프로젝트 문서에서 `M`은 한 가지 뜻으로 고정되어 있지 않다.

| 표기 예 | 뜻 |
|---|---|
| `M-1`, `M-2` | 리뷰에서 발견한 **Medium(중간 심각도)** 항목 번호 |
| `M1`~`M5` | 특정 PLAN이나 마이그레이션의 **단계 번호** |
| `M0`~`M8` | 과거 로드맵에서 사용한 **마일스톤 번호**일 수 있음 |
| `M-lock` | 설계가 구현 중 바뀌지 않도록 고정한 **필수 결정 사항** |

따라서 `M3`처럼 단독으로 쓰지 말고 가능하면 “PATCH M3”, “Medium M-3”, “마일스톤 M3”처럼 범위를 붙인다.

### `done`도 한 가지 뜻이 아니다

| 표현 | 일반적인 뜻 |
|---|---|
| **board `done`** | 사람이 검증하고 완료로 확정한 작업 상태 |
| **`done_proposal`** | 워커가 완료되었다고 제안한 상태. 아직 사람 또는 타워의 검증이 필요하다. |
| **`needs_verification`** | 결과는 왔지만 완료 확정을 위해 검증이 필요하다는 상태 |
| **프로세스 종료** | 실행 중인 프로세스나 pane이 끝났다는 뜻일 뿐, 작업 성공이나 board `done`을 자동으로 뜻하지 않는다. |

## 2. 문서와 의사결정 체계

| 용어 | 풀어쓴 말 | 이 프로젝트에서의 뜻 |
|---|---|---|
| **SSOT** | Single Source of Truth | 어떤 정보에 대해 최종 기준으로 삼는 단 하나의 정본. 예: 구현 계획의 SSOT는 `docs/PLAN.md`다. |
| **정본 / canonical** | 최종 기준 문서 | 충돌이 생겼을 때 우선하는 현재 규칙이나 계약이다. |
| **원장 / ledger** | 누적 기록부 | 변경과 판정 이력을 순서대로 보존하는 기록이다. 현재 상태를 요약하는 문서와 구분한다. |
| **archive** | 보관본 | 현재 작업에는 직접 쓰지 않지만 과거 근거로 남겨 둔 기록이다. |
| **snapshot** | 시점 복사본 | 특정 순간의 상태를 고정해 둔 자료다. 이후의 최신 상태를 자동 반영하지 않는다. |
| **PLAN** | 구현 계획 | 버전, 범위, 계약, 수용 조건을 정의하는 `docs/PLAN.md`와 그 버전 단위를 뜻한다. |
| **PATCH** | 제한된 변경 단위 | 기존 방향 안에서 수행하는 작은 설계·구현 변경이다. 문맥에 따라 PLAN 안의 `PATCH 1` 같은 실행 단계를 뜻하기도 한다. |
| **R{n}** | Review round number | `R45`, `R46`처럼 번호가 붙은 공식 검토 회차다. `R45b`는 같은 회차의 후속 검토를 뜻한다. |
| **High / Medium / Low** | 심각도 등급 | 리뷰 발견 사항의 영향도다. High는 진행을 막는 중대 문제, Medium은 수정이 필요한 유의미한 문제, Low는 낮은 위험 또는 백로그 후보다. |
| **author-close** | 작성자 종결 | Low 항목 중 정해진 조건을 만족하는 것을 외부 재검토 없이 작성자가 종료하는 절차다. 아무 항목이나 임의로 닫는다는 뜻은 아니다. |
| **approved** | 승인됨 | 현재 정의된 범위가 구현 단계로 넘어갈 수 있다는 판정이다. |
| **pending-review** | 검토 대기 | 설계나 PLAN이 작성되었지만 정식 검토가 끝나지 않은 상태다. |
| **pending-revision** | 수정 대기 | 검토 결과를 반영해 문서를 다시 고쳐야 하는 상태다. |
| **reject / on-hold** | 거절 / 보류 | 현 상태로 진행하지 않거나 외부 조건이 해결될 때까지 멈춘 상태다. |
| **Decision note** | 결정 메모 | 선택한 안과 이유, 버린 대안을 기록한 짧은 의사결정 근거다. |
| **provenance** | 출처 추적 정보 | 결과를 누가, 어떤 모델·버전·입력·코드에서 만들었는지 확인할 수 있게 하는 정보다. |
| **evidence pack** | 근거 묶음 | 검토나 게이트 판정에 필요한 코드, 테스트, 로그, 측정 결과를 한 번에 확인할 수 있게 모은 자료다. |
| **verdict** | 최종 판정 | 리뷰가 현재 문서를 승인, 수정 요구, 보류 중 어디로 판단했는지 요약한 결론이다. |
| **finding** | 발견 사항 | 리뷰에서 확인한 결함, 위험, 모순 또는 개선점 한 건이다. |
| **SSOT conflict** | 정본 충돌 | 제안 내용이 현재 우선권을 가진 PLAN·워크플로·프로토콜과 서로 다른 규칙을 말하는 상태다. |
| **UNVERIFIED** | 미검증 | 아직 참인지 거짓인지 확인하지 않았다는 표시다. 틀렸다고 확정한 표시는 아니다. |
| **Out of scope** | 현재 범위 밖 | 중요하지 않다는 뜻이 아니라 이번 버전이나 변경 단위에서는 다루지 않는다는 뜻이다. |
| **follow-up** | 후속 작업 | 현재 게이트를 막지는 않지만 이후에 별도로 처리해야 하는 일이다. |
| **backlog** | 작업 후보 목록 | 아직 실행 시점이나 범위가 확정되지 않은 작업의 대기 목록이다. |

### 번호가 붙은 설계 항목

| 표기 | 뜻 |
|---|---|
| `U1`~`U11` | 통합 설계에서 구현이 지켜야 하는 구속력 있는 불변식 또는 잠금 항목 |
| `G-1`~`G-11` | 위 설계 잠금이 실제로 지켜지는지 검증하는 회귀·수용 게이트 |
| `UK-1`, `UK-2` | 아직 답이 정해지지 않은 unknown(미결정 사항) 번호 |
| `A1`, `A2`, `A4` | 특정 설계 문서 안에서 비교하는 접근안 또는 하위 설계 번호. 공통 전역 의미는 없다. |

번호의 뜻은 번호 자체가 아니라 그 번호를 선언한 문서가 결정한다. 다른 문서의 같은 번호가 같은 대상을 뜻한다고 가정하지 않는다.

## 3. 작업 진행 표현

| 용어 | 일반적인 설명 |
|---|---|
| **gate** | 다음 단계로 넘어가기 전에 반드시 통과해야 하는 조건 또는 검증 지점 |
| **gate wave / wave** | 하나의 게이트를 위해 계획, 검토, 구현, 테스트, 문서 동기화, 배포까지 이어서 처리하는 작업 묶음 |
| **Current action** | 지금 세션이 바로 수행해야 하는 단 하나의 행동 |
| **next action** | 현재 행동이 끝난 뒤 수행할 다음 행동 |
| **One-line resume** | 다음 세션이 한 줄만 읽어도 재개 지점을 알 수 있게 한 요약 |
| **Don’t redo** | 이미 끝났거나 다시 수행하면 안 되는 작업 목록 |
| **활성 함정** | 현재 작업에서 특히 재발하기 쉬운 오류나 주의사항 |
| **미확인 이월** | 검증하지 못한 채 다음 세션으로 넘긴 사실이나 가정 |
| **blocker** | 해결되기 전에는 현재 게이트를 안전하게 진행할 수 없는 장애물 |
| **candidate** | 채택 가능성이 있지만 아직 확정되지 않은 작업 또는 설계 후보 |
| **scope** | 이번 변경에서 책임지고 다루는 범위 |
| **surface** | 사용자, API 또는 다른 구성요소에 드러나는 접점 |
| **contract** | 구성요소끼리 반드시 지켜야 하는 입력, 출력, 상태, 오류 처리 규칙 |
| **schema** | 데이터 필드와 형식, 필수 여부를 정의한 구조 |
| **invariant** | 실행 경로나 예외와 관계없이 항상 참이어야 하는 조건 |
| **binding lock / M-lock** | 구현자가 임의로 바꾸지 못하도록 검토에서 고정한 설계 결정 |
| **freeze** | 특정 시점부터 범위나 계약 변경을 중단하고 검증에 집중하는 상태 |
| **deviation** | 승인된 설계와 실제 구현 사이의 차이. 숨기지 않고 `implementation-notes.md` 등에 근거와 함께 기록한다. |
| **owner** | 제품 방향이나 최종 권한을 가진 사람 |
| **architect / session model** | 설계 작성, 검토 조율, 구현 결과 검증을 맡는 세션의 주 에이전트 |
| **implementer** | 승인된 명세에 따라 코드와 테스트를 작성하는 에이전트 |
| **reviewer / advisor** | 설계나 구현의 결함, 모순, 위험을 독립적으로 찾는 에이전트 |
| **lane** | 역할과 모델을 분리한 실행 경로. 예: `codex-arch`, `grok-impl`, `codex-impl`, `codex-rev` |

`blocked`는 board의 작업 상태 이름으로도 쓰이고, 세션을 멈춰야 하는 실제 장애물을 뜻하기도 한다. 문서에서는 가능하면 “board blocked”와 “외부 blocker”처럼 구분한다.

## 4. Loom 개발·도그푸딩·하네스

| 용어 | 일반적인 설명 |
|---|---|
| **dogfood / dogfooding** | 우리가 만드는 Loom을 Loom 자체 개발에 실제로 사용하면서 문제와 요구를 발견하는 방식 |
| **agent harness** | 여러 에이전트가 안정적으로 일하도록 역할, 컨텍스트, 메모리, 권한, 이력, 검증 절차를 묶어 제공하는 작업 기반 |
| **development flywheel** | 제품 사용에서 얻은 문제를 제품 개선과 운영 규칙으로 되돌려 반복할수록 개발 체계가 좋아지는 순환 구조 |
| **learning loop** | 실행 → 관찰 → 교훈 추출 → 규칙·제품 반영 → 재검증의 반복 과정 |
| **Primary loop** | Loom 제품을 만들고 Loom으로 그 개발을 수행하는 핵심 순환 |
| **Supporting harness loop** | 핵심 순환에서 검증된 운영 방식을 재사용 가능한 하네스로 일반화하는 보조 순환 |
| **repo-enforced** | 사람의 기억이 아니라 저장소의 테스트, 스키마, 훅, CI 같은 장치가 규칙을 강제하는 상태 |
| **portable** | 특정 저장소나 개인 환경에 묶이지 않고 다른 프로젝트로 옮길 수 있는 상태 |
| **reused** | 실제 다른 작업이나 프로젝트에서 다시 사용해 효과를 확인한 상태 |
| **WIP** | Work In Progress | 동시에 진행 중인 작업 수. WIP 제한은 너무 많은 작업을 벌여 컨텍스트가 분산되는 것을 막는다. |

`Primary loop`, `Supporting harness loop`, `repo-enforced`, `portable`, `reused`는 현재 제안 문서인 `docs/spikes/LOOM-DEVELOPMENT-FLYWHEEL.md`의 설명 용어다. 정식 PLAN에 편입되기 전까지는 제품 계약으로 해석하지 않는다.

## 5. 실행 구조

| 용어 | 일반적인 설명 |
|---|---|
| **PANE-DEATH** | worker가 실행되던 pane 또는 프로세스의 종료를 어떻게 관측하고, 작업 결과·완료 권한·자원 정리와 어떻게 분리할지 다루는 문제 영역 |
| **peer** | Loom room에 참여해 메시지를 주고받는 하나의 에이전트 또는 실행 주체 |
| **relay** | 피어 사이의 WebSocket 메시지를 받아 전달하고 필요한 상태를 보관하는 중계 서버 |
| **bridge** | Loom의 카드·handoff와 외부 에이전트 실행 환경을 연결하는 프로세스 |
| **tower** | 작업을 배정하고 결과를 받아 적용 여부를 판단하는 조정 측 주체 |
| **worker** | 배정된 개별 작업을 실제로 수행하는 에이전트 프로세스 |
| **Herdr** | 워커와 터미널 pane을 생성·관찰·종료하는 외부 실행 계층의 이름 |
| **pane** | 워커가 실행되는 터미널 세션 또는 터미널 분할 창 단위 |
| **card** | board에서 추적하고 워커에게 배정할 수 있는 작업 단위 |
| **board** | 카드와 그 상태를 저장·조회하는 작업 목록 |
| **flight** | 한 번의 활성 dispatch 시도에 대한 메모리 안의 생명주기 기록 |
| **dispatch** | 카드나 작업을 특정 워커에게 보내 실행을 시작하는 행위 |
| **issuer** | 특정 dispatch의 결과를 발행할 권한을 가진 단 하나의 주체 |
| **result** | 워커 또는 bridge가 작업 수행 뒤 돌려주는 구조화된 결과 |
| **generation / attempt** | 같은 카드가 재시도될 때 이전 실행과 새 실행을 구분하는 세대 또는 시도 번호 |
| **`cardId`** | 어떤 카드에 대한 사건인지 식별하는 값 |
| **`dispatchHandoffId`** | 어떤 dispatch handoff에서 나온 결과인지 식별하는 값 |
| **`seq`** | 같은 흐름 안의 순서나 최신 시도를 구분하는 증가 번호 |
| **currency gate** | 도착한 결과가 현재 활성 dispatch의 ID와 순서에 맞는 최신 결과인지 검사하는 관문 |
| **terminal state** | 성공, 실패, 취소처럼 더 이상 정상 진행 상태로 되돌아가지 않는 최종 상태 |
| **terminal guard** | 이미 최종 상태가 된 작업에 두 번째 종료·결과·정리를 적용하지 못하게 막는 검사 |
| **cleanup** | 워커, pane, 메모리 기록 등 실행 자원을 안전하게 정리하는 과정 |
| **cleanup grant** | 특정 자원을 정리해도 된다는 단일 권한 또는 토큰 |
| **tombstone** | 대상이 이미 종료·삭제되었다는 사실을 잠시 남기는 표시. 늦게 온 메시지가 옛 대상을 되살리지 못하게 한다. |
| **presence** | 특정 피어·프로세스·pane이 현재 살아 있거나 연결돼 있는지에 대한 관찰 상태 |
| **`presence_unknown`** | 살아 있음도 종료도 확정할 수 없는 상태 |
| **`pane_exited` / `process_exit`** | 터미널 pane 또는 프로세스가 종료되었다는 사건. 작업 성공을 자동으로 뜻하지 않는다. |

### 상태기계와 권한 표현

| 표현 | 일반적인 설명 |
|---|---|
| **state machine** | 가능한 상태와 상태 사이의 이동 조건을 명시한 모델 |
| **transition / 전이** | 한 상태에서 다른 상태로 이동하는 사건과 규칙 |
| **absorbing / 흡수 상태** | 한 번 들어가면 늦게 온 사건으로 이전 진행 상태로 돌아가지 않는 최종 상태 |
| **authority** | 특정 상태 변경이나 결과 발행을 결정할 수 있는 권한 |
| **authority boundary** | 어떤 구성요소까지 관찰·제안할 수 있고 어디부터 확정할 수 있는지를 나누는 경계 |
| **authority cut** | 여러 설계안을 합칠 때 권한 경계를 기준으로 책임을 잘라 배치하는 방식 |
| **code substrate / 코드 기질** | 새 설계를 실제로 얹을 현재 코드베이스와 그 구조 |
| **ingress** | 외부 메시지나 결과가 구성요소 안으로 들어오는 입구 |
| **observability point / 관측점** | 특정 분기가 실제로 실행됐는지 로그·카운터·상태로 확인할 수 있는 지점 |
| **liveness** | 시스템이 멈춘 채 고착되지 않고 결국 다음 상태로 진행하는 성질 |
| **safety** | 중복 완료, 잘못된 적용, 권한 위반 같은 금지된 상태가 발생하지 않는 성질 |

## 6. 메시지 전달과 신뢰성

| 용어 | 풀어쓴 말 | 일반적인 설명 |
|---|---|---|
| **ACK** | Acknowledgement | relay가 메시지를 받았거나 전달 큐에 넣었다는 확인 응답 |
| **strict ACK** | 엄격한 확인 조건 | 현재 설계에서 결과 전달 성공은 허용된 상태와 정확한 수신자 수 같은 조건을 모두 만족해야 한다는 규칙 |
| **`queued`** | 큐에 들어감 | relay가 전달할 메시지로 받아들였다는 ACK 상태 |
| **`delivered`** | 전달됨 | relay 관점에서 대상에게 전달했다는 ACK 상태 |
| **`recipientCount`** | 수신자 수 | relay가 인정한 대상 수. 엄격한 단일 수신자 경로에서는 정확히 `1`이어야 한다. |
| **`result_relay_accepted`** | 결과를 relay가 수락함 | 전송 단계가 수락됐다는 내부 상태. tower 적용이나 작업 완료를 증명하지 않는다. |
| **`send_unknown`** | 전송 결과 불명 | 전송이 성공했는지 안전하게 확정하지 못한 상태. 성공으로 간주하지 않고 복구·조사가 필요하다. |
| **quarantine** | 격리 원장 | 자동 적용하기 위험한 미해결 결과를 유실하지 않고 별도로 보관하는 내구성 기록 |
| **durable** | 재시작 후에도 남음 | 프로세스가 죽거나 재시작해도 디스크 등의 저장소에서 복구할 수 있는 성질 |
| **replay** | 재생 | 재시작·재연결 뒤 과거 이벤트나 로그를 다시 처리하는 것 |
| **dedup** | Deduplication | 같은 메시지나 결과가 여러 번 와도 중복 효과가 생기지 않게 제거하는 것 |
| **idempotent** | 멱등적 | 같은 요청을 반복해도 한 번 처리한 것과 같은 안전한 최종 결과가 되는 성질 |
| **at-most-one** | 최대 한 번 | 효과가 두 번 이상 생기지 않는다는 뜻. 한 번도 일어나지 않을 가능성은 있다. |
| **exactly-once** | 정확히 한 번 | 효과가 빠지지도, 중복되지도 않고 정확히 한 번 생긴다는 강한 보장. Loom 문서가 wire 전체에 이 보장을 선언한다고 가정하면 안 된다. |
| **CAS** | Compare-And-Swap | 예상한 기존 값이 맞을 때만 새 값으로 바꾸는 원자적 갱신. 여러 종료 신호 중 최초 하나만 권한을 얻도록 쓰인다. |
| **latch** | 일회성 잠금 상태 | 한 번 켜지면 중복 행동을 막기 위해 유지되는 상태 |
| **receipt** | 처리 영수증 | 메시지를 받았거나 적용했다는 사실과 식별 정보를 남긴 기록 |
| **apply receipt** | 적용 영수증 | 단순 수신을 넘어 tower나 소비자가 결과를 실제 적용했음을 보여 주는 기록 |
| **outbox** | 발신 대기 원장 | 보낼 메시지를 먼저 내구성 있게 저장한 뒤 전송·재시도하는 패턴 |
| **reconcile** | 대조 복구 | 서로 다른 저장소나 상태를 비교해 누락·불일치를 찾아 맞추는 과정 |
| **wire contract** | 전송 형식 계약 | 프로세스나 네트워크 경계를 넘어 직렬화되는 필드, 값, 오류의 규칙 |
| **enum** | 제한된 값 목록 | 상태 필드가 가질 수 있는 허용 값의 고정 목록 |

### 실패 처리 표현

| 표현 | 일반적인 설명 |
|---|---|
| **fail-open** | 검사나 보조 장치가 실패해도 기존 동작을 계속 허용한다. 선택적 센서처럼 실패가 핵심 경로를 막으면 안 되는 곳에 제한적으로 쓴다. |
| **fail-closed** | 확실히 안전하다고 증명하지 못하면 동작을 거절한다. 권한, 결과 적용, 최신성 검사 같은 곳에 쓴다. |
| **fail-visible** | 실패를 조용히 삼키지 않고 오류, 상태, 카운터, quarantine 등으로 관찰 가능하게 남긴다. |
| **fire-and-forget** | 비동기 작업을 시작한 뒤 성공·실패를 기다리거나 소비하지 않는 방식. 결과 전달처럼 유실이 위험한 경로에서는 금지 대상이다. |

## 7. 테스트와 검증

| 용어 | 일반적인 설명 |
|---|---|
| **expected-red** | 구현 전에 새 계약을 테스트로 먼저 적고, 아직 구현이 없어서 의도대로 실패하는지 확인한 상태 |
| **tests-only** | 제품 실행 코드는 바꾸지 않고 테스트와 필요한 테스트 장치만 변경하는 단계 |
| **production code** | 테스트·문서가 아니라 실제 사용자 실행 동작에 영향을 주는 코드 |
| **red / green** | 테스트 실패 / 테스트 통과 상태 |
| **baseline** | 새 변경 전 비교 기준이 되는 현재 동작과 테스트 결과 |
| **regression** | 이미 되던 동작이 새 변경 때문에 깨지는 문제 |
| **regression gate** | 과거 동작이 다시 깨지지 않았음을 확인해야 통과하는 검증 조건 |
| **unit test** | 작은 함수나 모듈을 격리해 검증하는 테스트 |
| **integration test** | 여러 구성요소가 실제 계약대로 함께 동작하는지 검증하는 테스트 |
| **E2E test** | End-to-End test | 사용 시작부터 최종 결과까지 전체 경로를 검증하는 테스트 |
| **smoke test** | 핵심 경로가 최소한 실행 가능한지 빠르게 확인하는 시험 |
| **live smoke** | 실제 또는 실제에 가까운 프로세스·네트워크 환경에서 수행하는 smoke test |
| **flake / flaky test** | 코드가 같아도 타이밍이나 환경 때문에 간헐적으로 성공·실패하는 불안정한 테스트 |
| **fixture** | 테스트가 반복해서 사용하는 고정 입력 데이터나 환경 |
| **fake** | 외부 시스템의 필요한 동작만 단순하게 흉내 낸 테스트 구현 |
| **seam / injection seam** | 실제 의존성을 fake나 제어 가능한 동작으로 교체할 수 있도록 만든 경계 |
| **positive assertion** | 특정 사건이나 상태가 실제로 발생했음을 확인하는 검증 |
| **negative assertion** | 특정 사건이나 부작용이 발생하지 않았음을 확인하는 검증 |
| **source assertion** | 실행 결과가 아니라 소스 텍스트나 특정 코드 형태가 존재하는지 검사하는 검증. 행동 테스트를 완전히 대신하지는 못한다. |
| **dead code / caller 0** | 실행 경로에서 호출되지 않는 코드 / 호출자가 하나도 없는 코드 |
| **ambient env** | 테스트가 명시적으로 주입하지 않았지만 실행 머신에 우연히 존재하는 환경 변수나 설정 |
| **latched env** | 모듈 로드나 시작 시 한 번 읽힌 뒤 실행 중에는 바뀌지 않는 환경 값 |
| **locked spec** | 리뷰와 승인으로 구현 계약이 고정된 명세 |
| **self-contradiction** | 한 문서 안의 요구 사항끼리 동시에 만족할 수 없게 서로 충돌하는 상태 |
| **직독** | 요약만 믿지 않고 관련 소스나 문서를 직접 읽어 확인하는 것 |
| **실측** | 추정이 아니라 명령, 로그, 테스트로 현재 값을 직접 측정하는 것 |
| **축자 대조** | 이름과 문장을 실제 원문과 글자 단위로 비교하는 것 |

## 8. 기술 약어

| 약어 | 원문 | 쉬운 설명 |
|---|---|---|
| **CLI** | Command-Line Interface | 터미널 명령으로 사용하는 인터페이스. 예: `loom` |
| **TUI** | Terminal User Interface | 터미널 안에서 화면과 키 입력으로 조작하는 사용자 인터페이스 |
| **PTY** | Pseudo-Terminal | 프로그램이 실제 터미널처럼 입출력하도록 제공하는 가상 터미널 |
| **API** | Application Programming Interface | 프로그램끼리 기능과 데이터를 주고받는 공개 접점 |
| **RPC** | Remote Procedure Call | 다른 프로세스나 머신의 함수를 호출하듯 요청하는 통신 방식 |
| **IPC** | Inter-Process Communication | 같은 머신의 프로세스끼리 데이터를 주고받는 방식 |
| **MCP** | Model Context Protocol | 모델이 도구와 외부 컨텍스트를 표준 방식으로 사용하는 프로토콜 |
| **WS / WebSocket** | WebSocket | 연결을 유지하며 양방향 메시지를 주고받는 네트워크 방식 |
| **JSONL** | JSON Lines | 한 줄에 JSON 객체 하나씩 저장하는 로그 형식. 일부 줄이 손상되어도 나머지를 순차 처리하기 쉽다. |
| **TTL** | Time To Live | 데이터나 상태를 보관하는 유효 시간 |
| **GC** | Garbage Collection | 더 이상 필요 없는 오래된 상태나 자원을 정리하는 작업 |
| **CRDT** | Conflict-free Replicated Data Type | 여러 곳에서 동시에 수정해도 정해진 규칙으로 충돌 없이 합칠 수 있는 데이터 구조 |
| **UX** | User Experience | 사용자가 제품을 쓰면서 겪는 전체 경험 |
| **DX** | Developer Experience | 개발자가 개발·테스트·운영 도구를 쓰면서 겪는 경험 |
| **SHA** | Secure Hash Algorithm | Git 커밋 등 내용을 식별하는 해시 값에 흔히 쓰이는 알고리즘 이름 |
| **CI** | Continuous Integration | 변경마다 자동으로 빌드와 테스트를 실행해 통합 문제를 찾는 체계 |
| **VPS** | Virtual Private Server | 원격에서 계속 실행할 수 있는 가상 서버 |
| **WSL** | Windows Subsystem for Linux | Windows 안에서 Linux 환경을 실행하는 기능 |
| **SSH** | Secure Shell | 암호화된 연결로 원격 머신의 명령을 실행하는 방식 |
| **PID** | Process Identifier | 운영체제가 실행 중인 프로세스에 부여하는 식별 번호 |

### 운영 문서에서 자주 보이는 쉘 표현

| 표현 | 일반적인 설명 |
|---|---|
| **cwd** | Current Working Directory. 명령이 현재 기준으로 삼는 디렉터리 |
| **PATH** | 명령 실행 파일을 찾을 디렉터리 목록을 담은 환경 변수 |

| `LOOM_*` | 현재 Loom 제품이 읽는 환경 변수 접두사 |
| `FABLE_*` | 과거 이름의 환경 변수 접두사. 0.10+ 규칙에서는 읽지 않고 필요한 경우 경고만 한다. |

## 9. board 상태 빠른 참조

| 상태 | 뜻 |
|---|---|
| `todo` | 아직 시작하지 않은 작업 |
| `doing` | 누군가 맡아 진행 중인 작업 |
| `blocked` | 외부 조건이나 선행 문제 때문에 진행할 수 없는 작업 |
| `done` | 요구된 검증을 거쳐 완료로 확정한 작업 |

상태 전이는 단순한 문구 변경이 아니라 권한과 증거를 동반한다. 특히 worker나 bridge의 종료 신호만으로 `done`을 자동 생성하지 않는 것이 현재 handoff·pane-death 설계의 핵심 구분이다.
