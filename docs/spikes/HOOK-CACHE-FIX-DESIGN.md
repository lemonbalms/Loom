## 0-bis. 평가 프레임 — 「TS로 잠글 수 있는가」 (오너 확정 · 1급 축)

**근거 (한 줄)**: 원인 A는 "hook을 한 이벤트에 둘 걸면 안 된다"는 규약이 코드로 보장되지 않고
사람 주의에만 맡겨져 깨진 것 — **규약 의존을 색출해 코드로 잠그는 것이 재발 방지의 본체다.**

| 축 | TS 처리 | 비고 |
|---|---|---|
| hook 순서 결정론화 | ✅ 완전 | 단일 출력이면 레이스 자체가 성립 불가 |
| 예산 배분·캡 관리 | ✅ 완전 | state 고정분 / HANDOFF 변동분 우선순위 |
| 절단 라우드화 | ✅ 완전 | 조용한 손실 → 경고 + lint 실패 |
| 센티널 보존 | ✅ 완전 | `AGENTS.md:13-15` 폴백 의존 문자열, 유닛으로 잠금 |
| 캐시 결정론 검증 | ✅ 완전 | sha 동일성 자동 검사로 회귀 차단 |
| 카테고리 파일 재편 | ⚠️ 부분 | 기계적 이동 가능, **무엇을 어디로는 판단** |
| lessons 요약 문안 | ❌ 불가 | 의미 보존 압축 — 31줄, 상위 8줄이 56% |
| 교훈 간 우선순위 | ❌ 불가 | 어느 트리거에 자수를 더 줄 것인가 |
| claude-mem 원인 B | ❌ 무관 | 정책·blast radius 결정 |

> **규칙 (명문화)**: **❌·⚠️ 항목만 사람이 손대고, ✅ 항목은 반드시 코드+테스트로 잠근다.**
> ✅ 축을 "이번엔 주의해서 하자"로 처리하는 것은 원인 A의 재생산이다. 사람의 몫은 판단이 개입하는
> ⚠️·❌ 축(무엇을 어디로 옮길지, 무엇을 남길지, 정책 결정)으로 한정한다.

## 0. 요약 (읽는 사람이 3줄만 읽는다면)

1. **권고 = 안 M-1 「단일 커맨드 병합 + lessons 인덱스 줄-예산 다이어트 + 라우드 예산 린트」.**
   원인 A(순서 레이스)를 구조적으로 소거하고, 동시에 "lessons 블록이 캡 코앞" 결함과
   `truncateContext()` 조용한 손실까지 한 번에 없앤다. 리추얼 기능(state+lessons 세션 시작 도달)은
   100% 보존된다.
2. **본 설계 과정에서 캡의 정체가 실측으로 확정됐다 — 종전 이해가 두 군데 틀렸다.**
   캡은 **커맨드당 10,000자**(총량 아님)이고, 초과 시 **조용한 절단이 아니라 "디스크 영속화 +
   2,000자 프리뷰 + 파일 경로" 봉투로 대체**된다. 그 **경로에 세션마다 바뀌는 UUID 2개가
   들어간다** → 캡 초과는 *영구 캐시 미스 + 왕복 부활*이라는 **최악 조합**. 따라서 "캡을
   회피/수용한다"류 안(M-2)은 실측으로 기각된다.
3. **원인 B(claude-mem 분 단위 ts)는 오너 결정 사항.** 다만 실측 결과 **미스 1회의 실제 비용이
   33,915토큰 cache-write**로 한정됨이 확인됐다(전체 프리픽스가 아니라 `msg[0]` 구간만 무효화 —
   시스템/툴 45,157토큰은 여전히 히트). 우선순위 판단에 이 수치를 쓸 것.

### 1.2 【신규 실측 A】 캡은 **커맨드당 10,000자**, 초과 = 영속화 봉투 (조용한 절단 아님)

Claude Code 2.1.215 번들(`~/.local/share/claude/versions/2.1.215`) 정적 분석:

```js
async function WLt(e,t,r,n=kou){            // = persistHookOutput
  if (e.length <= n) return e;              // kou = 1e4  (= 10,000)
  let o = await _ot(e, `hook-${t}-${r}`);   // 디스크 영속화
  ...
  return yot(o);                            // <persisted-output> 봉투로 치환
}
```

- 상수: `kou=1e4`. 호출부는 **hook 결과 1건마다** — `yield {additionalContexts:[await WLt(G.additionalContext, `${r}-${j}`, "additionalContext")]}`.
  → **커맨드당 캡이 확정**. 총량 캡이 아니다.
- 봉투 형태(`yot`): `<persisted-output>\nOutput too large (N). Full output saved to: <path>\nPreview (first 2.0KB):\n<프리뷰>\n…\n</persisted-output>` (`EXt = 2000`).

**라이브 실증** — 스크래치패드에 14,011자를 뱉는 SessionStart hook을 임시 `--settings` 파일로 등록해
(리포 무수정) 로깅 프록시로 실제 요청 본문을 캡처:

- 결과: `msg[0][1]` 안에 **2,025자짜리 `<persisted-output>` 봉투**만 남고 `CAPPROBE-TAIL` 소실.
- 봉투에 박힌 경로:
  `/Users/…/projects/-Users-…-fable-advisor/**837e77cf-25df-…**/tool-results/hook-**de96c37e-bb4f-…**-5-additionalContext.txt`
  → **세션 디렉터리 UUID + hook 인스턴스 UUID 2개가 매 세션 변한다.**

> **설계상 결정적 함의**: 캡을 넘기면 (a) 주입 텍스트가 매 세션 반드시 달라져 **원인 A/B를 고쳐도
> 캐시가 절대 히트하지 않고**, (b) 본문을 얻으려면 에이전트가 그 파일을 Read 해야 하므로 **왕복이
> 부활**한다. "캡을 넘겨도 괜찮다/알아서 잘라준다"는 전제는 틀렸다.

**동시에 총량 캡 부재도 실증됨**: 기존 캡처 20건 전부 주입 블록 총합 **16,334자**인데 영속화 봉투가
전혀 없다(=총량 10,000 캡이면 발화했어야 함).

### 1.3 【신규 실측 B】 concat 순서 = **완료 순서** · claude-mem은 20/20 마지막

번들의 hook 실행부는 결과를 **완료되는 대로 스트리밍**한다(`for await (let G of Cao(P))`) —
정렬 단계가 없다. 캡처 20건 실측 분해(`payloads/*req*.json` → `messages[0].content[*]`):

| 태그 | 블록 총자수 | 순서 (오프셋) |
|---|---:|---|
| p1, p2, q2 | 16,334 | prefix(18) → **lessons**(56) → **state**(9456) → claude-mem(13553) |
| p3, p4, q1, q3, q4 | 16,334 | prefix(18) → **state**(56) → **lessons**(4153) → claude-mem(13553) |
| u1–u4 (LOOM hook 제거군) | 2,837 | prefix(18) → claude-mem(56) |

- **LOOM 2블록은 뒤집힌다(3 vs 5).** 총자수는 항상 16,334 → 길이 비교로 안 잡힘. 규명 재확인.
- **claude-mem은 20/20 마지막.** 단 이는 "보장된 순서"가 아니라 **claude-mem이 항상 더 느리기
  때문**(node 기동 + 워커 HTTP vs bun 스크립트)으로 보는 것이 옳다. 설계에서 이 순서에 의존하지 말 것.
- 블록 구분자는 `\n\n---\n`(JSON 봉투형 hook 사이). 프리픽스는
  `<system-reminder>\nSessionStart hook additional context: `(57자).

### 1.4 【신규 실측 C】 미스 1회의 실제 비용 = **33,915토큰 cache-write** (전체 프리픽스 아님)

`analyze.py payloads/p3__req003.json` — cache_control 브레이크포인트 위치:

```
system[1]            @141,922 chars (~35,480 tok)
system[2]            @159,092 chars (~39,773 tok)
msg[0]:user[5]:text  @222,301 chars (~55,575 tok)
```

주입 블록은 **`msg[0]` 안 = 마지막 system 브레이크포인트 뒤**에 있다. 따라서 블록이 바뀌어도
시스템/툴 프리픽스는 무효화되지 않는다. 응답 usage 실측이 이를 그대로 확인한다:

| 상태 | cache_read | cache_creation |
|---|---:|---:|
| 풀히트 (p4, q3, q4) | 79,072 | 0 |
| 미스 (p2, p3, q1, q2) | 45,157 | **33,915** |
| LOOM hook 제거군 미스 (u1, u4) | 45,157 | 23,265 |

→ **미스 1회 = 33,915토큰이 cache-read(0.1×) 대신 cache-write(1.25×)로 청구**. 세션당 1회.
이것이 원인 A·B 각각을 고칠 때 회수되는 실제 금액의 상한이다.

## 2. 원인 A 해소안 — 후보 5개

각 안은 §1의 실측을 제약으로 받는다. 특히 **"단일 additionalContext는 10,000자 미만이어야 한다"**가
절대 제약이다(초과 = 영구 미스 + 왕복, §1.2).

### M-1 — 단일 커맨드 병합 + lessons 인덱스 줄-예산 다이어트 + 라우드 린트 ★권고

**구현 개요**
1. `.claude/settings.json` SessionStart handler 2개 → **1개**
   (`session-context.ts --part all`, 기존 `state`/`lessons`는 진단용으로 존치).
2. `session-context.ts`에 `buildAllContext() = [buildStateContext(), buildLessonsContext()].join("\n\n---\n")`
   (구분자는 현행 하네스 구분자와 동형으로 두어 문서 형태 변화 최소화).
3. `tasks/lessons.md` 인덱스를 **줄당 예산 ≤120자**로 정리. **요약·삭제가 아니라 이동** —
   각 줄의 서사·경위·커밋해시·재범 기록은 이미 존재하는 `tasks/lessons/<category>.md` 로 내린다
   (파일 자체 규칙 3 "요약·삭제 금지, 이동만"과 정확히 일치).
4. `truncateContext()`의 **조용한 절단을 폐기**하고 두 갈래로 교체:
   - 런타임: 예산 초과 시 블록 **맨 앞**에 `⚠ [LOOM-SESSION-CONTEXT] 예산 초과 — 인덱스 N자 초과,
     tasks/lessons.md 정리 필요` 를 붙이고(=에이전트가 즉시 인지) 그 다음에 절단.
   - 커밋 시점: `bun run session-context:lint` (기존 `handoff:lint` 선례) — 병합 블록 총자수 >
     `SOFT_CAP 8500` 이면 **실패**. 조용한 손실의 발생 자체를 커밋 게이트에서 차단.

**예산 산술** (목표: 캡 10,000 대비 15% 여유 = **≤8,500**)

```
state                                    4,096   (HANDOFF 파생 · handoff:lint가 이미 상한 관리)
구분자 + 센티널                             ~60
────────────────────────────────────────────────
lessons 가용                              4,344
  ├ 헤더/사용규칙/카테고리줄                  766  (현행 유지)
  └ 인덱스 32줄 예산                       3,578  → 줄당 평균 112자
```

실측 분포 대조: 현재 32줄 중 **120자 이하는 1줄**, 150자 이하 3줄, 200자 이하 13줄.
전 줄 120자 상한 적용 시 합 **3,836자**(목표 3,578 대비 258 초과) → 실무 목표는
**"줄당 ≤120자 + 총 인덱스 ≤3,600자"** 2중 예산으로 두고, 초과분은 카테고리 파일로 마저 내린다.
가장 긴 줄(625자, `[workers] (9) sonnet claude-mem 거부 루프`)만 정리해도 ~500자가 회수된다.

**성장 대비**: 40줄까지 늘어도 40×120 + 766 + 4,096 + 60 = **9,722** → 캡 미만이지만
`SOFT_CAP 8500`은 초과 → 린트가 **미리 시끄럽게** 잡는다. 이것이 설계 의도다.

| 평가축 | 판정 |
|---|---|
| 캡 통과 | ✅ 8,500 목표 · 캡 대비 15% 여유 · 린트가 회귀 차단 |
| 캐시 안정성 | ✅ 커맨드 1개 = 순열 1개. 원인 A **구조적 소거**(레이스 자체가 사라짐) |
| 리추얼 보존 | ✅ state·lessons 둘 다 세션 시작 도달. 인덱스 줄은 "트리거+규칙 알맹이"로 **본래 설계에 복귀**. 상세는 원래부터 카테고리 파일 로드가 의무(`CLAUDE.md` 4번) — **왕복 신규 발생 없음** |
| 조용한 손실 | ✅ 제거(라우드 마커 + 커밋 게이트) |
| 구현 규모 | 中. 코드 ~40줄(병합 함수·린트·settings 1곳) + **문안 작업**(인덱스 32줄 정리, 볼륨) |
| 리스크 | 인덱스 정리 시 **트리거 판별력 훼손** 가능 — 줄에서 "언제 이 카테고리를 열어야 하는가"가 사라지면 리추얼이 실질 붕괴. → 정리 규칙을 "트리거 문구는 절대 삭제, 경위·해시·재범기록만 이동"으로 명문화하고 1회 검수 게이트를 둘 것 |

### M-2 — 단일 커맨드 병합 + 캡 초과 수용 ❌ 실측 기각

병합만 하고 다이어트를 하지 않는 안(13,497자). §1.2 실증에 따라 결과는
`<persisted-output>` 봉투 + **세션마다 바뀌는 UUID 경로 2개**.

| 평가축 | 판정 |
|---|---|
| 캡 통과 | ❌ 초과 |
| 캐시 안정성 | ❌ **원인 A/B를 다 고쳐도 100% 미스** (경로 UUID가 매 세션 변동) |
| 리추얼 보존 | ❌ 2,000자 프리뷰만 남고 본문은 파일 Read 왕복 |
| 조용한 손실 | △ 손실은 아님(파일 보존)이나 도달 실패 |
| 구현 규모 | 소 |

**질문 "캡이 커맨드당인가 총량인가"에 대한 답: 커맨드당(10,000). 확인 방법은 §1.2에 기술된 그대로
재현 가능** — 임시 `--settings` 파일 + 14k자 hook + 로깅 프록시. 리포 무수정으로 수행 가능.

### M-3 — 2커맨드 유지 + 파일 핸드셰이크로 순서 결정론화

**실현 가능성 판정: 가능하다.** (냉정히 따진 결과 "불가능"이 아니다.)

기구: hook stdin JSON에는 `session_id`가 들어온다.
- ① `--part state`: 본문 emit **직후** `${TMPDIR}/loom-sc-${session_id}.state.done` 생성.
- ② `--part lessons`: 시작 시 그 파일을 짧은 주기로 폴링(상한 예: 3s, 타임아웃 시 그냥 진행 =
  fail-open), 감지 후 emit.
→ ②의 프로세스 종료가 항상 ①보다 늦으므로 **완료 순서가 결정론적**. `session_id` 키잉으로 이전
세션의 stale 센티널 오염도 차단.

| 평가축 | 판정 |
|---|---|
| 캡 통과 | ✅ 각 블록이 독립적으로 캡 이하 — **문안 다이어트 불필요** |
| 캐시 안정성 | ○ 원인 A는 해소되나 **기구 의존적**(타임아웃 발화 시 순서 미보장 → 그날만 미스) |
| 리추얼 보존 | ✅ 완전 보존 · 콘텐츠 무변경 |
| 조용한 손실 | ❌ **미해소** — lessons는 여전히 `HARD_CAP 9500` 대비 여유 101자. 1줄 추가 시 조용한 절단 발화 |
| 구현 규모 | 中(폴링·타임아웃·stale 처리·tmp 정리). 실패 모드가 시간 의존이라 **테스트가 어렵다** |
| 리스크 | 세션 시작 지연 +수백 ms(직렬화). "시간에 기대어 순서를 만드는" 구조라 원인 A와 같은 계열의 취약성을 남김 |

**M-1 대비**: 콘텐츠를 안 건드린다는 유일한 장점이 있으나, ③ 조용한 손실을 못 없애고 리스크

불필요해진다.

## 4. 권고안 — **M-1** (기준 ①>②>③>④ 순 적용 결과)

- **① 리추얼 보존**: M-1·M-3·M-5만 통과. M-2·M-4 탈락.
- **② 캐시 안정성**: 남은 셋 중 M-1이 유일하게 **구조적**(커맨드가 1개면 순열이 1개뿐 — 타이밍에
  기대지 않는다). M-3은 타임아웃 경로에서 원인 A가 부활한다.
- **③ 조용한 손실 제거**: M-1만 해소. M-3·M-5는 `HARD_CAP` 여유 101자 문제를 그대로 남긴다.
- **④ 단순성**: M-1이 가장 무겁지만(문안 32줄 정리), 무거운 부분이 **코드가 아니라 문안**이고
  이는 위임 가능한 볼륨 작업이다(`CLAUDE.md` 오케스트레이션 규칙 1의 분업 대상).

**핵심 논거**: 원인 A·B와 "인덱스가 캡 코앞"이라는 세 증상이 전부 **"lessons 블록이 너무 크다"**
한 곳으로 수렴한다는 브리프의 진단이 실측으로 확인됐다. M-3은 증상 하나만 우회하고 원인은
그대로 둔다. M-1은 원인을 제거하고 나머지가 따라 해소되는 유일한 안이다.

**적용 순서 제안**
1. 인덱스 다이어트(문안) → `SOFT_CAP` 도달 확인 (여기까지 캐시 효과 0, 되돌리기 쉬움)
2. `--part all` 추가 + 린트 추가 (코드)
3. `.claude/settings.json` handler 2→1 (여기서 원인 A 소멸)
4. §6 검증 계획 실행

**되돌리기**: 3번은 커밋 1개 revert. 1번은 인덱스 문안이 카테고리 파일로 "이동"만 됐으므로
정보 손실 없이 복원 가능.

### 4-bis. 본세션 판정 (2026-07-20 · 아키텍트) — **M-3 비권고 · 권고 = 단일 커맨드 병합 + 구조 분리 + 결정론 검증**

**M-3(파일 핸드셰이크) 비권고.** 두 가지 이유로 확정한다.

1. **실패 유형이 원인 A와 동일하다.** M-3의 타임아웃 fail-open 경로는 "기능은 멀쩡한데 캐시만
   조용히 깨지는" 상태를 만든다 — 이는 원인 A가 우리를 물었던 바로 그 실패 유형이며, M-3은 그것을
   **설계에 내장**한다. 0-bis 프레임으로 말하면, "hook 순서 결정론화"는 ✅(코드로 완전히 잠글 수
   있는) 축인데 M-3은 그 축을 **시간에 기대는 확률적 기구**로 되돌린다. ✅ 축을 ✅로 처리하지 않는
   설계는 채택하지 않는다.
2. **유일한 우위가 소멸했다.** M-3의 실질 우위는 §7-bis (e)의 "예산 2×10,000자 유지 = 문안
   다이어트 불필요"뿐이었다. 그런데 —
   - (a) `HARD_CAP 9500` 대비 **여유 101자**로 인한 조용한 손실은 **M-3으로 고쳐지지 않는다**
     (§2 M-3 평가표 ③). 인덱스가 1줄 자라는 순간 절단이 발화하므로 **어차피 축소가 필요하다.**
   - (b) 압축의 대가로 여겨졌던 정보 손실은 **구조 분리**로 사라진다 — **요약은 주입하고, 상세는
     같은 파일 `<details>` 안에 보존**한다. 즉 압축이 **무손실**이 되므로 "다이어트를 피하기 위해
     2×10,000 예산을 지킨다"는 M-3의 논거 자체가 성립하지 않는다.

**따라서 권고 = 「단일 커맨드 병합 + 구조 분리 + 결정론 검증」.** M-1의 골자를 유지하되 다이어트를
"삭제·요약"이 아니라 **구조 분리(요약 주입 / 상세 `<details>` 보존)** 로 재정의하고, §6 V-1의 sha
동일성 검사를 **선택 검증이 아니라 필수 코드 잠금**(0-bis "캐시 결정론 검증 ✅")으로 승격한다.
§7-bis (d)에 기록된 "인덱스만 읽고 세부까지 알던 경로가 사라진다"는 최대 미검증 리스크도, 상세가
같은 파일에 남으면 **왕복 없이 펼쳐 볼 수 있어** 실질 완화된다.

**오너에게 필요한 판단**: (i) 홈 설정/플러그인 설치본 수정을 전 프로젝트에 적용할지,

## 6. 검증 계획 — 수정 후 캐시 안정성 실측

**`scratchpad/cache-miss-probe/proxy.py` 방식은 그대로 재사용 가능하다.** 본 설계 과정에서
동일 하네스로 §1.2 캡 실증을 이미 수행해 재사용성이 실증됐다(리포 무수정, 임시 `--settings`
파일만 추가).

### V-1 순서 결정론 (원인 A) — API 호출 없이 선행 가능
- `bun run scripts/session-context.ts --part all` 을 20회 실행 → 출력 sha256 **전부 동일** 확인.
- 커맨드가 1개이므로 순열이 없다는 것은 구조적 사실이나, 회귀 방지용 유닛으로 고정한다.
- **합격 기준**: 20/20 동일 sha.

### V-2 캡 준수 — 정적
- `--part all` 출력의 `additionalContext.length` < 8,500(SOFT_CAP) 그리고 < 10,000(하드).
- `bun run session-context:lint` 가 인위적으로 부풀린 `tasks/lessons.md` 에 대해 **비영 종료**하는지.
- **합격 기준**: 실측 길이 보고 + 린트 fail 재현.

### V-3 라이브 캐시 실측 (프록시 재사용)
1. `proxy.py 8788` 기동, `ANTHROPIC_BASE_URL=http://127.0.0.1:8788`.
2. `run.sh <tag>` 로 동일 프롬프트 세션을 **4회** 실행 (`r1`…`r4`),
   **의도적으로 분 경계를 넘겨서**(예: 각 실행 사이 70초 간격) 수행.
3. 각 `payloads/*req*.json` 에서 `messages[0].content[*]` 중 `SessionStart hook` 포함 블록을 추출해
   - (a) 블록 sha256
   - (b) `[LOOM-SESSION-CONTEXT` 센티널 오프셋(순서)
   - (c) `<persisted-output>` 부재
   를 비교.
4. `payloads/*resp*.txt` 에서 `cache_creation_input_tokens` / `cache_read_input_tokens` 추출.

**판정표**

| 조건 | 기대 결과 |
|---|---|
| 원인 A만 수정 (M-1 적용, 원인 B 미조치) | LOOM 구간 바이트 동일 4/4 · **claude-mem 헤더 분 자리만 상이** · 분 경계 넘은 실행은 여전히 `create≈33,915`(단 블록 축소로 값 감소) |
| 원인 A + B 모두 수정 | **블록 sha 4/4 동일 · `create=0`, `read≈79,072` (풀히트 4/4)** |
| 회귀(캡 초과) | `<persisted-output>` 검출 → 즉시 실패 판정 |

**중요 — 절대 수치 금지 원칙**(`tasks/lessons.md:40` verification (10)): `create`/`read`의 절대값은
모델·툴 수·MCP 구성에 따라 통째로 변한다. **판정은 "같은 세션 조건 4회 내 sha 동일 여부"와
"create==0 여부"라는 상대/이진 지표로만** 내린다. 위 79,072/33,915는 본 설계의 baseline 기록용이지
합격선이 아니다.

### V-4 리추얼 기능 회귀 (기준 ① 보호)
- 다이어트 후 인덱스만 보고 "작업 유형 → 로드할 카테고리"가 판별되는지 대조:
  32줄 각각에 대해 트리거 문구가 남아 있는지 체크리스트 검수(1회, 사람 또는 별도 리뷰 레인).
- **합격 기준**: 32/32 줄에서 트리거 식별 가능. 미달 줄은 다이어트 되돌림.

## 7-bis. M-1 채택 시 영향 분석 (오너 질의 「1번으로 했을때 영향은?」)

### (a) 캐시 — **단독 효과는 기대보다 작다**

| | 현재 | M-1만 | M-1 + 원인 B |
|---|---|---|---|
| 원인 A 미스 | 발생 | **소멸(구조적)** | 소멸 |
| 원인 B 미스 | 발생 | **잔존** | 소멸 |
| 실효 | 사실상 매 세션 미스 | **"1분 이내 재시작에서만 히트"** | 풀히트 |
| 미스 1회 비용 | 33,915 tok create | ~31,000 tok (블록 16,334→~11,300자 축소분 반영, 추정) | — |

- **"M-1만"의 상태는 대조군 실측과 동일하다** — LOOM 훅만 제거(claude-mem 잔존) 시 같은 분 내 히트
  2/2, 분 경계 넘으면 미스. 즉 M-1 단독의 실사용 캐시 이득은 **작다**.
- 그럼에도 **A 수정은 필요조건**: B만 고치고 A를 두면 훅 2개 순열이 계속 뒤집혀 캐시는 여전히 깨진다.
- → **결론: M-1과 원인 B 조치는 세트로 판단해야 한다.** M-1을 "캐시 때문에" 정당화하면 실측치가
  기대에 못 미친다. M-1의 자체 정당화는 (b)(c)에 있다.

### (b) 컨텍스트 — **확실한 이득**

- lessons 블록 **9,399자 / 6,625토큰 → ~4,600자 / ~3,050토큰**.
- 매 세션 **~3,575토큰 절약 = 200k 창의 1.79%p**. (WP5가 세운 ≥3%p 문턱에는 단독으로 미달하나,
  WP5 baseline A 대비 절감 1.19%p보다는 크다.)

### (c) 조용한 정보 손실 — **제거** (M-1의 가장 확실한 단독 가치)

현재 `HARD_CAP 9500` 여유 101자 = **인덱스 1줄(중앙값 237자) 추가 시 즉시 조용한 절단 발화**.
M-1 후에는 여유 ~4,000자 + 초과 시 라우드 마커 + 커밋 린트 차단.

### (d) 리추얼 — 보존되나 **성격이 바뀐다** (최대 미검증 리스크)

- state·lessons 둘 다 세션 시작 도달 → 기준 ① 충족.
- 그러나 인덱스 줄에서 **경위·커밋해시·재범기록·①②③ 세부**가 카테고리 파일로 내려간다.

### (e) 예산 구조 — **M-1의 진짜 구조적 대가** (반드시 인지할 것)

- 게다가 state는 HANDOFF 파생이므로 **HANDOFF가 커지면 lessons 예산을 잡아먹는다** —
  지금까지 없던 커플링이 생기고, `handoff:lint`(top-80 >8,192B)와 신설 `session-context:lint`가
  서로를 밀어내는 관계가 된다.
- **이것이 M-3(핸드셰이크, 2×10,000 유지)의 유일한 실질 우위다.** 인덱스가 앞으로 크게 자랄
  전망이면 이 대가를 다시 저울질할 것.

### (f) 구현 범위 · 되돌리기

- 손대는 곳: `.claude/settings.json`(handler 2→1) · `scripts/session-context.ts`(+`--part all`,
  `truncateContext` 교체, ~40줄) · `tasks/lessons.md`(31줄 재작성) · `tasks/lessons/*.md` 5개(유입) ·
  `package.json`(린트 별칭) · 유닛.
- **주의**: `AGENTS.md:13-15` 무훅 폴백 분기가 `[LOOM-SESSION-CONTEXT` 센티널 문자열에 의존한다.
  병합 블록에 두 센티널을 **문자 그대로 유지**할 것 — 이름을 바꾸면 폴백이 조용히 깨진다.
- `--part state|lessons`는 진단용으로 **존치** → 기존 테스트·스파이크 문서 앵커가 살아남는다.
- 되돌리기: settings 1줄 revert로 즉시 원복(단 원인 A 부활). 문안은 "이동"뿐이라 정보 손실 없이
  복원 가능.

### (g) 한 줄 판정

**M-1은 "캐시 수리"라기보다 "조용한 손실 제거 + 인덱스를 인덱스로 되돌리기"가 본체이고,
캐시는 원인 B와 함께 갈 때만 회수된다.** 채택 시 원인 B 결정을 같은 웨이브에 묶는 것을 권고한다.

## 8. 미해결로 남는 질문

1. **claude-mem이 20/20 마지막인 이유가 "항상 느려서"인지 "소스 우선순위 정렬"인지 미확정.**
   번들에 소스 우선순위 정렬 함수(`Kou`, `userSettings→projectSettings→…→pluginHook(999)`)가
   존재하나 이는 설정 **표시**용으로 보이고, 실행 결과는 완료 순서 스트리밍이다. 확정하려면
   claude-mem을 인위적으로 빠르게(또는 우리 훅을 느리게) 만든 대조 실험이 필요.
   → **M-1은 이 답에 의존하지 않는다**(우리 커맨드가 1개면 무관).
2. **`SOFT_CAP` 값 8,500의 근거는 "캡 대비 15% 여유"라는 공학적 판단이지 실측이 아니다.**
   인덱스 성장률(월 몇 줄) 실측이 있으면 더 정확히 잡을 수 있다.
3. **인덱스 120자 상한이 트리거 판별력을 유지하기에 충분한가는 미검증.** V-4가 이를
   사후 검사하지만, 사전에 3~5줄 파일럿으로 감을 잡는 편이 안전하다.
4. **`source=resume` 에서의 거동은 본 설계 범위 밖.** 우리 matcher는 `startup|clear`뿐이나
   `tasks/lessons.md:50` platform (18)②는 SessionStart가 resume에서도 발화한다고 기록한다.
   resume 세션에서 우리 훅이 안 뜨는 것이 의도인지 재확인 필요(별건).
5. **`~/.claude/settings.json` 의 gsd SessionStart 훅 2개**는 이 리포에서 사실상 no-op
   (`gsd-session-state.sh` 는 `.planning/config.json` 부재 시 즉시 exit 0)이나, `.planning/` 이
   생기면 **세 번째 레이서**가 된다. 현재는 무해, 향후 감시 대상.
6. **claude-mem 플러그인 실사용 경로가 `marketplaces/thedotmack/plugin` 인지
   `plugins/cache/thedotmack/claude-mem/<ver>` 인지** — hooks.json 의 해석 로직이 cache 우선이다.
   B-3/B-4 적용 시 **실제 로드되는 경로를 먼저 확정**해야 한다(둘 다 수정하는 편이 안전).
